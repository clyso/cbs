# 006 — Versions

This is the reference design for the versions subsystem of the `cbscore`
library: the version-string grammar and parse helpers, the `VersionType` table,
version-descriptor creation (`versions create`, including the new auto-UUIDv7
path), the release listing (`versions list`, fixing the broken Python call
site), the external image-descriptor lookup, and the configurable
version-descriptor store location (designed here, built later). Read 002 for the
`VersionDescriptor` wire type and the `VersionType` enum (defined there; the
label table lands here), 004 for `Config`, and 005 for `list_releases`.

Source of truth: `cbscore/versions/utils.py`, `cbscore/versions/create.py`,
`cbscore/versions/errors.py`, `cbscore/images/desc.py`,
`cbscore/cmds/versions.py` (the CLI wiring itself is specified in 010).

## Version strings & parse helpers

Source: `versions/utils.py`. The grammar (a single regex, `utils.py:44-59`):

```
[<prefix>-]  [v]  <major>  [.<minor>  [.<patch>  [-<suffix>]]]
```

- `prefix` is `\w+`; `major` is mandatory; `minor`, `patch`, `suffix` optional;
  a leading `v` is optional. `suffix` is `[\w_.-]+`.
- `parse_version(s) -> (prefix?, major, minor?, patch?, suffix?)` — raises
  `MalformedVersion` (002) on no match.
- `get_major_version(s)` — requires major **and** minor; returns
  `"<major>.<minor>"` (CES treats the first two components as the "major").
- `get_minor_version(s)` — raises `MalformedVersion` on a malformed string; for
  a well-formed string it returns `"<major>.<minor>.<patch>"` when patch is
  present, otherwise `None`.
- `normalize_version(s)` — requires major and minor; re-emits
  `[<prefix>-]v<major>.<minor>[.<patch>][-<suffix>]`.
- `parse_component_refs(list) -> map<name, ref>` — parses `COMPONENT@REF`
  entries (regex `^([\w_-]+)@([\d\w_./-]+)$`).

These helpers depend on `regex`, so they live in `cbscore` (the library), not in
the zero-IO `cbscore-types` (001/002). A golden test ports the Python parametric
cases (`utils.py:161-197, 221-241`) verbatim.

## `VersionType` and the type table

`VersionType` (the enum) is defined in `cbscore-types` (002) with lowercase wire
values (`release`/`dev`/`test`/`ci`). The **label → (type, description)** table
is version-domain data and lives here (`utils.py:33-38`):

| label     | `VersionType` | description            |
| --------- | ------------- | ---------------------- |
| `release` | `Release`     | `General Availability` |
| `dev`     | `Dev`         | `Development`          |
| `test`    | `Test`        | `Testing`              |
| `ci`      | `Ci`          | `CI/CD`                |

- `get_version_type(name) -> VersionType` — a **name → type lookup** over this
  table (not a `regex` parse; resolves 000's M3). Unknown name → `VersionError`.
- `get_version_type_desc(type) -> &str` — type → description; used in titles.
  The title says "General Availability", **not** "Release" — this is the
  description column (a faithful Python behavior, not a 000-review finding).

## Version-descriptor creation (`versions create`)

Source: `versions/create.py`. `version_create_helper(...)` → `create(...)`
assembles a `VersionDescriptor` (002) from the CLI inputs (version, type name,
`COMPONENT@REF` map, components paths, URI overrides, distro, el_version,
registry, image name/tag, git user/email).

- **Component resolution**: `load_components(components_paths)` (default
  `<cwd>/components`), then each ref's repo URL comes from the component
  definition, overridable by `-o COMPONENT=URI` (006 owns the `COMPONENT=URI`
  parsing; the component model `CoreComponent` / `load_components` is owned by
  007 and referenced here).
- **Title** (`_do_version_title`, `create.py:45-76`): for a parseable version
  (minor **and** patch required) the title is
  `Release <description> [<PREFIX> ]version <major>.<minor>.<patch>[ (<suffix-parts>)]`,
  where each `-`-separated suffix part is uppercased on its first `.`-segment
  (e.g. `rc.1` → `RC 1`, joined by `, `). The `<description>` is the type's
  table description (the `get_version_type_desc` value, above).
- **Image**: `image.tag` falls back to the version string when no `--image-tag`
  is given (`create.py:133`).
- **Validation**: `_validate_version` requires minor and patch, so a
  human-supplied version must be at least `M.m.p`.
- **Write site**: the descriptor is written to the version store at
  `<store>/<type>/<version>.json`; an existing file at that path is a "version
  already exists" error. The store root is covered below (Configurable
  location).

## Auto version (UUIDv7) — new feature

This is a new capability (not in Python), in scope from the start (001).
`versions create` makes the `VERSION` positional **optional**:

- `resolve_version(cli: Option<&str>) -> String` — supplied → use it; omitted →
  a freshly generated **UUIDv7** (sortable by creation time).
- `validate_version(s)` accepts **either** the Python version-string shape
  (minor and patch present, per `_validate_version`) **or** a valid UUIDv7.
  (This is also a small correctness gain over the un-validated string the wire
  type would otherwise accept.)
- **Title for a UUIDv7 version**: the major/minor/patch parse does not apply, so
  the title is `Release <description> version created at <timestamp>` (ISO-8601
  UTC, seconds precision). The `<timestamp>` is the time **embedded in the
  UUIDv7** (UUIDv7 encodes a Unix-millisecond creation time), not wall-clock
  "now" — so a user-supplied UUIDv7 yields its own creation time and a generated
  one yields the moment of generation. The version-string path keeps the
  `_do_version_title` format above.
- `desc.version` stays a plain string (002) — no wire change; only its values
  widen. The UUIDv7 string (36-char hyphenated) is a valid OCI tag, so the
  `image.tag` fallback still works; operators wanting a stable tag pass
  `--image-tag`.
- The trailing image-descriptor check (`get_image_desc`, below) is **skipped**
  for a UUIDv7 version — there is no `M.m.p` to key an image descriptor on.

## `versions list` — fixing the broken call site (C7)

Source: `cmds/versions.py` + `releases/s3.py` (005). The Python `versions_list`
calls `list_releases(secrets, url)` with two arguments, but the function
requires four (`cmds/versions.py:129` vs the signature in 005) — it raises
`TypeError` and never works. The port **fixes** it: `bucket` and `loc` come from
`config.storage.s3.releases`, and `--from` supplies the S3 host URL. Both
`config.storage` and `config.storage.s3` are optional (004); when either is
absent, `versions list` fails with a clear, operator-actionable error ("no S3
storage configured; `versions list` requires `storage.s3` in the config"), never
a panic. The listing/parsing itself is `list_releases` (005), which is lenient
(skips malformed descriptors).

## External image descriptor (`get_image_desc`)

Source: `images/desc.py`. `versions create` ends with a non-fatal "is there an
image descriptor for this version?" check, so this lookup and the
`ImageDescriptor` type are owned here (002 deferred `ImageDescriptor` to 006).
`ImageDescriptor` is an **external, repo-authored, unversioned** input (001) —
no `schema_version`.

```rust
struct ImageDescriptor { releases: Vec<String>, images: Vec<ImageLocations> }
struct ImageLocations { src: String, dst: String }
```

**The Python `get_image_desc` is broken** (`images/desc.py:42-91`), so the port
implements its _intended_ contract, not its behavior: the filename filter at
`desc.py:50` is a **raw** string `r"^.*{m[1]}.*.json"` (not an f-string), so
`{m[1]}` is never interpolated, no candidate ever matches, and the function
**always** raises `NoSuchVersionError` — the "image descriptor missing" warning
fires on every create. Two genuine matches would raise `ImageDescriptorError`
(conflict), but a single match followed by further candidates returns the
**last-parsed** descriptor rather than the matched one; and a malformed
candidate is re-raised rather than skipped.

The intended `get_image_desc(version) -> ImageDescriptor` the port implements:

- Extract `<M>.<m>.<p>` from the version with `parse_version` (the leading `v`
  optional, fixing Python's `v`-requiring regex at `desc.py:43`). A version with
  no `M.m.p` — a UUIDv7, or a patch-less `20.2` — has nothing to key on and
  **skips the lookup** (see below); it is never an error.
- Walk `<git-repo-root>/desc/` for candidate `*.json` files. The name match is a
  **cheap pre-filter** on the `M.m.p` substring (the interpolation the raw
  string was meant to do); it may over-match (`20.2.1` also matches
  `20.2.10.json`) — harmless, because the next step is authoritative.
  `NoSuchVersion` if the `desc/` directory is missing.
- Parse each candidate, **skipping** a malformed one (matching `list_releases`'
  leniency, 005). The **authoritative** match is exact membership of the version
  in a descriptor's `releases` list (as Python does, `desc.py:76`): the version
  must appear in `releases` in the **same form** `versions create` stored it
  (the raw, un-normalized `desc.version`) — the `v`-optional extraction above
  only drives the file pre-filter, not the match, so this is not a normalization
  equivalence. Two matches → `ImageDescriptorError` (conflict); none →
  `NoSuchVersion`.

`versions create` runs this check only for versions that yield an `M.m.p`; a
**UUIDv7** or a patch-less version is **skipped entirely** (an image descriptor
is keyed by `M.m.p`). For an `M.m.p` version, a `NoSuchVersion` (or missing
`desc/` directory) is reported as a benign note ("image descriptor missing"),
**never** an error — unlike Python, where the missing-`v` `MalformedVersion` is
a `CESError` subclass routed to the _error_ branch at
`cmds/versions.py:112-117`. With the fix the note fires only when a descriptor
is genuinely absent — today it fires unconditionally, and would error outright
for a UUIDv7 or `v`-less version.

## Configurable version-descriptor location (M5 — designed, built later)

Python hardcodes the store path as `<root>/_versions/<type>/<version>.json`
under the git repo root (`cmds/versions.py:86-90`, with a "make this
configurable" FIXME). This design specifies the configurable form now; it is
**built in M5/C9** (001), not in the `versions create` milestone (M1/C1), which
uses the hardcoded default.

- **Surface**: an additive optional field on 004's `PathsConfig` —
  `Config.paths.versions: Option<Utf8PathBuf>` (key `versions`) — plus a
  `--versions-dir <path>` flag on `versions create`. (004's `PathsConfig` sketch
  cross-references this M5 field.)
- **Precedence**: CLI flag > config field > default
  (`<git rev-parse --show-toplevel>/_versions`).
- **Layout preserved**: the configured root behaves exactly like `_versions` —
  per-type subdirectories and `<version>.json`. A single pure helper,
  `descriptor_path(root, type, version) -> <root>/<type>/<version>.json`, is the
  one source of truth for the path (it lives in `cbscore-types`, used by the
  writer and any future reader).
- **Write-only**: `versions create` is the only writer; readers (`build`, the
  runner) take an explicit descriptor path from the caller, so no ambiguity over
  "which store".
- **Adding `paths.versions`** is an additive optional config field —
  `absent → unchanged behavior`, no `schema-version` bump (per the pragmatic
  policy, 001).
- **Error**: when no override is set and the caller is not in a git checkout,
  the error names **both** `--versions-dir` and `Config.paths.versions` so the
  operator knows the two ways to fix it.

## Errors

Per 002's split: the **parse/validation** errors — `VersionError`,
`MalformedVersion`, `InvalidVersionDescriptor` — are type-layer and live in
`cbscore-types`. The **IO-triggered** errors raised by this subsystem live here:
`NoSuchVersionDescriptor { path }` (the store file is absent — raised by the
reader), `NoSuchVersion` (no image descriptor matched), and
`ImageDescriptorError` (two image descriptors claim the same version). The last
two are new to this subsystem and intentionally absent from 002's type-layer
taxonomy — they are IO-triggered, from `get_image_desc`'s directory walk.
`versions create` maps an existing store path to a "version already exists"
error.

## Fidelity notes

- **`get_version_type` is a name→type lookup** (no `regex`), distinct from
  `parse_version` (000's M3).
- **Titles use the type description** ("General Availability", …), not the type
  name (a faithful Python behavior; not a 000-review finding).
- **`_validate_version` requires minor and patch** — a human version is at least
  `M.m.p`; the UUIDv7 path is the only way to omit that shape.
- **UUIDv7 auto-version is new** (in scope from M1); it widens `desc.version`
  values only, not the wire type.
- **`versions list` is fixed** — `bucket`/`loc` from config, `--from` is the
  host URL (C7).
- **Configurable store is M5** — M1's `versions create` uses the hardcoded
  `<git-root>/_versions` default.
- **`ImageDescriptor` is unversioned** (external repo input).
- **`get_image_desc` is fixed, not ported** — the Python raw-string filter never
  matches (always reports "missing"); the port implements the intended lookup (a
  source defect, like the broken `versions list`).

## Testing

- **Parse/normalize golden tests**: port the Python parametric cases
  (`utils.py:161-241`) verbatim — valid/invalid version strings and normalize
  outputs.
- **Title generation**: `Release General Availability version 20.2.1`, prefix
  and suffix variants (`rc.1` → `(RC 1)`); a `dev` type yields `Development`.
- **`get_version_type`**: each label maps to its type; unknown → `VersionError`;
  confirm it does **not** parse version strings.
- **UUIDv7**: omitted `VERSION` yields a valid UUIDv7 `desc.version` and a
  "created at <timestamp>" title; `validate_version` accepts both a `M.m.p`
  string and a UUIDv7, and rejects e.g. `19`, `19.2`, `foobar`; the trailing
  image-descriptor check is skipped (no error) for a UUIDv7.
- **`get_image_desc`**: a version present in one descriptor's `releases`
  resolves; a missing `desc/` dir or no match → `NoSuchVersion`; two matches →
  conflict error.
- **`descriptor_path`** (M5): `<root>/<type>/<version>.json` for the precedence
  ladder (flag > config > git-root default).
