# Phase M4 — Worker integration / in-process cutover (C8)

Derived from the capability commit map in `design/001` (M4/C8). The payoff of
the port: `cbsd-worker` stops shelling out to `python3 cbscore-wrapper.py` and
calls `cbscore::runner::run` (009) **in-process**, consuming its typed
`Result`/report natively. Owning design: 011 (worker integration); the runner it
now calls is 009; the descriptor translation it absorbs is 006/004. Read 011
closely before authoring — the panic-isolation wiring (single completion-task
owner; register-before-spawn) is subtle and load-bearing.

## Goal

The worker runs a build by spawning an in-process `runner::run` task instead of
a Python subprocess: it translates the `cbsd_proto::BuildDescriptor` to a
cbscore `VersionDescriptor` in Rust, streams output through the runner's
callback, maps the typed `Result` directly to `BuildFinished`, cancels via a
`CancellationToken`, and contains a panic at the tokio task boundary
(`JoinError::is_panic()`). Build isolation is unchanged (still a podman builder
container). `Revoked` becomes precise, partial reports survive failures, and the
worker image no longer carries Python.

## Commit 1: in-process cutover

The behavioral cutover. The subprocess mechanism and the in-process mechanism
are interdependent (the new path mounts `cbsbuild` into the builder, so the
binary must be present in the worker image in the **same** commit), so this is
one coherent change. The now-unused `python3`/`cbscore`/wrapper artifacts stay
in the image for exactly this commit and are removed in Commit 2. Owns: 011,
006/004 (translation), 009 (runner).

**Workspace link** — add `cbsd-worker`'s dependency on the `cbscore` crate
(`cbsd-rs/Cargo.toml` members already include the three crates from C0). This is
the link that makes `runner::run` an in-process call. (`panic = "unwind"` is
**already pinned** in `[profile.release]` from M0/C0 — verify it, do not
re-add.)

**Descriptor translation** (new worker-side glue, `cbsd-worker/src/build/`)
mirroring `cbscore-wrapper.py` exactly, now in Rust via 006's
`version_create_helper`: load `Config` **first** (it reads
`config.storage.registry`), set `config.paths.components = [component_dir]`;
build the `VersionDescriptor` — `component_refs`/`component_uri_overrides` from
`components[]`; `el_version` from `build.os_version` (`elN`→`N`, reject
otherwise); `registry` from `config.storage.registry.url` (**guard** if unset);
`image_name`/`image_tag` from `dst_image` (**guard** non-empty `tag`); `user_*`
from `signed_off_by`; write it to a temp file (RAII) for `runner::run`.

**Build task + completion task** (the panic-isolation model, 011 §"Crash
containment"):

- the **build task** (`tokio::spawn`) calls
  `runner::run(&desc_path, &config, opts)` and **returns its typed outcome** —
  it does **not** emit a terminal;
- the **completion task** (`tokio::spawn`) is the **sole owner** of the build
  task's `JoinHandle` and the **sole terminal producer**: it awaits the handle
  once and maps `Ok(result)` → mapped terminal, `Err(e).is_panic()` →
  synthesized `BuildFinished { Failure, error: "build task panicked" }` — so a
  panic still yields a terminal and `retire()` always runs (no permanent-busy
  wedge);
- **register-before-spawn ordering** (load-bearing): the handler
  `register_accepted(build_id, token, component_dir)` (token, **no** handle) →
  spawns both tasks → `attach_completion_task(build_id, completion_handle)`. A
  fast-failing build (bad `os_version`, missing registry, empty tag, config-load
  error) must find its record already present or the terminal is dropped as an
  orphan and the build hangs server-side.

**Native result → `BuildFinished`** (011 table) — `Ok(Some)`→`Success`+report,
`Ok(None)`→`Success`, `Cancelled`→`Revoked`, `NonZeroExit{report,..}`→`Failure`
**+ partial report**, `Podman`/other→`Failure`+error text. **Re-home the 64 KiB
`MAX_REPORT_SIZE` cap** (was in `output.rs`) onto the completion path before
placing a report in `BuildFinished`.

**Cancellation rewrite** (011 §"Cancellation rewrite") — a per-build
`CancellationToken` created once before registration; **all four** current
`exec.kill()` sites (`on_build_revoke`, `shutdown`, the two spool-teardown
sites) become `token.cancel()`; 009's `run` `select!`s it →
`podman_stop(name=ctr_name)` → `RunnerError::Cancelled`. **First-terminal-wins
arbitration**: the two spool sites have already synthesized an authoritative
`Failure`, so the completion path suppresses a second (`Revoked`) terminal when
`spool_exhausted`/`pending_terminal` is set. `setsid`, the SIGTERM→SIGKILL
escalation timer, and `DEFAULT_SIGKILL_TIMEOUT_SECS` are removed.

**Output streaming rewrite** — keep the batching (`BATCH_MAX_LINES = 50`,
`BATCH_FLUSH_INTERVAL = 200ms`, `BuildOutput` framing) but change its
**source**: lines arrive through `runner::run`'s async callback (009 `RunOpts`
log sink), not `ChildStdout`. Delete the `{"type":"result"}` sentinel parsing
and `WrapperResult` extraction.

**RunOpts** — `run_name = "cbs-" + trace_id.replace('-',"")[..12]`,
`replace_if_exists = true` (the deterministic name 009 expects from the worker;
amends `cbsd-rs/CLAUDE.md` invariant #4 — in-process arg, not `CBS_TRACE_ID`
env); `timeout` from the worker's configured build timeout **defaulting to 7200
s** (preserving the wrapper's `CBS_BUILD_TIMEOUT`; never falls through to 009's
4 h CLI default); the streaming callback; the build's `CancellationToken`.

**Removed Rust** — `executor.rs` in its entirety (subprocess spawn, `setsid`,
SIGTERM/SIGKILL, `pid`, `classify_exit_code` incl. the `137`/`143`→`Revoked`
heuristic); `output.rs`'s sentinel parsing and `ChildStdout` reading; the
`CBS_TRACE_ID`/`CBSCORE_CONFIG`/`CBS_BUILD_TIMEOUT`/`CBSCORE_PATH` env plumbing.
The supervisor's `executor: Option<BuildExecutor>` + `output_task` become
`cancel_token` + the **completion-task** `JoinHandle` (`Option`-and-`.take()`
for one await; `retire()`/`shutdown()` both tolerate `None`); `shutdown()`
awaits it as the **container-teardown barrier**. The `Supervisor` four-phase
state machine, spool/reconnect drain, and `component.rs` tarball handling are
**kept**.

**Image (this commit's half)** — `container/ContainerFile.cbsd-rs` **adds**
`COPY --from=rust-builder /build/target/release/cbsbuild /usr/local/bin/cbsbuild`
to the worker image (alongside the `cbsd-worker` COPY at line 158) so
`runner::run` has the binary to mount at `/runner/cbsbuild`. The existing
`worker-base` Python stack (`cbscore`/`cbscore-wrapper.py`) stays (now unused)
until Commit 2.

**Revert/window note (review S-4).** The commit message must document the
transient state this commit creates: the worker now builds **in-process**, but
the `worker-base` Python stack and `cbscore-wrapper.py` are still present in the
image (unused). Reverting **just this commit** restores the subprocess path
cleanly **only while Commit 2 has not landed**; once Commit 2 strips Python, a
revert of Commit 1 alone yields a non-functional worker (no in-process path and
no Python) — so the two commits revert as a pair after Commit 2.

**Testable:** result mapping for all five outcomes; **panic-isolation acceptance
gate** — a panic in the build task yields a `Failure` terminal via the
completion path, the supervisor reaches `retire()`, and a subsequent `BuildNew`
is accepted (process neither aborts nor wedges); a companion test asserts
`panic = "unwind"` is pinned; `BuildRevoke` → container stopped by name →
`Revoked`, while a spool-overflow keeps its `Failure` authoritative; fast-fail
(invalid `os_version`) still reaches the server (record registered before the
terminal); no-config-timeout passes 7200 s; an over-64 KiB report is dropped on
both Success and Failure; callback lines batch identically and no
`{"type":"result"}` line appears; **end-to-end** — a full build runs through the
in-process path (with `cbsbuild` mounted from the image) and report + logs reach
the server.

## Commit 2: image & wrapper cleanup (re-base off Python)

Remove the now-dead Python build path. Owns: 011 §"Dependency, image, and config
changes". **This is a base-image re-base, not a line deletion (review C-2).**
The real `container/ContainerFile.cbsd-rs` builds the worker on a **shared
`worker-base` stage** that is **`FROM python:3.13-alpine3.21`** (line 46) and
installs `uv` + `cbscore` + the Python venv; that stage feeds **both** the
production `cbsd-rs-worker` (line 156 `FROM worker-base`) **and** the dev
`cbsd-rs-dev-worker` (line 216 `FROM worker-base`). "Dropping python3" therefore
means:

- Re-base the worker image(s) off a **non-Python base** (e.g. `alpine:3.21`, as
  `rust-builder` and the server already use), and move the runtime tooling the
  worker still needs (`podman`/`buildah`/`skopeo`) into the new base — the
  in-process runner still shells out to them via 003.
- Drop the `uv`/`cbscore`/venv install and the `cbscore-wrapper.py` COPY (the
  wrapper currently lands at `/opt/cbsd-rs/cbscore-wrapper.py`, line 159);
  delete `cbsd-rs/scripts/cbscore-wrapper.py` from the repo.
- Decide whether `worker-base` is **retired** (folded into each worker stage) or
  **re-pointed** to the non-Python base — both `cbsd-rs-worker` and
  `cbsd-rs-dev-worker` must be updated together, and the dev worker's extra
  needs re-checked.

**Testable:** both the prod and dev worker images build with **no Python in the
worker layer**; a full in-process build still succeeds on the re-based image;
`cbscore-wrapper.py` is gone and nothing references it; `podman`/`buildah`/
`skopeo` are still present and functional in the worker.

## Notes for the plan-review

- **Commit 1 sizing.** The cutover (translation + two-task model +
  cancellation + output + supervisor edits + ContainerFile) is large and may
  exceed ~800 lines. It is genuinely atomic (the binary mount and the in-process
  call are interdependent; a partial cutover is non-functional), so a larger
  commit is justified — but confirm there is no clean internal seam the review
  prefers (e.g. landing the descriptor-translation module + its unit tests as a
  precursor commit **only if** it has a real consumer, else it is dead code).
- **One-commit alternative.** Commit 1 leaves the `worker-base` Python stack
  present-but-unused for one commit. Merging Commit 2 into Commit 1 avoids that
  window but folds the **base-image re-base** (review C-2 — re-basing the shared
  `worker-base` stage off Python, touching both prod and dev workers) into the
  already-large behavioral cutover. Recommended: keep split (isolates the image
  re-base, keeps the behavioral cutover reviewable on its own).
- Verify the **register-before-attach** ordering and the **single
  completion-task owner** exactly as 011 specifies — the naive "task sends its
  own terminal + `retire()` awaits it" deadlocks on panic. This is the
  highest-risk wiring in the milestone.
