# Phase M2 — Build (C2–C6)

Derived from the capability commit map in `design/001` (M2). This is the bulk of
the port: `cbsbuild build <descriptor>` going from spinning up the builder
container end-to-end (C2) to a full-parity build (C6). Work is sliced **by
working increment** — each commit makes the build _do_ one more thing — never by
layer. Owning designs: 009 (runner), 007 (builder), 004 (config/secrets/Vault),
005 (S3/releases), 008 (containers/images), 003 (podman/buildah/skopeo), 010
(build CLI). Read 009 and 007 before authoring code for any commit here.

## Goal

`cbsbuild build <descriptor>` reproduces the Python two-phase build with
behavioral parity (minus the Python bugs the port fixes): it spins the builder
container from `desc.distro`, runs the in-container `Builder` (prepare →
rpmbuild → sign → upload → container image), recovers `build-report.json` across
the scratch mount (partial-on-failure), and degrades gracefully when
`storage.s3` is absent. The host runner leaks no plaintext-secret temp files;
`--force` actually rebuilds; a failed `buildah push` errors; `can_sign` never
leaks.

## Commit 1 (C2a): keystone — builder container spins up end-to-end

The keystone: host orchestration + the in-container entry that writes a report,
wired through the CLI. The single largest seam; everything else extends it.
Owns: 009 (runner), 004 (config + secrets-file format), 003 (podman), 010
(`build` + `runner build`).

**Config & secrets-file types** (`cbscore-types` + `cbscore`, 004 — the
_format_, not resolution): `Config` (+ `PathsConfig`, `StorageConfig`,
`S3StorageConfig`/`S3LocationConfig`, `RegistryStorageConfig`, `SigningConfig`,
`LoggingConfig`), `VaultConfig`, `Secrets` (the four credential-family maps with
the custom per-family `Deserialize` discriminators, incl. the added git `type:`
tag) — all `rename_all = "kebab-case"`, `schema-version` machinery, YAML-primary
load/store/merge, `Config::get_secrets()`/`get_vault_config()`. `ConfigError`,
`SecretsError`. **No `SecretsMgr`/Vault resolution yet** (C3/C4); the runner
only _marshals_ the secrets file.

**podman wrapper** (`utils::podman`, 003): `podman_run(image, RunArgs)`,
`podman_stop(name, timeout)`, `PodmanError`, the fixed prelude +
mount/env/device flags, the cidfile→`podman_stop` timeout hook, the
dual-deadline behavior.

**runner** (`cbscore::runner`, 009):
`run(desc_path, &Config, RunOpts) -> Result<Option<BuildArtifactReport>, RunnerError>`
— input validation (`VersionDescriptor::read`, cbsbuild-artifact
exists+executable), component aggregation into one temp dir, **secrets-file
marshal** (`config.get_secrets()` → temp file), config path-rewrite to
`/runner/...` (clear `logging` to `None`), `podman_run` of `desc.distro` with
the mount table/env (`CBS_DEBUG=<1|0>`, **no host `HOME`**)/devices/flags,
**report round-trip** (read `<scratch>/build-report.json` **before** the rc
check, then unlink), `gen_run_name`, the `select!` on the caller's
`CancellationToken` → `podman_stop(name=ctr_name)`.
`RunnerError { NonZeroExit { report, stderr }, Podman, Cancelled, + marshalling }`.
**Every temp input guarded by RAII** (invariant 5 / 009 fixes the Python
plaintext-secrets + config leak); the non-zero-exit variant **carries the
partial report** (009 fixes the discard).

**Minimal in-container Builder** (`cbscore::builder`, 007 skeleton):
`runner build` constructs a `Builder` that (for the keystone) writes a
`skipped`/empty `BuildArtifactReport` to the scratch mount and returns — enough
to prove the round-trip. Full stages land in C2b–C6.

**CLI** (`cbsbuild`, 010): the `build <DESCRIPTOR>` command (flag table from 010
§build — `--timeout` default 14400, `--sign-with-gpg-id`/`--sign-with- transit`
folding into `config.signing`, `--log-file` pre-check, `--skip-build`,
`--force`, **value-taking** `--tls-verify` via the shared BOOL parser;
**dropped** `--cbscore-path`, `-e/--cbs-entrypoint`) → `runner::run`; **no
secrets marshalling in `build`** (010 fixes the dead `cmd_build` plaintext
write). The hidden `runner build` command (010): the **`HOME` hook first** (set
`HOME=/runner` iff unset/empty or `/`), read the descriptor, construct+run the
`Builder`; exit 0/non-zero only (009 reads `rc != 0`).

**Testable:** `cbsbuild build <desc>` against a real `rockylinux:9` builder
spins the container, runs the mounted `cbsbuild runner build`, and returns a
(skipped) report; mount/argv/env assembly matches 009; a report on a non-zero
exit **rides `RunnerError::NonZeroExit`** (not discarded) and the scratch file
is unlinked; firing the cancel token stops the container by name → `Cancelled`;
**no plaintext-secrets temp file survives** any path;
`Config`/`Secrets`/`VaultConfig` round-trip + golden-file parity (git `type:`
tag selects the right variant).

## Commit 2 (C2b): `prepare_builder` — in-container toolchain install

Owns: 007 stage 1 (`prepare.py::prepare_builder`). `Builder.prepare_builder()` —
`dnf update`, `epel-release`, enable `crb`, install the toolchain (`git`,
`rpm-build`, `rpmdevtools`, `createrepo`, `rpm-sign`, `buildah`, `skopeo`,
`ccache`, … the full list), then `cosign` from the pinned GitHub release RPM
(tolerating exit 2 "already installed"). Each step streams via the subprocess
`out_cb`. **Testable:** a build runs `prepare_builder` to completion in the
container; the toolchain (incl. `cosign`) is present; re-running tolerates
already-installed.

## Commit 3 (C3): build compiles a component's RPMs

Owns: 007 (component model, prepare, rpmbuild), 003 (C3 git ops), 004
(`git_url_for`). **Foundational git + secrets-mgr land with their first
consumer** (`prepare_components`).

- **git C3 ops** (`utils::git`, 003): `git_clone` (mirror + update-in-place,
  `repo` is a `CmdArg`/`SecureUrl`), `git_checkout` (worktree on a new branch,
  `/`→`--` + random suffix), `git_remove_worktree`, `git_apply`, `git_get_sha1`.
- **SecretsMgr (minimal)** (`utils::secrets::mgr`, 004): wraps `Secrets`;
  `git_url_for(url) -> Guard<CmdArg>` with the **plain** ssh (temp `~/.ssh`
  key + alias, RAII guard) / https / token / **no-match** cases (longest-prefix
  URI match). **Vault-backed git resolution (`vault-ssh`/`vault-https`) defers
  to C4a** — it needs the Vault client. The C3 **commit message must state this
  partial explicitly** (review S-2): "git_url_for resolves plain/no-match
  secrets only; vault-git completes at C4a". Plain + no-match covers public and
  plain-cred repos — the common case.
- **component model** (`cbscore::core::component`, 007): `CoreComponent`
  (`cbs.component.yaml`, **unversioned**), `load_components` (skip dir w/o
  manifest = warn; malformed = error-skip), the four-script contract.
- **prepare** (007 stage 1): `prepare_components` — per-component **parallel**
  (`JoinSet`, abort-on-first-error) clone → checkout → patch-apply, recording
  `BuildComponentInfo`; the intricate `_get_patch_list` selection (numeric
  prefix; exact/minor/major version dirs; deepest-first) ported faithfully.
- **rpmbuild** (007 stage 2): `_install_deps` (**sequential** — mutates dnf
  state), `build_rpms` (**parallel**, per-component topdir, `CES_CCACHE_PATH`
  when configured, `skip_build` makes the topdir but runs nothing).

**Testable:** a build clones a **plain/public** component repo through
`git_url_for` (the C3 scope), checks it out, applies the right patch set (golden
test on ordering), and produces RPMs under `<rpms>/<name>/<version>/RPMS`;
`load_components` skips/error-skips correctly; a single-component failure aborts
the fan-out (fail-fast); `_install_deps` runs in order. (A vault-git clone is a
C4a test, not C3.)

## Commit 4 (C4a): Vault + SecretsMgr + S3 client (credential & object infra)

The credential/object **infrastructure** layer — split out from the build stages
that consume it (review S-1: a single C4 is >800 lines with two new heavy deps).
This is a foundation commit validated by **live round-trip tests** (MinIO + a
Vault dev server), the repo's established pattern for IO-primitive commits; its
production consumers land in Commit 5 (C4b). This is also where **`aws-sdk-s3`
and `vaultrs` first enter `cbscore`'s manifest** (the C0 musl probe proved them
already). Owns: 004 (Vault client + SecretsMgr complete + `s3_creds` +
`gpg_signing_key`), 005 (S3 client).

- **TLS provider + trust source (per design 012 carve-out).** Both clients do
  TLS (S3 endpoint, Vault). Pin a musl-clean **rustls** provider (`ring`/
  `aws-lc-rs`, no `openssl-sys`) **and** the **trust source**: prefer the
  **system trust store** (`rustls-native-certs`) so internal/private CAs (a
  self-hosted Vault / MinIO / RGW) are honored — which means the builder image
  must ship `ca-certificates` (ensure it in `prepare_builder`, C2b/007). This is
  a runtime-userspace dependency static linkage does **not** cover (012).
- **CI gate — extend the C0 static-musl assertion to the shipped binary.** C0's
  probe (`ci/musl-probe`) proved only its **own** graph links static (no
  `openssl-sys`/`aws-lc-rs`); now that `aws-sdk-s3`/`vaultrs` are real `cbscore`
  deps, the CI must additionally assert the **shipped `cbsbuild`** still links
  fully static-musl with neither crate's TLS stack pulling `openssl-sys` or
  `aws-lc-rs`. A feature-flag slip here (e.g. enabling `default-https-client`,
  which pulls `aws-lc-rs`, or a `native-tls` edge) would pass the probe yet
  break the real binary — so the probe alone is insufficient at C4a.
- **Vault client** (`utils::vault`, 004): `vaultrs`-backed; auth order **AppRole
  → userpass → token**; KV v2 mount **`ces-kv`** (pinned); `read_secret(path)`;
  `VaultError`. Built from `VaultConfig`.
- **SecretsMgr (complete)** (004): vault-backed resolution for every family
  (**completing the C3 vault-git variants** now the Vault client exists);
  `s3_creds(url)` (exact key), `gpg_signing_key(id)` (exact id, RAII temp
  GNUPGHOME keyring), `has_*` predicates; `SecretsMgrError`.
- **S3 client** (`utils::s3`, 005): `aws-sdk-s3` with **explicit static creds**
  from the secrets store, scheme-normalized custom `endpoint_url`, placeholder
  region, **`force_path_style(true)`** (invariant 9 / H4); the **write** path
  `s3_upload_str_obj`/`s3_upload_json`, `s3_upload_files(public)` (file-backed
  `ByteStream`, single PUT); and the **download primitive**
  `s3_download_str_obj` (**404 → `None`**, not an error) — its first consumer is
  `release_desc_upload`'s read-modify-write in C4b, so it lands here, not at C6
  (review S-7). `s3_list` (the directory listing) is **not** here — its first
  consumer is `versions list` (M3/C7). `S3FileLocator`, `S3Error`.

**Testable:** **MinIO parity** (the H4 gate) — `put_object`/`get_object` round-
trip with injected creds, custom `endpoint_url`, and path-style addressing; a
download of a missing key returns `None` (not an error); **Vault auth order**
picks AppRole when several methods are configured; a `vault-git`/`gpg` secret
resolves its `ces-kv` `key`; resolved credentials render redacted in any
log/`Debug` path.

## Commit 5 (C4b): sign RPMs + createrepo + upload + releases write

The build stages that consume the C4a infrastructure — this is the commit that
makes the build **sign and publish**. Owns: 007 stages 3–4, 005 (release ops
write path). Consumes C4a's `SecretsMgr`/S3 client (no dead code: C4a's
resolvers and S3 ops get their production callers here).

- **signing** (007 stage 3): `sign_rpms` (only when `config.signing.gpg` set) —
  resolve keyring via `gpg_signing_key`, `rpm --addsign` with the `--define`
  keyring/name and the **`SecureArg`-wrapped** loopback passphrase; parallel
  across components, sequential within.
- **upload + releases write** (007 stage 4, 005): `createrepo`; `s3_upload_rpms`
  (**parallel**, the asymmetric `RPMS`-flattened / `SRPMS/`-prefixed S3 layout
  `<loc>/<name>/rpm-<version>/el<el>.clyso/...`); `get_component_release_rpm`
  (runs `get_release_rpm.sh <el>`, **no cwd** — ROADMAP F3 quirk reproduced);
  `release_desc_upload` (read-modify-write merge into `builds[arch]`, using
  C4a's `s3_download_str_obj` to load the existing release),
  `release_upload_components` (parallel). **Silent component drop** (no
  `get_release_rpm.sh` → component dropped from the release though its RPMs are
  uploaded) reproduced **with a clear error log** (ROADMAP F2 quirk).

**Testable:** with `storage.s3` configured, a build signs RPMs (passphrase never
in a log line) and uploads RPMs + `<loc>/<version>.json` +
`<loc>/<name>/<version>.json`; a re-build of a different arch **merges** into
the existing `builds` map (RMW) rather than overwriting; with no `storage.s3`
the build runs but uploads nothing.

## Commit 6 (C5a): assemble the container image

Owns: 008 (`ContainerDescriptor`, `ComponentContainer`,
`ContainerBuilder.build`).

- **`ContainerDescriptor`** (`containers::desc`, 008): external **unversioned**
  `container.yaml`, **template-then-parse** (`{version}`/`{el}`/`{git_sha1}`/…
  interpolated before YAML parse); the **repos union** discriminated by `source`
  scheme (`file://`/`http(s)://`/`copr://`) via custom `Deserialize` (no added
  tag); `ContainerConfig`/`ContainerPre`/`ContainerPackages`/`ContainerScript`.
- **`ComponentContainer`** (`containers::component`, 008): version-proximity
  descriptor selection (`_get_container_desc`, golden test); `apply_pre`
  (scripts copy/run/rm; `keys` `rpm --import`; dnf-vs-URL package split; repos
  file/URL/COPR), `get_packages` (required union; optional ignored),
  `apply_post`, `apply_config` (env keys **upper-cased**);
  `find_path_relative_to`.
- **`ContainerBuilder.build`** (`containers::build`, 008): **strictly
  sequential** get_components (interpolation vars; `git_ref` == version quirk) →
  `buildah_new_container` → apply_pre → one `dnf install` of the required union
  (`--enablerepo=crb`, weak-deps off) → apply_post → apply_config. URI helpers
  (`get_container_image_base_uri`, `get_container_canonical_uri`).
  `ContainerError`.

**C5a/C5b split is a sizing decision, kept (review S-3).** C5a (descriptor
parse + version-proximity selection + the sequential assembly) is ~600–800 lines
on its own; merging the buildah `finish`/push (C5b) would push the commit well
over budget. The split is justified by size, not layering: C5a is a genuine,
**independently testable** increment (the assembled buildah container is
inspected via `build()` integration tests — committed image config, installed
packages, applied repos — without a registry push), and keeping C5b separate
isolates the push-rc **bug fix** for focused review. The image only reaches a
registry at C5b.

**Testable:** `container.yaml` template-then-parse + repos-union golden tests;
descriptor version-proximity selection; `build()` runs the sequence in order,
skips a component absent from the release build, errors on an empty set;
`apply_pre` variants (scripts/keys/dnf-vs-URL/file-URL-COPR repos) apply.

## Commit 7 (C5b): push the image

Owns: 003 (buildah wrapper), 008 (`finish` sequence), 004 (`registry_creds`).
Lands the buildah push path; kept separate from C5a so the push-rc fix is
isolated and testable (see C5a's split rationale).

- **buildah wrapper** (`utils::buildah`, 003): `buildah_run`, `BuildahContainer`
  (`set_config`/`copy`/`run`/`finish`), `buildah_new_container`, `BuildahError`.
- **`registry_creds(uri)`** (004, longest-prefix) resolver.
- **`finish`** (008): OCI annotations → `commit --squash` to the canonical URI →
  `registry_creds` (warn+unauth on none) →
  `push --digestfile [--creds <Password>]` → read digest. **Push-rc fix (Python
  bug):** check the push rc **immediately**, fail with a push-specific error,
  require a **non-empty digest** before any signing (Python logs-only and can
  sign the wrong ref).

**Testable:** `finish` commits the canonical URI and pushes with a digest;
`--creds` is redacted; **a failed `buildah push` errors** (not silent success);
no registry creds → unauthenticated with a warning; the assembled image appears
in the registry.

## Commit 8 (C6): full parity — skip-if-exists, reuse release, transit-sign

Owns: 003 (skopeo wrapper), 005 (release read ops), 007 (orchestration
reuse/force), 008 (cosign transit). Completes `Builder.run`'s decision flow.

- **skopeo wrapper** (`images::skopeo`, 003): `skopeo_image_exists`/
  `skopeo_inspect` (`--creds <Password>`, exit-2/"not found" → `ImageNotFound`),
  `SkopeoError`.
- **release read ops** (005): `check_release_exists`,
  `check_released_components` (**parallel**, fail-fast) — built on C4a's
  `s3_download_str_obj` (the download primitive landed there for
  `release_desc_upload`'s RMW). (`s3_list` is **not** needed here — its first
  consumer is `list_releases`; it lands in M3/C7.)
- **orchestration** (007 `Builder.run` decision flow): **skopeo short-circuit**
  (image exists → `skipped: true` report, return early); **existing-release
  check** (reuse a complete release) with the **`--force` fix** (force ignores
  an existing `release_desc` so `_build_release` actually runs — Python's force
  is a no-op when a complete release exists); `check_released_components` reuse
  (skipped under `--force`).
- **cosign transit** (008 `images::signing`): `transit(id) -> (mount, key)`
  resolver (004); `async_sign` — `cosign sign --key=hashivault://<key>` with
  `PasswordArg` registry creds and `VAULT_ADDR`/`VAULT_TOKEN`/
  `TRANSIT_SECRET_ENGINE_PATH` via the **environment** (never argv, never
  logged); **`can_sign` hardened** to return `False` on missing creds (Python
  leaks a `ValueError`). `SigningError`.

**Testable:** an existing image short-circuits to a `skipped` report; an
existing complete release is reused, but **`--force` rebuilds even when one
exists** (the fix); `check_released_components` omits absent components;
transit-sign builds the cosign command with the Vault env and the token never
appears in a log; `can_sign` returns `False` (no raise) without a Vault/transit
key.

## Notes for the plan-review

Resolutions of the v1 plan-review findings (now folded into the breakdown
above):

- **C4 sizing (S-1) — resolved.** The single C4 is split into the committed
  **C4a** (Vault + SecretsMgr + S3 client, validated by MinIO/Vault round-trips)
  and **C4b** (signing + createrepo + upload + releases write). The build is now
  **8 commits** (C2a, C2b, C3, C4a, C4b, C5a, C5b, C6).
- **Vault-git seam (S-2) — resolved.** C3 ships plain/no-match `git_url_for`
  only (its commit message states the partial); vault-git completes at C4a with
  the Vault client. C3's acceptance test clones a plain/public repo.
- **C5a/C5b split (S-3) — decided: keep split** (sizing-driven, C5a
  independently testable via `build()` inspection; isolates the push-rc fix).
- **S3 download primitive (S-7) — resolved.** `s3_download_str_obj` lands at
  **C4a** (first consumer: `release_desc_upload` RMW), so M3/C7 reuses it from
  C4a, not C6; C6 adds only the release-level read ops on top.

Standing watch-items for implementation:

- **Reproduced quirks** (ROADMAP F2/F3 — silent component drop;
  `get_release_rpm` no-cwd) are reproduced **with clear error logs**, not fixed
  mid-port. Confirm the logs are present and the ROADMAP entries are
  cross-referenced in the code.
- **C4a as a foundation commit.** Its S3 client + Vault client have only test
  consumers until C4b/C6; this is justified by the live MinIO/Vault round-trip
  tests (the repo's IO-primitive-commit precedent), not a layering shortcut.
