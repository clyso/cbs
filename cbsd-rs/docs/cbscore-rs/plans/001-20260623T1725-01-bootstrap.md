# Phase M0 — Bootstrap & de-risk (C0)

Derived from the capability commit map in
`design/001-20260621T0949-architecture-and-implementation-spine.md` (M0/C0). C0
is an explicit one-time **de-risk spike** for blocker B1, not a feature commit:
it stands up the three new workspace crates and proves the static-musl toolchain
(including the heaviest dependencies, `aws-sdk-s3` and `vaultrs`) **links fully
statically** — no dynamic libc, no `openssl-sys` — **before** any subsystem work
begins. Static linkage is the real gate; it is distro-independent (a static musl
binary makes raw syscalls and carries its own libc), so the proof is a
**link-time** property, not "runs on distro X".

## Goal

A compiling `cbsd-rs/` workspace with the three new members (`cbscore-types`,
`cbscore`, `cbsbuild`) building under `cargo build --workspace`; a **verifiably
static-musl** `cbsbuild` (`ldd` → "not a dynamic executable") that prints its
version; and a CI job proving `aws-sdk-s3` + `vaultrs` compile and link to a
fully static `x86_64-unknown-linux-musl` binary. No subsystem logic yet — this
is the scaffold and the toolchain proof the rest of the port stands on. (Rocky 9
remains the runtime **default** `--distro` per design 010, but it is **not** a
special test target — see the acceptance notes.)

## Commit 1: workspace skeleton + `cbsbuild` binary + panic pin

Stand up the three crates and make `cbsbuild` a real build output. Owns: design
001 (crate layout, build target, failure-isolation invariant 6).

**Workspace root** (`cbsd-rs/Cargo.toml`):

- Add `cbscore-types`, `cbscore`, `cbsbuild` to `[workspace] members` (currently
  `cbsd-common, cbsd-proto, cbsd-server, cbsd-worker, cbc`).
- Add a pinned `[profile.release] panic = "unwind"` (invariant 6 / decision 2 —
  the workspace sets no `panic` today, so `unwind` already applies; pinning
  prevents a future flip to `abort` silently voiding the in-process panic
  isolation). Apply at the workspace root so every member inherits it.
- Leave `resolver = "2"` as-is unless a feature-unification issue surfaces
  (bumping to `3` would re-unify features across the existing five crates — out
  of scope for a scaffold commit; note it for a later, isolated change).

**`cbscore-types`** (lib, zero-IO) — minimal skeleton: `lib.rs` with a crate doc
comment and the tracing-target string constants (002) as the one piece of real,
immediately-referenceable content. Deps limited to `serde`, `serde_json`,
`thiserror`, `camino`, `uuid` (declared but only what compiles cleanly with no
dead `#[allow]`). No wire types yet — those land with their first consumer in
M1/M2.

**`cbscore`** (lib) — minimal `lib.rs` with a crate doc comment re-exporting
nothing yet; depends on `cbscore-types`. No `tokio`/`aws-sdk-s3`/`vaultrs` in
the manifest yet (they enter at the commit that first uses them — subprocess at
M1, S3/Vault at M2/C4), so C0 carries no dead dependency edge.

**`cbsbuild`** (bin) — `main.rs` with a clap root that parses `--version` (and
the `--config`/`--debug` globals as no-op placeholders wired in M1/010) and
prints the crate version. `anyhow` at the `main` boundary (001). Depends on
`cbscore`. This is the artifact the runner later mounts into the builder.

**ContainerFile** (`container/ContainerFile.cbsd-rs`, repo root) — the
`rust-builder` stage is `FROM alpine:3.21` and runs
`cargo build --release --workspace` (no `--target`; on Alpine the host triple is
musl, so binaries are static-musl implicitly). Adding `cbsbuild` to `members`
makes it a build output with no ContainerFile change required here; the
worker-image `COPY` of `cbsbuild` is deferred to M4/C8 (it has no consumer until
then).

**Testable:**

- `cargo build --workspace` succeeds and produces `target/release/cbsbuild`.
- `cbsbuild --version` prints the version; `cargo clippy --workspace` and
  `cargo fmt --all --check` are clean.
- **Acceptance gate (B1) — primary, distro-independent:** the `cbsbuild` built
  on `alpine:3.21` is **fully static** — `ldd target/release/cbsbuild` reports
  "not a dynamic executable" and `file` reports "statically linked". This is the
  real proof (no libc/OpenSSL linkage) and needs no EL image.
- **Runtime smoke (secondary):** the static binary runs and prints its version
  in a **glibc EL builder container** — catching a "not actually static / still
  wants an ELF interpreter" regression that a musl host would hide. The distro
  is arbitrary for a static binary; use a representative EL image (Rocky 9 as
  the current default, or equivalently Alma / EL10 — running el9 **and** el10
  makes the distro-independence explicit).

## Commit 2: CI static-musl linkage proof (aws-sdk-s3, vaultrs)

Prove the two heaviest, most musl-risky dependencies link statically and run on
the musl target — the core of the B1 de-risk — **without** adding them to any
shipped crate's manifest (design 001: the proof "lives in a CI job … not as a
shipped dependency edge"; `aws-sdk-s3` enters `cbscore` at C4/C6, `vaultrs` at
C4).

**Probe artifact** — a standalone, CI-only probe **excluded from the default
workspace build** (e.g. `cbsd-rs/ci/musl-probe/` with its own `Cargo.toml`,
listed under `[workspace] exclude` or built via an explicit `--manifest-path`),
so `cargo build --workspace` never pulls these deps into the shipped graph. The
probe constructs (but does not exercise against a live endpoint) an `aws-sdk-s3`
client with a custom endpoint + static credentials provider +
`force_path_style(true)` (the exact shape C4/C5 use — invariant 9) and a
`vaultrs` client, so the link includes their TLS/crypto stacks. It must use a
**musl-clean crypto provider** (e.g. `rustls` with `ring`/`aws-lc-rs` selected
for musl), not OpenSSL dynamic linking.

**CI — must be a PR/push gate, not the release workflow (review C-1).** The
repo's only workflow today, `.github/workflows/release-cbsd-rs.yaml`, triggers
**solely** on `push: tags: v*` and `workflow_dispatch` — so a job added _there_
would run at release time, **after** M1–M4, defeating C0's "proven before any
subsystem work begins" mandate. The proof must live in a **new
push/pull_request-triggered workflow** (e.g.
`.github/workflows/ci-cbsd-rs.yaml`, `on: [push, pull_request]`) — which the
repo lacks and this commit adds. That workflow installs the musl target and runs
`cargo build --release --target x86_64-unknown-linux-musl --manifest-path cbsd-rs/ci/musl-probe/Cargo.toml`,
asserts the binary is fully static (`ldd` → "not a dynamic executable"), then
smoke-runs it in a representative glibc EL container (e.g. `rockylinux:9`, but
any EL image is equivalent for a static binary). (The musl release matrix in
`release-cbsd-rs.yaml` reuses the proven toolchain at release time; the **gate**
is the new PR/push workflow.)

**Testable:**

- The new PR/push CI workflow builds the probe for `x86_64-unknown-linux-musl`
  with zero linker errors and a fully static binary (`ldd` reports "not a
  dynamic executable") — and it runs on a PR, **not** only on a release tag.
- The probe binary is **fully static** (`ldd` → "not a dynamic executable") —
  the real, distro-independent gate — and additionally smoke-runs in a
  representative glibc EL container (Rocky 9 / Alma / EL10 are equivalent for a
  static binary), constructing both clients and exiting 0.
- `cargo build --workspace` (the shipped graph) still does **not** depend on
  `aws-sdk-s3`/`vaultrs` — verified by `cargo tree` showing them absent until
  C4/C6.

## Notes for the plan-review

- C0 is scaffolding-shaped by design (000-review D1); its testable artifacts are
  the **link-time static check** and the musl-probe CI job. Confirm the probe
  genuinely keeps the deps out of the shipped graph (the no-dead-dependency
  claim) and that the musl crypto-provider choice (rustls + ring/aws-lc-rs, no
  `openssl-sys`) is pinned, not incidental.
- **Acceptance is governed by design 012**
  (`design/012-…-static-musl-acceptance- and-distro-independence.md`), which
  extends/refines design 001's build-target section and invariant 7. A
  static-musl binary has no libc linkage and is distro-independent (it runs the
  same on Rocky/Alma/EL9/EL10), so the operative gate is the **link-time
  staticness** check (`ldd`/`file`) and the runtime smoke run uses a
  _representative_ EL image, **not** Rocky 9 as a canonical target. The C0
  acceptance criteria above derive from **012**, not from this plan's prose —
  001 carries forward pointers to 012 at both affected places.
- Whether Commit 1 and Commit 2 stay split or merge into one de-risk commit is a
  sizing call for review — both are small; keeping them split isolates the
  toolchain-proof churn from the crate scaffold.

## As-built (C0 complete)

Landed as `7b5086c` (scaffold) and `a18c1f9` (de-risk). Where the realisation
refined this plan's prose:

- **Crypto provider: rustls + `ring`** (chosen over `aws-lc-rs`). The probe and
  C4's S3/Vault clients pin it: `aws-sdk-s3` drops `default-https-client` (which
  pulls `aws-lc-rs`) and builds the connector explicitly via
  `aws-smithy-http-client`'s `rustls-ring`; `vaultrs`/`rustls` →
  `reqwest`/`rustls-tls` → ring. Verified graph: single `rustls` 0.23, **no
  `aws-lc-rs`, no `openssl-sys`**.
- **Static check is `file`/`readelf`, not `ldd`.** musl `ldd` is unreliable — it
  prints the `ld-musl` loader path even for a static-PIE — so the gate asserts
  **no `INTERP` segment** (`readelf -l`) plus `file` "static-pie linked", and
  the cross-distro smoke run. (This plan's `ldd → "not a dynamic executable"`
  wording is glibc phrasing; it holds on a glibc host but not on the Alpine
  build host.)
- **Built natively in Alpine** (`docker run alpine:3.21`, no `--target`)
  mirroring `container/ContainerFile.cbsd-rs`'s rust-builder, not ubuntu +
  `musl-tools` + `--target` (the plan's illustrative command). The probe is kept
  out of the shipped graph via `[workspace] exclude`; its `Cargo.lock` is
  committed and the gate builds `--locked` for determinism.
- **EL smoke ran on `rockylinux:9` (el9) and `almalinux:10` (el10)** — the
  static binary runs on both, making distro- and EL-version-independence
  explicit.
- **`cbscore-types` ships with empty `[dependencies]`** (the tracing-target
  constants need none); `serde`/`uuid`/etc. land with their first consumer in
  M1+, which is stricter than this plan's "deps limited to …" wording and avoids
  a dead dependency edge.
- **Forward note (C4a):** the probe proves only its **own** graph links static;
  when `aws-sdk-s3`/`vaultrs` become real `cbscore` deps at C4a, CI must extend
  the static-musl assertion to the **shipped `cbsbuild`** (see
  `001-…-03-build.md`, C4a) — a feature slip there would pass the probe yet
  break the real binary.
