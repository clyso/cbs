wording/spec fix, not structural, so it keeps the design at go-with-changes and
is the one item that blocks a clean "go."

### F1 — `get_image_desc` framing (partially resolved)

The doc was reframed correctly. 006:137-143 now states plainly that the Python
`get_image_desc` is **broken**, names the raw-string filter at `desc.py:50`
(`r"^.*{m[1]}.*.json"`, not an f-string → `{m[1]}` never interpolates → no
candidate matches → **always** raises `NoSuchVersionError`), and the fidelity
note 006:218-220 says the port implements the _intended_ lookup, not the
behavior. The "literal `v` required" claim (006:148) is faithful to `desc.py:43`
(`r".*v(\d+\.\d+\.\d+).*"`). The "skip malformed, matching `list_releases`'
leniency" claim (006:152-153) is correct as a stated **deviation** from Python
(`desc.py:72-74` re-raises; `list_releases` `s3.py:276-279` logs and `continue`s
— I confirmed both directly), and the doc labels the Python re-raise honestly at
006:142-143. That is the F1 the mandate asked for, and it lands.

**What is not resolved:** 006:143 says the Python "also returns the last-parsed
(not the matched) descriptor on the **dead multi-match path**." This is
inaccurate. In `desc.py:68-89`, a **second** match raises `ImageDescriptorError`
(`desc.py:77-82`); it does not return a descriptor. The last-parsed-not-matched
return (`desc.py:71` reassigns `desc` on every candidate; `desc.py:89` returns
it) manifests on the **single-match path when the matched file is not last in
walk order** — i.e. multiple _candidates_, one _match_ — which is exactly how v1
phrased it ("≥2 candidate files"). The rewrite compressed "multi-candidate" into
"multi-match" and got it wrong, and it now contradicts 006:154's own "Two
matches → `ImageDescriptorError` (conflict)". The mandate explicitly asks about
multi-match accuracy, so this is called out. Fix: reword 006:143 to "returns the
last-parsed candidate (not the matched one) when several files match the
filename filter but only one lists the version" — and drop "multi-match", which
the next paragraph correctly reserves for the conflict case.

006:30-32 now reads "raises `MalformedVersion` on a malformed string; for a
well-formed string it returns `"<major>.<minor>.<patch>"` when patch is present,
otherwise `None`." This matches `utils.py:90-104` exactly: the `parse_version`

`Result<Option<String>, MalformedVersion>` rather than dropping the malformed
case. Resolved.

## New findings (by severity)

### N1 — UUIDv7 / `v`-less version × the trailing `get_image_desc` check (Important)

**Design claim** (006:157-159): "`versions create` treats `NoSuchVersion` as a
warning ('image descriptor missing'), not a failure — but, with the fix, that
warning fires only when a descriptor is genuinely absent (today it fires
unconditionally)."

**What the source shows.** In `cmds/versions.py:109-117`, after writing the
descriptor, `versions create` unconditionally calls
`get_image_desc(desc.version)` on the **raw** version (`create.py:136` stores
the unmodified input as `desc.version` — not normalized). `get_image_desc`'s
first step is `re.match(r".*v(\d+\.\d+\.\d+).*", version)` (`desc.py:43`), which
**requires a literal `v`** before `M.m.p`; on no match it raises
`MalformedVersionError` (`desc.py:44-45`). I confirmed in `errors.py` that both
`MalformedVersionError` (`errors.py:30`) and `NoSuchVersionError`
(`errors.py:36`) subclass `CESError` (`errors.py:18`). The call site catches
`NoSuchVersionError` first (prints the benign "missing" message,
`cmds/versions.py:112-113`) and `CESError` second (prints "error obtaining image
descriptor … ", `cmds/versions.py:114-117`). So a `MalformedVersion` lands on
the **error** branch, not the "missing" branch.

Therefore, for two classes of valid version the trailing check does **not**
produce the "missing" warning the design promises:

- **Every UUIDv7** — the headline new feature (006:88-109). A UUIDv7 has no
  `v<M.m.p>`, so `get_image_desc` always raises `MalformedVersion`. (It also
  could never appear in any descriptor's `releases`, so the lookup is
  meaningless for a UUID in the first place.)
- **Any `v`-less version string** — `_validate_version` accepts `20.2.1`
  (`parse_version`'s `v?` is optional, `utils.py:49`); the design's own title
  test uses `20.2.1` (006:227). `desc.version` keeps that raw `20.2.1`, which
  `desc.py:43` rejects → `MalformedVersion` → "error obtaining image
  descriptor", never "missing".

**Gap.** 006 is silent on what `versions create` does with a `MalformedVersion`
(or any non-`NoSuchVersion` error) from the trailing check, and 006:157-159
overclaims that the only outcomes are "found" or the benign "missing" warning.
For the primary new feature (UUIDv7) the realistic outcome under the described
contract is the **error** message on every create. This is the same class of
"verify the claim against the source" gap that F1 targeted; the F1 rewrite
introduced it by promising a clean post-fix behavior it did not trace through
the UUIDv7 path.

**Recommended change.** State explicitly that `versions create` **skips** the
image-descriptor lookup when `desc.version` is a UUIDv7 (a UUID cannot appear in
any descriptor's `releases`), and pin how a `v`-less version string is handled
by the trailing check — either normalize before the lookup, or treat the
`MalformedVersion` from `get_image_desc` as the same benign "missing" outcome
rather than an error. Then narrow 006:157-159 to the version-string path it
actually covers.

### N2 — (folded into F1; no separate deduction)

The multi-match inaccuracy at 006:143 is scored once under F1 residue above to
avoid double-counting. It is restated here only because the mandate asks
specifically about multi-match accuracy: **multi-match raises**
`ImageDescriptorError` (`desc.py:77-82`); it does not "return the last-parsed
descriptor." See F1 for the precise reword.
