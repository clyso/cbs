# Review — 006 Versions (design) v1

Adversarial design review of
`cbsd-rs/docs/cbscore-rs/design/006-20260621T1831-versions.md`. The mandate:
verify every claim against the real Python source, never trust the author, and
hunt title/format parity bugs, the no-S3-config edge, UUIDv7 validation gaps,

## Verdict

**go-with-changes.**

The architecture is sound and the bulk of the parity claims hold up against the
Python source — including the one the mandate pushed hardest on (title
generation), which is faithful, not buggy. But the document fails its own
"verify every claim" bar in one place: it describes `get_image_desc` as if the
Python were correct, when the Python is in fact broken three different ways.
That, plus two silent edges (the no-S3-config path in C7, and the UUIDv7 title
timestamp source), are fixable with wording changes; none is a structural no-go.

## Findings (by severity)

### F1 — `get_image_desc` is described as parity, but the Python is broken (Important)

**Design claim** (006:130-138): `get_image_desc(version)` extracts
`v<M>.<m>.<p>` via regex (else `MalformedVersion`), walks
`<git-repo-root>/desc/` for `*.json` whose **name contains the `M.m.p` string**,
parses each candidate, and returns the one whose `releases` list contains the
exact version; two matches → conflict, none → `NoSuchVersion`. This reads as a
faithful description of the Python.

**What the Python actually shows:**

1. **The filename filter matches nothing.** `desc.py:49-50`:

   ```python
   def _file_matches(f: str) -> bool:
       return re.match(r"^.*{m[1]}.*.json", f) is not None
   ```

   This is a raw string (`r"..."`), **not** an f-string (`rf"..."`), so `{m[1]}`
   is a literal brace sequence, never interpolated with the captured `M.m.p`.
   The pattern only matches filenames that literally contain the four characters
   `{m[1]}`. For any real version the candidate list is empty, the loop body
   never runs, and the function falls through to `raise NoSuchVersionError()` at
   `desc.py:91`. In other words, today the trailing "image descriptor missing"
   warning in `versions create` (`cmds/versions.py:111-113`) fires
   **unconditionally**.

2. **Wrong descriptor returned on multiple files.** The capture loop tracks the
   match in `found_at` (`desc.py:83`) but then returns `desc` (`desc.py:89`),
   which is the **last-parsed** candidate, not the one that matched at
   `found_at`. With ≥2 candidate files this returns the wrong descriptor.

3. **Not lenient on malformed JSON.** A candidate whose JSON fails to parse is
   **re-raised** (`desc.py:72-74`), aborting the whole lookup — unlike
   `list_releases`, which the same design (006:115) correctly describes as
   lenient.

**Gap:** The design represents the _intended_ contains-match semantics as if
they were the Python's behavior, in a document whose explicit mandate is to
verify every claim against the source. It is silent that the port is _fixing_ a
broken function — which is materially different from "porting parity" and
changes the runtime behavior operators see (the warning stops firing on every
create).

**Recommended change:** Add a fidelity note that Python `get_image_desc` is
broken — `desc.py:50` f-string bug means the filename filter matches nothing and
the function always raises `NoSuchVersion`; it also returns the last-parsed (not
the matched) descriptor on multiple candidates, and re-raises on malformed
candidate JSON. State explicitly that 006 ports the _intended_ contract
(contains-match on `M.m.p`, return the matched descriptor, lenient-skip
malformed candidates), not the broken behavior, and add a test asserting the
warning fires only when no descriptor matches — not on every create.

### F2 — C7 is silent on `config.storage` / `config.storage.s3` being `None` (Important)

**Design claim** (006:109-115): the `versions list` port sources `bucket` and
`loc` from `config.storage.s3.releases`, and `--from` supplies the S3 host URL.

**What the source shows:** In 004 (and the Python `config.py:142,176`),
`Config.storage` is `Option<StorageConfig>` (004:49) and `storage.s3` is
`Option<S3StorageConfig>` (004:67). Both can legitimately be absent. The Python
`list_releases` requires `(secrets, url, bucket, bucket_loc)`
(`releases/s3.py:237-239`), so the port must unwrap two `Option`s to get
`bucket`/`loc`.

**Gap:** 006 does not say what happens when `config.storage` or
`config.storage.s3` is `None`. The design must specify a clean,
operator-actionable error (e.g. "no S3 releases storage configured; set
`storage.s3.releases` in the config"), not an `unwrap()`/panic. This is
precisely the missing-validation edge the mandate flags. Note 005 (005:122)
repeats the same claim and is likewise silent — but 006 is the
versions-subsystem owner, so the error belongs here.

**Recommended change:** Add to the C7 section: when `config.storage`/`.s3` is
`None`, return a `ConfigError`-class error that names the missing
`storage.s3.releases` config so the operator knows how to fix it. Mirror the
two-way-fix wording the design already uses for the versions-dir error
(006:166-168).

### F3 — UUIDv7 title timestamp source is unspecified (Minor)

**Design claim** (006:98-100): for a UUIDv7 version the title is
`Release <description> version created at <timestamp>` (ISO-8601 UTC, seconds
precision).

**What the source shows:** This path is net-new (no Python analogue), so there
is no parity question — only soundness. The design (006:94-96) lets
`validate_version` accept a **user-supplied** UUIDv7, not only a freshly
generated one. So "now" (wall-clock at create time) would be the wrong timestamp
for a supplied UUID whose embedded time is in the past.

**Gap:** "created at <timestamp>" never says where the timestamp comes from. The
two candidates (wall-clock now vs. the time embedded in the UUIDv7) diverge
exactly when the UUID is supplied rather than generated.

**Recommended change:** Pin the title timestamp to the time **embedded in the
UUIDv7 itself** (UUIDv7 carries a millisecond Unix timestamp in its high bits),
so the title is consistent whether the UUID is generated or supplied. State the
extraction explicitly.

_Affirmation (no finding):_ the "accept a version-string OR a UUIDv7" decision
has no collision gap. A `parse_version`-valid string needs a `\d+` major
immediately after the optional `prefix-`/`v` and uses `.` separators
(`utils.py:44-59`); a UUIDv7 is hex `8-4-4-4-12` with `-` separators and no `.`.
The two grammars are disjoint, so the `OR` is unambiguous in both directions.

### F4 — `paths.versions` is added to a `PathsConfig` that 004 owns (Minor)

**Design claim** (006:151,163-165): the configurable store adds
`Config.paths.versions: Option<Utf8PathBuf>`, and doing so is additive (no
`schema-version` bump).

**What the source shows:** `Config.paths` is owned by 004, whose `PathsConfig`
(004:57-62) lists `components`, `scratch`, `scratch_containers`, `ccache` — no
`versions`. The Python `PathsConfig` (`config.py:103-113`) likewise has no
`versions` field.

**Gap:** The additivity itself is a settled decision and is not relitigated
here. But 006 introduces a field into a struct another document owns, without
004 reflecting it. That is cross-doc drift: a reader of 004 will see an
authoritative `PathsConfig` that omits the field 006 relies on.

**Recommended change:** Note in 006 that the `paths.versions` field is owned by
004's `PathsConfig` and that 004 must be updated (in the M5/C9 timeframe) to
carry it; or move the field declaration's authority to 004 and have 006
reference it. Either resolves the ownership ambiguity.

### F5 — `ImageDescriptorError` has no home in 002's error taxonomy (Minor)

**Design claim** (006:176-178): the IO-triggered errors raised by this subsystem
are `NoSuchVersionDescriptor { path }`, `NoSuchVersion`, and
`ImageDescriptorError` (the `get_image_desc` conflict, `desc.py:82`).

**What the source shows:** 002's error split (002:236-268) enumerates
`MalformedVersion`, `VersionError`, `InvalidVersionDescriptor`,
`NoSuchVersionDescriptor`, and `NoSuchVersion`. It does **not** mention
`ImageDescriptorError`. The Python `ImageDescriptorError` is a distinct class
(`images/errors.py:19`).

**Gap:** 006 introduces a fourth IO-error variant the 002 taxonomy never names.
Minor, but the error split is supposed to be authoritative in 002; a variant
living only in 006 is a taxonomy gap.

**Recommended change:** Either add `ImageDescriptorError` (conflict) to 002's
IO-error enumeration with a forward reference to 006, or have 006 state
explicitly that this variant is owned here and is intentionally outside the 002
list (since `ImageDescriptor` itself is owned by 006).

### F6 — `get_minor_version` "else None" glosses raise-on-malformed (Minor)

**Design claim** (006:30-31): `get_minor_version(s)` requires major, minor, and
patch; returns `"<major>.<minor>.<patch>"`, **else `None`**.

**What the source shows:** `utils.py:90-104` raises `MalformedVersionError` when
the string does not parse at all (the `parse_version` call, re-raised at
`utils.py:98-99`), and returns `None` only when the string parses but lacks
`patch` (`utils.py:101-102`). So the function has two distinct failure modes,
not one.

**Gap:** "else `None`" collapses both the raise and the `None` return into a
single outcome. A Rust port that maps this to `Option<String>` would drop the
malformed-input error; the faithful shape is
`Result<Option<String>, MalformedVersion>` (or equivalent).

**Recommended change:** Reword to: returns `Some("M.m.p")` when all three
components are present, `None` when the string parses but `patch` is absent, and
raises `MalformedVersion` when the string does not parse.

## Affirmations (claims verified correct)

- **Title generation is faithful** (the mandate's hardest target — no bug here).
  `_do_version_title` (`create.py:45-76`) produces
  `Release <desc> [<PREFIX> ]version <M>.<m>.<p>[ (<suffix>)]`: prefix
  uppercased + space only when present (`create.py:58`), the literal token
  `version` before `M.m.p` (`create.py:59`), and the suffix split on `-` then
  `.` (maxsplit 1), first segment uppercased, joined by `, `, wrapped in parens
  (`create.py:61-74`). `rc.1 → (RC 1)` and `asd-qwe → (ASD, QWE)` both check
  out, as does the table description ("General Availability") rather than the
  type name (006:74-77 matches `create.py:76` + `get_version_type_desc`,
  `utils.py:135-140`).
- **`_validate_version` requires minor and patch** — `create.py:37-42` returns
  `minor is not None and patch is not None`, so a human version must be at least
  `M.m.p`. 006:80-81 is correct.
- **`get_version_type` is a name→type lookup, not a regex parse** —
  `utils.py:127-132` does `_release_types.get(name.lower())`; unknown →
  `VersionError`. 006:54-55 is correct.
- **The label table** (006:47-52) matches `_release_types` (`utils.py:33-38`)
  exactly, including `release → General Availability`.
- **The parse grammar** (006:18-25) matches the regex `utils.py:44-59`: optional
  `\w+` prefix, optional `v`, mandatory `\d+` major, optional
  `.minor`/`.patch`/`-suffix` (suffix `[\w_.-]+`).
- **`get_image_desc`'s version regex requires a literal `v`** — `desc.py:43` is
  `r".*v(\d+\.\d+\.\d+).*"`; a `v`-less version raises `MalformedVersionError`.
  006:132 is correct.
- **`ImageDescriptor`/`ImageLocations` field names** (006:126-127) match
  `desc.py:32-39` (`releases`, `images`; `src`, `dst`).
- **The broken `versions list` call site** — `cmds/versions.py:129` calls
  `list_releases(secrets, s3_address_url)` (2 args) against a 4-arg signature
  (`releases/s3.py:237-239`), so it raises `TypeError`. 006:107-115 correctly
  identifies the bug and the C7 fix shape (modulo F2's silent edge).
- **`-o COMPONENT=URI` parsing** (006:69) matches `version_create_helper`
  (`create.py:182-190`, `split("=", 1)`).
- **`image.tag` falls back to the version** (006:78-79) matches `create.py:133`
  (`image_tag if image_tag else version`).
- **`parse_component_refs`** (006:34-35) matches `utils.py:143-155` (regex
  `^([\w_-]+)@([\d\w_./-]+)$`).
- **Milestone routing** — C1 (UUIDv7 in scope from start), C7 (versions list
  fix), C9/M5 (configurable store) all match 001:197-241,261-266.
