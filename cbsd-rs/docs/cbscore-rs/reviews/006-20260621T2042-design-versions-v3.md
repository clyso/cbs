## Verdict

**SOUND — go (with two wording fixes).**

The image-descriptor design is sound in direction and implementable. F1 (the
broken-vs-intended framing, the always-empty candidate list, the
last-parsed-vs-matched bug, the malformed re-raise, the `v`-requiring regex) and
N1 (the UUIDv7 / `v`-less version × trailing-check interaction) are **both
genuinely resolved** in the current 006 text, and I verified the resolutions
against the Python rather than taking v1/v2 at their word — the current doc's
wording about _which_ path triggers the last-parsed bug is now correct (single
match with trailing candidates, **not** multi-match).

The current 006 is materially more accurate than the v2 snapshot the prior
review scored: 006:144-147 now correctly attributes the last-parsed bug to "a
single match followed by further candidates" and reserves "two genuine matches"
for the conflict — so the v2 F1-residue is closed, and the v2 N1 is closed by
006:110-111 and 006:164-172.

What remains are **two new under-specifications** in the _intended_ contract
that neither v1 nor v2 raised, both inside the lookup semantics the maintainer
is right to be skeptical about: (V1) the contract keys the filename filter on
the extracted `M.m.p` but keys `releases` membership on "the exact version"
without ever pinning the canonical form, so a `v`-less input can silently fail
to match a descriptor that lists the canonical `vM.m.p`; and (V2) "name contains
the `M.m.p` string" is an unanchored, unescaped substring match that
over-matches, and the contract is silent on a parseable-but-patch-less input.
Both are spec/wording fixes, not structural defects — they do not block the
direction, but they should be closed before implementation so the port does not
bake in a silent-miss.

## Accuracy: does 006 describe the broken Python correctly?

Every particular checks out against `desc.py`. Anchored to line numbers and
confirmed empirically.

### Always-empty candidate list — CORRECT (006:139-143, 156-158)

`desc.py:49-50`:

```python
def _file_matches(f: str) -> bool:
    return re.match(r"^.*{m[1]}.*.json", f) is not None
```

This is a **raw** string `r"..."`, not an f-string `rf"..."`, so `{m[1]}` is
literal text and is never interpolated with the captured `M.m.p`. The pattern
only matches a filename literally containing the four characters `{m[1]}`. I
confirmed empirically that real names — including `20.2.1.json` and
`ces-v20.2.1.json` — return `False`; only a literal `{m[1]}` substring returns
`True`. So `_gen_candidates` (`desc.py:52-53`) yields nothing, the walk at
`desc.py:60-62` extends an empty list, the loop at `desc.py:68` never runs,
`found_at` stays `None`, and the function falls through to
`raise NoSuchVersionError()` (`desc.py:91`) on every call. 006:139-143 and the
fidelity note 006:231-233 state this correctly.

### Last-parsed-vs-matched bug, and WHICH path triggers it — CORRECT (006:144-147)

I simulated `desc.py:66-91` faithfully (`desc` is reassigned at `desc.py:71` on
**every** iteration; `desc.py:84`'s `desc = desc` is a no-op):

- **Single match, matched file not last in walk order** → returns the
  **last-parsed** candidate, not the one recorded in `found_at`. This is the
  bug.
- **Single match, matched file is last** → returns the matched descriptor (the
  bug does not manifest).
- **Two genuine matches** → `desc.py:77-82` raises `ImageDescriptorError`
  (conflict); it never returns the last-parsed descriptor.
- **No match** → `raise NoSuchVersionError` (`desc.py:91`).

006:144-147 now says: "Two genuine matches would raise `ImageDescriptorError`
(conflict), but a single match followed by further candidates returns the
**last-parsed** descriptor rather than the matched one." That is exactly the
simulated behavior, and it is the precise correction the v2 review's F1-residue
asked for. **The v2 F1-residue is fixed in the current text** — no deduction.
(The maintainer's specific question — "single match with trailing candidates vs.
two matches → conflict" — is answered correctly by the doc.)

### Malformed-JSON handling — CORRECT (006:147, 159-160)

`desc.py:72-74` catches the parse exception and `raise e` — it **re-raises**,
aborting the whole lookup; it does not skip. 006:147 states the Python
"re-raise[s] rather than skip[s]," and 006:159-160 specifies the port instead
**skips** malformed candidates, explicitly framed as a deviation matching
`list_releases`' leniency (005). Accurate, and honestly labelled as a behavior
change, not parity.

### `v`-requiring regex — CORRECT (006:151-155)

`desc.py:43` is `re.match(r".*v(\d+\.\d+\.\d+).*", version)`. I confirmed:
`v20.2.1` → captures `20.2.1`; `20.2.1` → `None` (no literal `v`); `20.2` →
`None` (patch mandatory); a UUIDv7 → `None`. So the regex requires a literal `v`
immediately before `M.m.p` and requires all three components. 006:152-154 states
this and that the port relaxes it via `parse_version` (whose `v?` is optional,
`utils.py:49`). Accurate.

### `desc.version` is the raw input — CORRECT (006:166-172)

I verified independently of v2 (trust-nothing): `create.py:136` sets
`version=version` — the descriptor stores the **unmodified** CLI input;
`normalize_version` is never called on this path. So `versions create 20.2.1`
yields `desc.version == "20.2.1"`, and the trailing `get_image_desc` is called
with the raw `20.2.1`, which `desc.py:43` rejects → `MalformedVersion`. That
confirms the premise behind 006:166-172's claim that today the check "would
error outright for a UUIDv7 or `v`-less version" — the raw-storage premise
holds, so the accuracy claim is correct.

### Exception hierarchy / call-site routing — CORRECT

`errors.py:30` (`MalformedVersionError`) and `errors.py:36`
(`NoSuchVersionError`) both subclass `CESError` (`errors.py:18`);
`images/errors.py:19` (`ImageDescriptorError`) also subclasses `CESError`. The
call site (`cmds/versions.py:112-117`) catches `NoSuchVersionError` first
(benign "missing" message) and `CESError` second (error message). So today a
`MalformedVersion` (raw `v`-less input or a UUIDv7) lands on the **error**
branch, never the "missing" branch — exactly what 006:166-172 asserts.

**No accuracy mis-statement remains.** Every claim about the broken Python is
correct, including the WHICH-path detail the prior round got wrong.

## Soundness of the intended contract

The contract at 006:149-162 is internally consistent and implementable in its
broad shape: extract `M.m.p` v-optionally via `parse_version` (achievable — I
confirmed `parse_version` extracts `M.m.p` from `20.2.1`, `v20.2.1`, and
`ces-v20.2.1`); walk `<git-root>/desc/`; skip malformed candidates; missing
`desc/` → `NoSuchVersion`; two matches → conflict; none → `NoSuchVersion`;
UUIDv7 → skip the lookup; not-found/not-applicable → benign note. Those edges
are coherent and map cleanly to a Rust implementation.

Two edges are **under-specified**, both about what "the exact version" means as
the membership key — the maintainer's skepticism is well-aimed here.

### V1 (Important-ish) — membership key vs. filter key are not the same string

The contract keys two different things on two different values without
reconciling them:

- the **filename filter** is keyed on the extracted `M.m.p` (006:156-157: "files
  whose name contains the `M.m.p` string");
- the **`releases` membership** is keyed on "the exact version" (006:159-161:
  "the descriptor whose `releases` list contains the exact version").

In the Python, `desc.py:76` is `if version in desc.releases:` where `version` is
the **full, raw argument** — not the extracted `M.m.p`, not a normalized form. I
confirmed the consequence empirically: a descriptor that lists the canonical
`v20.2.1` in `releases` does **not** match a raw input of `20.2.1` (or
`ces-v20.2.1`). So the very relaxation the design introduces to make a `v`-less
`20.2.1` "work" at the filter step (006:153-155) produces a **silent miss** at
the membership step if descriptors list the canonical `vM.m.p` — the lookup
returns `NoSuchVersion`, the call site prints the benign "image descriptor
missing" note, and a descriptor that _should_ have matched is silently ignored.
The design never pins the canonical form used for membership, so an implementer
cannot tell whether to test the raw input, the extracted `M.m.p`, or a
normalized `vM.m.p` against `releases` — and the three disagree exactly on the
inputs the relaxation targets.

**Recommended change:** pin the membership key. State whether `releases`
membership tests the raw version, the extracted `M.m.p`, or the normalized
`vM.m.p`, and — given the descriptors are repo-authored and the codebase's own
canonical form is `vM.m.p` (`normalize_version`, `utils.py:107-124`) — specify
the match so a `v`-less CLI input still resolves a descriptor that lists
`vM.m.p` (e.g. normalize both sides, or match on the extracted `M.m.p` against a
normalized release entry). Add a test with input `20.2.1` against a descriptor
listing `v20.2.1`.

### V2 (Minor) — substring over-match + patch-less input

Two smaller gaps in the same area:

- **Over-match.** "Name contains the `M.m.p` string" (006:156-157) describes an
  unanchored, unescaped substring match (the interpolation the raw string was
  meant to do). I confirmed `M.m.p = 20.2.1` matches `120.2.1.json`,
  `20.2.10.json`, and even `20X2Y1.json` (the `.` are regex wildcards). The
  over-match is mostly masked because `releases` membership is the authoritative
  check — but it widens the candidate set and therefore feeds the
  two-matches-conflict logic, so it is worth a sentence. The port should treat
  the filename as a coarse pre-filter and either escape/anchor it or state
  plainly that `releases` membership is the sole authority (and the filename
  filter is advisory).
- **Patch-less parse.** `parse_version` accepts `20.2` (patch `None`) and `20`
  (minor `None`). The standalone `get_image_desc` contract says "extract
  `<M>.<m>.<p>`" but is silent on a parseable-but-patch-less input. On the
  `versions create` path this is guarded by `_validate_version` (minor **and**
  patch required, `create.py:37-42`), but `get_image_desc`'s own contract is
  what 006 is specifying, and it does not say whether a patch-less version is a
  `MalformedVersion`, a `NoSuchVersion`, or uses a two-component key.

**Recommended change:** add one line to the contract — the filename filter is
advisory (membership in `releases` is authoritative), and a version that parses
but lacks patch is rejected as malformed (or define the two-component behavior
explicitly).

## Verdict on F1 and N1

Both are **genuinely resolved** in the current 006 — verified against the
Python, not taken from v1/v2.

- **F1 (resolved).** The doc frames `get_image_desc` as
  fix-the-intended-contract, not parity (006:139-147, 231-233); names the
  always-empty candidate list from the raw-string filter (006:139-143); states
  the malformed re-raise (006:147); and — the v2 residue — now attributes the
  last-parsed bug to "a single match followed by further candidates"
  (006:144-147), which my simulation confirms is the correct path, with "two
  genuine matches" correctly reserved for the conflict. The v2
  self-contradiction ("dead multi-match path") is gone. No F1 residue remains.
- **N1 (resolved).** 006:110-111 and 006:164-172 state that the trailing check
  is **skipped entirely** for a UUIDv7 (no `M.m.p` to key on), and 006:166-172
  narrows the post-fix "missing" claim to the version-string path, explicitly
  contrasting it with today's Python where a UUIDv7 or `v`-less version routes a
  `MalformedVersion` to the error branch (`cmds/versions.py:112-117`). I
  confirmed the routing and the raw-storage premise (`create.py:136`) directly.
  The N1 spec gap is closed.

The only caveat is V1: the UUIDv7 skip (N1's fix) is airtight, but the
version-string path that N1 narrowed the claim to still has the V1
membership-key ambiguity — i.e. N1 correctly excluded UUIDv7, but the "fires
only when genuinely absent" promise (006:170-172) is only true once V1 pins the
membership key so a `v`-less input does not produce a false "missing."

## Findings (by severity)

1. **V1 (Important):** the intended contract keys the filename filter on the
   extracted `M.m.p` but keys `releases` membership on "the exact version"
   (`desc.py:76` uses the raw arg) without pinning the canonical form — a
   `v`-less CLI input silently misses a descriptor listing the canonical
   `vM.m.p`, yielding a false "image descriptor missing." Pin the membership
   key; add a `20.2.1` vs `v20.2.1` test.
2. **V2 (Minor):** "name contains `M.m.p`" is an unanchored, unescaped substring
   match (over-matches; dots are wildcards), and the contract is silent on a
   parseable-but-patch-less input. State that `releases` membership is
   authoritative and define the patch-less case.

## Required action before implementation

Close V1 (pin the membership-key canonical form so a `v`-less input resolves a
`vM.m.p` descriptor) and V2 (filename filter advisory; patch-less defined). Both
are wording fixes. With them, the image-descriptor design is a clean go; without
them, the implemented port risks a silent "missing" on exactly the `v`-less
inputs the relaxed filter was added to support.
