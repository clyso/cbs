# 06 — RBAC Administration

## Overview

CLI commands for managing roles, users, user-role
assignments (with scopes), and the build queue. Maps
to `/api/permissions/*` and `/api/admin/*` endpoints.

## Command structure

```
cbc admin
  roles
    list
    create <name>
    get <name>
    update <name>
    delete <name>
  users
    list
    get <email>
    activate <email>
    deactivate <email>
    roles
      set <email>
      add <email>
      remove <email>
  queue
```

## Role commands

### `cbc admin roles list`

```
$ cbc admin roles list

  NAME      BUILTIN  DESCRIPTION
  admin     yes      Full administrative access
  builder   yes      Create and manage own builds
  viewer    yes      Read-only access
  deployer  no       Deploy team role
```

**Endpoint:** `GET /api/permissions/roles`

**Requires:** `permissions:view`

The server's `RoleListItem` returns `name`,
`description`, `builtin`, `created_at` — no `caps`.
Capabilities are shown only in `roles get`.

### `cbc admin roles create <name> [options]`

```
$ cbc admin roles create deployer \
    --cap builds:create \
    --cap periodic:create \
    --description "Deploy team role"

role 'deployer' created
```

**Endpoint:** `POST /api/permissions/roles`

**Options:**

| Flag | Description |
|---|---|
| `--cap` | Capability (repeatable) |
| `--description` | Role description |

### `cbc admin roles get <name>`

```
$ cbc admin roles get builder

      name: builder
   builtin: yes
      desc: Create and manage own builds
      caps: builds:create
            builds:revoke:own
            builds:list:own
            builds:list:any
            apikeys:create:own
            workers:view
```

**Endpoint:** `GET /api/permissions/roles/{name}`

### `cbc admin roles update <name> [options]`

Replace the role's capability set.

```
$ cbc admin roles update deployer \
    --cap builds:create \
    --cap periodic:create \
    --cap periodic:view

role 'deployer' updated (3 capabilities)
```

**Endpoint:** `PUT /api/permissions/roles/{name}`

**Requires:** `permissions:manage`

The request body must include `name` matching the path
parameter. Capabilities replace the entire set.

Builtin roles cannot be updated — server returns 409.

**Options:**

| Flag | Description |
|---|---|
| `--cap` | Capability (repeatable, replaces all) |

### `cbc admin roles delete <name> [options]`

```
$ cbc admin roles delete deployer
role 'deployer' deleted
```

**Endpoint:** `DELETE /api/permissions/roles/{name}`

**Options:**

| Flag | Description |
|---|---|
| `--force` | Delete even if role has assignments |

Without `--force`, the server returns 409 if the role
has active user assignments. With `--force`, the
`?force=true` query parameter is appended and the
server cascades the deletion.

Builtin roles cannot be deleted — server returns 409
regardless of `--force`.

## User commands

### `cbc admin users list`

```
$ cbc admin users list

  EMAIL               NAME    ACTIVE  ROLES
  admin@clyso.com     Admin   yes     admin
  dev@clyso.com       Dev     yes     builder
  intern@clyso.com    Intern  no      viewer
```

**Endpoint:** `GET /api/permissions/users`

**Requires:** `permissions:view`

Scopes are omitted from the list view. Use
`admin users get` for full details including scopes.

### `cbc admin users get <email>`

Show user details with full role + scope information.

```
$ cbc admin users get dev@clyso.com

   email: dev@clyso.com
    name: Dev User
  active: yes

  roles:
    builder
      scopes:
        channel = ces-devel
        channel = ces-public
    viewer
      (no scopes)

  effective caps:
    builds:create, builds:revoke:own, ...
```

**Implementation:** Single request:
`GET /api/permissions/users` returns all users with
their roles and scopes. The client filters by email
client-side. No second request needed — the list
response already includes role assignments with scopes
for each user.

### `cbc admin users activate <email>`

```
$ cbc admin users activate intern@clyso.com
user 'intern@clyso.com' activated
```

**Endpoint:** `PUT /api/admin/users/{email}/activate`

### `cbc admin users deactivate <email>`

```
$ cbc admin users deactivate intern@clyso.com
user 'intern@clyso.com' deactivated
  tokens revoked: 2
  api keys revoked: 1
```

**Endpoint:**
`PUT /api/admin/users/{email}/deactivate`

Last-admin guard: the server prevents deactivating
the last user with `*` capability (returns 409). The
client prints the server's error message.

The guard also fires on `roles delete` and
`roles update` if the operation would remove all
wildcard holders.

## User role assignment commands

### `cbc admin users roles set <email> [options]`

Replace all roles for a user.

```
$ cbc admin users roles set dev@clyso.com \
    --role builder \
      --scope channel=ces-devel \
      --scope channel=ces-public \
    --role viewer

user 'dev@clyso.com' roles set: builder, viewer
```

**Endpoint:**
`PUT /api/permissions/users/{email}/roles`

**Options:**

| Flag | Description |
|---|---|
| `--role` | Role name (repeatable) |
| `--scope` | `type=pattern` (repeatable) |

Scopes apply to the preceding `--role`. For roles with
scope-dependent capabilities (e.g., `builds:create`),
at least one scope is required — the server returns 400
otherwise.

Scope types: `channel`, `registry`, `repository`.

**Request body:**

```json
{
  "roles": [
    {
      "role": "builder",
      "scopes": [
        {"type": "channel", "pattern": "ces-devel"},
        {"type": "channel", "pattern": "ces-public"}
      ]
    },
    {
      "role": "viewer",
      "scopes": []
    }
  ]
}
```

### `cbc admin users roles add <email> [options]`

Add a role to a user.

```
$ cbc admin users roles add dev@clyso.com \
    --role deployer \
    --scope channel=prod

role 'deployer' added to user 'dev@clyso.com'
```

**Endpoint:**
`POST /api/permissions/users/{email}/roles`

**Options:**

| Flag | Description |
|---|---|
| `--role` | Role name (required) |
| `--scope` | `type=pattern` (repeatable) |

### `cbc admin users roles remove <email> [options]`

Remove a role from a user.

```
$ cbc admin users roles remove dev@clyso.com \
    --role deployer

role 'deployer' removed from 'dev@clyso.com'
```

**Endpoint:**
`DELETE /api/permissions/users/{email}/roles/{role}`

**Options:**

| Flag | Description |
|---|---|
| `--role` | Role name to remove (required) |

## Queue command

### `cbc admin queue`

Show build queue status.

```
$ cbc admin queue

  priority  pending
  high      0
  normal    3
  low       1
```

**Endpoint:** `GET /api/admin/queue`

**Requires:** `admin:queue:view`

## Error handling

- Permission denied (403) → print the required
  capability.
- Last-admin guard (409) → print the server message.
- Role not found (404) →
  `"role '{name}' not found"`.
- User not found (404) →
  `"user '{email}' not found"`.
- Role assignment not found (404) →
  `"role assignment not found"`.
- Builtin role delete/update attempt (409) →
  `"cannot modify builtin role '{name}'"`.
- Role has assignments without `--force` (409) →
  `"role has active assignments — use --force"`.
- Scope-dependent role without scopes (400) →
  print the server's error message (names the
  scope-dependent capabilities).
