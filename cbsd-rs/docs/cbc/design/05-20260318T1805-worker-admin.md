# 05 — Worker Administration

## Overview

CLI commands for managing registered workers. Maps to
`/api/admin/workers` and `/api/workers` endpoints.

## Commands

### `cbc worker list`

List all registered workers with live status.

```
$ cbc worker list

  NAME            ARCH     STATUS    BUILD  LAST SEEN
  build-host-01   x86_64   building  #42    2026-03-18 14:30
  arm-builder-02  aarch64  offline   -      2026-03-17 09:00
```

**Endpoint:** `GET /api/workers`

**Requires:** `workers:view`

Tabular output. Columns: name, arch, status (connected,
building, stopping, disconnected, offline), current
build ID (or `-`), last seen timestamp.

### `cbc worker register <name> <arch>`

Register a new worker and print the worker token.

```
$ cbc worker register build-host-03 x86_64

worker 'build-host-03' registered
  id: 770a6600-...

worker token (save this - it cannot be recovered):

  eyJ3b3JrZXJfaWQ...

Set as CBSD_WORKER_TOKEN in the worker environment,
or as 'worker-token' in worker.yaml.
```

**Endpoint:** `POST /api/admin/workers`

**Arguments:**

- `NAME` (required) — worker name (alphanumeric,
  hyphens, underscores, 1-64 chars).
- `ARCH` (required) — `x86_64` or `aarch64`.

**Requires:** `workers:manage`

Client-side arch validation before the round-trip for
a better error message.

The `worker_token` is displayed prominently. This is
the only time the token is visible.

### `cbc worker deregister <id>`

Deregister a worker and revoke its API key.

```
$ cbc worker deregister 770a
worker 'build-host-03' deregistered (API key revoked)
```

**Endpoint:** `DELETE /api/admin/workers/{id}`

**Requires:** `workers:manage`

If the worker is connected, it will be force-
disconnected and any in-flight build re-queued.

### `cbc worker regenerate-token <id>`

Rotate a worker's API key.

```
$ cbc worker regenerate-token 770a

worker 'build-host-03' token regenerated

WARNING: if the worker is currently building, the
build will be re-queued.

new worker token (save this - it cannot be recovered):

  eyJ3b3JrZXJfaWQ...

The worker must be restarted with the new token.
```

**Endpoint:**
`POST /api/admin/workers/{id}/regenerate-token`

**Requires:** `workers:manage`

The connected worker will be force-disconnected. The
warning about in-flight build re-queuing is always
shown (the client cannot know if a build is active).

## ID prefix matching

Worker commands that take `<id>` accept either:
- Full UUID (`770a6600-e29b-41d4-...`)
- Prefix (e.g., `770a`)

Prefix matching is done client-side:
1. Fetch the full list via `GET /api/workers`.
2. Filter by `worker_id` prefix.
3. Zero matches → `"no worker matching '{prefix}'"`.
4. Multiple → `"ambiguous prefix '{prefix}' matches:
   {list}"`.
5. Exactly one → use it.

Prefix matching requires `workers:view`. Users with
`workers:manage` only (no `workers:view`) must use
full UUIDs — the list request returns 403 before the
actual operation. In that case, the client falls back
to treating the argument as a full UUID.

## Error handling

- Worker not found → 404 →
  `"worker '{id}' not found"`.
- Name already exists → 409 →
  `"worker name already exists"`.
- Missing capability → 403 →
  `"missing required capability: workers:manage"`.
- Invalid arch → 400, print server message.
- Deregister does not report whether a build was
  re-queued (the server response has no such field).
  This is a known limitation.
