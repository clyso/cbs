# 02 — Build Operations

## Overview

Build submission, listing, revocation, and component
discovery.

## Commands

### `cbc build new <version> [options]`

Submit a new build.

```
$ cbc build new 19.2.3 \
    --channel ces \
    --component ceph@v19.2.3 \
    --type release

  version: 19.2.3
  channel: ces
     type: release
    image: ceph/ceph:19.2.3
    comps: ceph@v19.2.3
   distro: rockylinux el9

build 42 submitted (queued)
```

The version, channel, image, etc. shown are echoed from
the locally constructed descriptor — the server's 202
response contains only `id`, `state`, and optional
`warning`.

**Endpoint:** `POST /api/builds/`

**Request body:** The POST body wraps the descriptor:

```json
{
  "descriptor": { ... BuildDescriptor ... },
  "priority": "normal"
}
```

Not a bare `BuildDescriptor`. The `SubmitBuildBody`
struct has `descriptor` and `priority` as top-level
fields.

**Arguments:**

- `VERSION` (required) — version string.

**Options:**

| Flag | Description | Default |
|---|---|---|
| `-t, --type` | Version type | `dev` |
| `-p, --channel` | Release channel | required |
| `-c, --component` | `name@gitref` (repeat) | required |
| `--repo-override` | `name=url` | none |
| `--distro` | Base distribution | `rockylinux` |
| `--os-version` | OS version string | `el9` |
| `--image-name` | Dest image name | `ceph/ceph` |
| `--image-tag` | Dest image tag | VERSION |
| `--arch` | Build architecture | `x86_64` |
| `--priority` | high/normal/low | `normal` |

Valid `--type` values: `release`, `dev`, `test`, `ci`
(matching `cbsd_proto::VersionType`).

**Behavior:**

1. Fetch current user via `GET /api/auth/whoami` to
   populate `signed_off_by` (name + email).
2. Parse `--component` values: split on `@` →
   `{name, git_ref}`. Error if no `@`.
3. Parse `--repo-override` values: split on `=` →
   `{name, repo_url}`.
4. Construct `BuildDescriptor` (from `cbsd-proto`).
   Note: `BuildComponent.git_ref` serializes as `"ref"`
   in JSON (serde rename).
5. Wrap in `{descriptor, priority}` body.
6. POST to `/api/builds/`.
7. Print the build ID and state from the response.
   If `warning` is present, print it.

### `cbc build list [options]`

List builds.

```
$ cbc build list

  #42  queued    admin@clyso.com  2026-03-18 14:30
  #41  success   admin@clyso.com  2026-03-18 12:00
  #40  failure   admin@clyso.com  2026-03-17 22:15
```

**Endpoint:** `GET /api/builds/`

**Options:**

| Flag | Description | Default |
|---|---|---|
| `--all` | Show all builds | false |
| `--state` | Filter by state | none |
| `-n, --limit` | Max lines shown | 20 |

`--all` omits the `?user=` query parameter (the server
returns all builds visible to the user's capabilities).
Without `--all`, the client sends `?user={email}`.

`--limit` is client-side truncation — the server has no
pagination. Acceptable at current scale; documented as
a known limitation.

**Output:** Tabular, one line per build. Columns: ID,
state, user, submitted timestamp. Sorted by ID
descending (newest first).

### `cbc build get <id>`

Show details of a single build.

```
$ cbc build get 42

      id: 42
   state: success
    user: admin@clyso.com
priority: normal
 version: 19.2.3
 channel: ces
    type: release
   image: ceph/ceph:19.2.3
   comps: ceph@v19.2.3
  distro: rockylinux el9
  worker: 550e8400-...
   trace: abc-123-...
  queued: 2026-03-18T14:30:00
 started: 2026-03-18T14:30:05
finished: 2026-03-18T15:45:20
```

**Endpoint:** `GET /api/builds/{id}`

Requires `builds:list:any` or `builds:list:own`. With
`:own` only, requesting another user's build returns
404 (not 403) to prevent enumeration.

### `cbc build revoke <id>`

Cancel a build.

```
$ cbc build revoke 42
build 42 revoked
```

**Endpoint:** `DELETE /api/builds/{id}`

The server returns 409 for builds already in a terminal
state (not 404). The client prints the server's error
message on any non-success response.

### `cbc build components`

List available build components.

```
$ cbc build components

  ceph
    versions: v19.2.3, v18.2.7, main
  ceph-nvmeof
    versions: v1.0.0, main
```

**Endpoint:** `GET /api/components/`

The server's `ComponentInfo` returns `name` and
`versions: Vec<String>`. There is no `repo` field in
the API response.

## Build descriptor construction

The `--component`, `--repo-override`, and other options
map to `cbsd_proto::BuildDescriptor`. The client
constructs the struct locally and serializes it as part
of the POST body.

The `signed_off_by` fields are populated from `whoami`
— the server overwrites them anyway, but including them
avoids a validation error if the server checks for
non-empty fields.

## Shared `BuildDescriptorArgs`

The options for constructing a `BuildDescriptor` are
shared between `build new` and `periodic new`. Extract
into a shared `clap` `Args` struct — but note that
`version` is NOT shared:

- `build new` takes `VERSION` as a **positional** arg
  (natural CLI: `cbc build new 19.2.3`).
- `periodic new` takes `--version` as a **named** option
  (consistent with the other `--` options).

The shared struct covers everything except `version`:

```rust
#[derive(Args)]
struct BuildDescriptorArgs {
    #[arg(long)]
    channel: String,
    #[arg(long = "component", num_args = 1..)]
    components: Vec<String>,
    #[arg(long = "type", default_value = "dev")]
    version_type: String,
    // ... distro, os_version, arch, image_name,
    //     image_tag, repo_override
}
```

`BuildNewArgs` has `version: String` as a positional
plus `#[command(flatten)] descriptor: BuildDescriptorArgs`.

`PeriodicNewArgs` has `#[arg(long)] version: String`
plus `#[command(flatten)] descriptor: BuildDescriptorArgs`.

The `BuildDescriptor` struct has nested sub-objects:
`dst_image: BuildDestImage` (from `--image-name`,
`--image-tag`) and `build: BuildTarget` (from `--distro`,
`--os-version`, `--arch`). The client constructs these
nested structs from the flat CLI options.

The `--type` CLI flag maps to the JSON field
`"version_type"` (not `"type"`).

## Error handling

- Unknown component name → server returns 400 with the
  list of unknown components. Print it.
- Missing required options → clap handles this.
- Build in terminal state (revoke) → 409, print server
  message.
- Build not found → 404.
