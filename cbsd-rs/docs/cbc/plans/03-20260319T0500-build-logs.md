## Why one commit

The three log commands (`tail`, `follow`, `get`) form a
single `build logs` subcommand group under the `build`
command from plan 02. They share:

- The `Logs` sub-subcommand enum and dispatch.
- The build ID path parameter pattern.
- The same `CbcClient` from plan 00.

`tail` and `get` are straightforward HTTP calls (~40-50
lines each). `follow` is the most complex (~120 lines)
due to SSE event handling and retry logic. Together they
total ~350 LOC — below the 400-800 target but above the
200-line minimum for a meaningful commit.

Splitting `follow` from `tail`/`get` would leave `tail`
and `get` as a ~100-line commit (too small), and `follow`
would need the subcommand group that `tail`/`get` already
established. One commit is the right granularity.

### Content

#### `cbc/Cargo.toml` change

Add dependency:

```toml
reqwest-eventsource = "0.6"
futures-util = "0.3"
```

`futures-util` is needed for `StreamExt::next()` on the
`EventSource` stream.

#### `builds.rs` changes

Add `Logs(LogsArgs)` variant to `BuildCommands`. Add
dispatch arm calling `logs::run()`.

#### `cbc/src/logs.rs`

**Subcommand structure:**

```rust
#[derive(Args)]
pub struct LogsArgs {
    #[command(subcommand)]
    command: LogsCommands,
}

enum LogsCommands {
    Tail(TailArgs),
    Follow(FollowArgs),
    Get(GetArgs),
}
```

**`build logs tail` flow:**

1. Load config.
2. `GET /api/builds/{id}/logs/tail?n={n}`.
   Default `n = 30`.
3. Deserialize response: `{ build_id, lines,
   total_lines, returned }`.
4. Print context line: `"(showing {returned} of
   {total_lines} lines)"`.
5. Print each line from the `lines` array.

**`build logs follow` flow:**

1. Load config.
2. Build request via `client.get_request(
   "builds/{id}/logs/follow")`.
3. Create `EventSource` from the request builder.
4. Event loop using `futures_util::StreamExt::next()`:
   - `Event::Open` — reset retry counter.
   - `Event::Message` with `event == "done"` — break.
   - `Event::Message` (other) — `println!("{}", data)`,
     reset retry counter.
   - `Err(_)` — increment retry counter. After 3
     consecutive errors, print error and break.
5. After `done`: fetch `GET /api/builds/{id}` to get
   final state. Print:
   `"--- build {id} finished: {state} ---"`.

**`build logs get` flow:**

1. Load config.
2. `GET /api/builds/{id}/logs` via `get_stream()`.
3. Default output path: `build-{id}.log`.
   Override with `-o, --output`.
4. Write response body to file in 8KB chunks using
   `tokio::io::AsyncWriteExt`.
5. Print: `"log written to {path} ({size})"`.
   Format size as human-readable (KB/MB).
