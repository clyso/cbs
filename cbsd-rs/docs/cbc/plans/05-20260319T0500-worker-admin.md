## Why one commit

The four worker commands (`list`, `register`, `deregister`,
`regenerate-token`) share:

- The `Worker` subcommand group and its `WorkerCommands`
  enum.
- The ID prefix matching helper (used by `deregister` and
  `regenerate-token`).
- The same `CbcClient` methods from plan 00.

`list` and `register` are straightforward (40-60 lines).
`deregister` and `regenerate-token` are thin wrappers that
delegate to the prefix resolver (~30 lines each). The prefix
matching logic itself is ~50 lines (fetch list, filter,
disambiguate, fallback to full UUID on 403).

At ~350 LOC this is below the 400-800 target but above the
200-line minimum. Splitting the prefix helper from the
commands that use it would create a dead-code intermediate
commit. One commit is the right granularity.

### Content

#### `main.rs` changes

Add `Worker(WorkerArgs)` variant to the `Commands` enum.
Add dispatch arm calling `worker::run()`.

#### `cbc/src/worker.rs`

**Subcommand structure:**

```rust
#[derive(Args)]
pub struct WorkerArgs {
    #[command(subcommand)]
    command: WorkerCommands,
}

enum WorkerCommands {
    List,
    Register(RegisterArgs),
    Deregister(DeregisterArgs),
    RegenerateToken(RegenerateTokenArgs),
}
```

**ID prefix matching helper:**

```rust
async fn resolve_worker_id(
    client: &CbcClient,
    prefix: &str,
) -> Result<String, Error>
```

1. Attempt `GET /api/workers` to fetch the full list.
2. On 403 (no `workers:view`), fall back to treating
   the argument as a full UUID and return it as-is.
3. Filter workers whose `worker_id` starts with `prefix`.
4. Zero matches: return error
   `"no worker matching '{prefix}'"`.
5. Multiple matches: return error
   `"ambiguous prefix '{prefix}' matches: {list}"`.
6. Exactly one: return the full `worker_id`.

**`worker list` flow:**

1. Load config. `GET /api/workers`.
2. Print tabular output: name, arch, status, current
   build (or `-`), last seen timestamp. Status values:
   connected, building, stopping, disconnected, offline.

**`worker register` flow:**

1. Load config.
2. Client-side arch validation: accept `x86_64` or
   `aarch64` only. Print error before the round-trip
   for invalid values.
3. `POST /api/admin/workers` with
   `{ "name": "<name>", "arch": "<arch>" }`.
4. Print worker ID from response.
5. Print the `worker_token` prominently with usage
   instructions (env var or config file).

**`worker deregister` flow:**

1. Load config.
2. Call `resolve_worker_id(client, &id)`.
3. `DELETE /api/admin/workers/{resolved_id}`.
4. Print `"worker '{name}' deregistered (API key
   revoked)"`. Name comes from the list response
   cached during prefix resolution; if the fallback
   path was taken (no list), print the ID instead.

**`worker regenerate-token` flow:**

1. Load config.
2. Call `resolve_worker_id(client, &id)`.
3. `POST /api/admin/workers/{resolved_id}/
   regenerate-token`.
4. Print warning about in-flight build re-queuing
   (always shown — client cannot know build state).
5. Print the new `worker_token` prominently with
   restart instructions.
