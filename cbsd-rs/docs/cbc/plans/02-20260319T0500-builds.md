## Why one commit

The five build commands (`new`, `list`, `get`, `revoke`,
`components`) share:

- The `BuildDescriptorArgs` clap `Args` struct (used by
  `build new`, reused later by `periodic new`).
- The same `CbcClient` methods from plan 00.
- The `Build` subcommand group and its `BuildCommands` enum.

Splitting by individual command would over-fragment — each
command is 40-80 lines, and they all depend on the shared
`BuildDescriptorArgs` and `BuildCommands` enum. A single
commit keeps the subcommand group coherent.

At ~500 LOC this fits within the 400-800 target.

**Subcommand structure:**

```rust
pub struct BuildArgs {

enum BuildCommands {
    New(BuildNewArgs),
    List(BuildListArgs),
    Get(BuildGetArgs),
    Revoke(BuildRevokeArgs),
    Components,
}
```

**Shared `BuildDescriptorArgs`:**

```rust
#[derive(Args)]
pub struct BuildDescriptorArgs {
    #[arg(short = 'p', long)]
    channel: String,
    #[arg(short, long = "component", num_args = 1..)]
    components: Vec<String>,
    #[arg(short = 't', long = "type",
          default_value = "dev")]
    version_type: String,
    #[arg(long, default_value = "rockylinux")]
    distro: String,
    #[arg(long, default_value = "el9")]
    os_version: String,
    #[arg(long, default_value = "ceph/ceph")]
    image_name: String,
    #[arg(long)]
    image_tag: Option<String>,
    #[arg(long, default_value = "x86_64")]
    arch: String,
    #[arg(long)]
    repo_override: Vec<String>,
    #[arg(long, default_value = "normal")]
    priority: String,
}
```

`BuildDescriptorArgs` is `pub` — it will be reused by
`periodic new` in a future plan. `BuildNewArgs` flattens
it alongside a positional `VERSION` argument.

**`build new` flow:**

1. Load config (require auth).
2. Fetch `GET /api/auth/whoami` to get `name` and
   `email` for `signed_off_by`.
3. Parse `--component` values: split on `@` to get
   `{name, git_ref}`. Error if no `@` separator.
4. Parse `--repo-override` values: split on first `=`
   to get `{name, repo_url}`. Match overrides to
   components by name. Error on unknown name.
5. Construct `cbsd_proto::BuildDescriptor` with nested
   `BuildDestImage` (from `--image-name`, `--image-tag`
   defaulting to VERSION) and `BuildTarget` (from
   `--distro`, `--os-version`, `--arch`).
6. Parse `--type` into `cbsd_proto::VersionType`.
   Error on invalid value.
7. Parse `--priority` into `cbsd_proto::Priority`.
8. Wrap in `{ descriptor, priority }` body and
   `POST /api/builds/`.
9. Print the locally-constructed descriptor summary
   (version, channel, type, image, components, distro)
   followed by build ID and state from response.
   Print warning if present.

**`build list` flow:**

1. Load config. `GET /api/auth/whoami` to get email.
2. `GET /api/builds/` with `?user={email}` unless
   `--all` is set. Optionally `&state={state}`.
3. Client-side truncation to `--limit` (default 20).
4. Print tabular output: ID, state, user, submitted
   timestamp. Sorted by ID descending.

**`build get` flow:**

1. Load config. `GET /api/builds/{id}`.
2. Deserialize `BuildRecord`. Parse `descriptor` JSON
   field to extract `BuildDescriptor` fields.
3. Print aligned key-value detail view (id, state,
   user, priority, version, channel, type, image,
   components, distro, worker, trace, timestamps).

**`build revoke` flow:**

1. Load config. `DELETE /api/builds/{id}`.
2. Print `"build {id} revoked"` on success.
3. On 409 (terminal state), print server error.

**`build components` flow:**

1. Load config. `GET /api/components/`.
2. Deserialize `Vec<ComponentInfo>` where
   `ComponentInfo { name, versions: Vec<String> }`.
3. Print each component name and its comma-separated
   versions.
