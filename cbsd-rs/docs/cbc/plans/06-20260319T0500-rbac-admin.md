## Why two commits

The 13 admin commands split into two natural groups with a
clean dependency boundary:

**Commit 1 — roles + queue (~350 LOC):** The five role
commands (`list`, `create`, `get`, `update`, `delete`) and
the `queue` command form a self-contained group. They share
the `admin roles` subcommand hierarchy and the `admin`
parent group. The queue command is a standalone leaf under
`admin` with no dependencies on user commands. This commit
establishes the `admin` module structure that commit 2
extends.

**Commit 2 — users + user roles (~400 LOC):** The four
user commands (`list`, `get`, `activate`, `deactivate`) and
three user-role commands (`set`, `add`, `remove`) form the
second group. The `--role`/`--scope` parsing in `roles set`
and `roles add` is the most complex part of the plan — scope
arguments bind to the preceding `--role`, requiring custom
parsing logic. This group has no dependency on role CRUD
(it references role names as opaque strings).

Each commit is independently useful: after commit 1 an
admin can manage roles and view the queue. After commit 2,
user management is fully operational. The split also keeps
both commits within the 300-500 range rather than one 750+
commit at the upper guideline limit.

**Subcommand structure:**

**`admin roles list` flow:**

1. Load config. `GET /api/permissions/roles`.
2. Print tabular output: name, builtin (yes/no),
   description. `created_at` is in the response but
   omitted from the table (not in the design spec).

**`admin roles create` flow:**

1. Load config.
2. `POST /api/permissions/roles` with:

   ```json
   {
     "name": "<name>",
     "caps": ["<cap>", ...],
     "description": "<desc>"
   }
   ```

3. `--cap` is repeatable. `--description` is optional.
4. Print `"role '{name}' created"`.

**`admin roles get` flow:**

1. Load config. `GET /api/permissions/roles/{name}`.
2. Print aligned key-value: name, builtin, description.
3. Print capabilities list (one per line, indented).

**`admin roles update` flow:**

1. Load config.
2. `PUT /api/permissions/roles/{name}` with:

   ```json
   {
     "name": "<name>",
     "caps": ["<cap>", ...]
   }
   ```

3. `--cap` is repeatable (replaces entire set).
4. Print `"role '{name}' updated ({n} capabilities)"`.
5. On 409 (builtin role), print server error.

**`admin roles delete` flow:**

1. Load config.
2. `DELETE /api/permissions/roles/{name}`. Append
   `?force=true` query parameter when `--force` is set.
3. Print `"role '{name}' deleted"`.
4. On 409 without `--force`, print
   `"role has active assignments -- use --force"`.
5. On 409 for builtin role, print server message.

#### `cbc/src/admin/queue.rs`

**`admin queue` flow:**

1. Load config. `GET /api/admin/queue`.
2. Print tabular output: priority, pending count.
   Rows: high, normal, low.

### Content

#### `admin.rs` changes

Add `pub mod users;` and `Users(UsersArgs)` variant to

#### `cbc/src/admin/users.rs`

**Subcommand structure:**

```rust
#[derive(Args)]
pub struct UsersArgs {
    #[command(subcommand)]
    command: UsersCommands,
}

enum UsersCommands {
    List,
    Get(GetArgs),
    Activate(ActivateArgs),
    Deactivate(DeactivateArgs),
    Roles(UserRolesArgs),
}

#[derive(Args)]
struct UserRolesArgs {
    #[command(subcommand)]
    command: UserRolesCommands,
}

enum UserRolesCommands {
    Set(RolesSetArgs),
    Add(RolesAddArgs),
    Remove(RolesRemoveArgs),
}
```

**`--role`/`--scope` parsing for `roles set`:**

The `--scope` arguments bind to the preceding `--role`.
This cannot be expressed with standard clap derives.
Instead, use raw `Vec<String>` arguments and parse
them manually:

```rust
#[derive(Args)]
struct RolesSetArgs {
    email: String,
    /// Repeatable: --role NAME [--scope TYPE=PAT ...]
    #[arg(long, num_args = 0..)]
    role: Vec<String>,
    #[arg(long, num_args = 0..)]
    scope: Vec<String>,
}
```

Post-parse processing walks the raw `role` and `scope`
values to associate scopes with their preceding role.
The algorithm:

1. Collect `(role_name, Vec<scope>)` tuples by
   iterating the raw argv order. Each `--role` starts
   a new tuple; `--scope` values append to the current.
2. Error if `--scope` appears before any `--role`.
3. Build the request body:

   ```json
   {
     "roles": [
       {
         "role": "builder",
         "scopes": [
           { "type": "channel",
             "pattern": "ces-devel" }
         ]
       }
     ]
   }
   ```

4. Scope values split on first `=`: left is type
   (`channel`, `registry`, `repository`), right is
   pattern.

**`admin users list` flow:**

1. Load config. `GET /api/permissions/users`.
2. Print tabular output: email, name, active (yes/no),
   roles (comma-separated names, no scopes).

**`admin users get` flow:**

1. Load config. `GET /api/permissions/users`.
2. Filter by email client-side (no separate endpoint).
3. On no match: `"user '{email}' not found"`.
4. Print aligned key-value: email, name, active.
5. Print roles section with indented scopes:

   ```
   roles:
     builder
       scopes:
         channel = ces-devel
         channel = ces-public
     viewer
       (no scopes)
   ```

6. Print effective capabilities.

**`admin users activate` flow:**

1. Load config.
   `PUT /api/admin/users/{email}/activate`.
2. Print `"user '{email}' activated"`.

**`admin users deactivate` flow:**

1. Load config.
   `PUT /api/admin/users/{email}/deactivate`.
2. Print `"user '{email}' deactivated"`.
3. Print token/key revocation counts from response.
4. On 409 (last admin guard), print server message.

**`admin users roles set` flow:**

1. Load config. Parse `--role`/`--scope` pairs.
2. `PUT /api/permissions/users/{email}/roles` with
   the constructed roles array.
3. Print `"user '{email}' roles set: {names}"`.

**`admin users roles add` flow:**

1. Load config. Parse `--role`/`--scope` pairs.
2. `POST /api/permissions/users/{email}/roles` with
   single role entry.
3. Print `"role '{role}' added to user '{email}'"`.

**`admin users roles remove` flow:**

1. Load config.
2. `DELETE /api/permissions/users/{email}/
   roles/{role}`.
3. Print `"role '{role}' removed from '{email}'"`.
