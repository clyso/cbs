# Design Review: 02 — Build Operations (v2)

**Verdict: Approve with conditions.**

Both v1 blockers resolved: `SubmitBuildBody` wrapper
documented, `build components` shows `versions` not `repo`.
Additional fixes: `--all` semantics, `--limit` as client-side,
`--type` values corrected to enum variants.

No blockers.

## Major Concerns

### M1 — `version` is positional in examples but `--version` in shared struct

The command example shows `cbc build new 19.2.3` (positional).
The `BuildDescriptorArgs` struct shows `#[arg(long)] version`.
These are inconsistent. Either the command uses
`cbc build new --version 19.2.3` or `version` is a positional
in `build new` but `--version` in `periodic new` (breaking
the shared struct).

**Fix:** Pick one. If positional for `build new`, do not
share that field via `BuildDescriptorArgs` — make it a
separate positional arg on `BuildNewArgs` only.

## Minor Issues

- **Nested struct construction not explicit.** The
  `BuildDescriptor` has nested `dst_image: BuildDestImage`
  and `build: BuildTarget`. The design says "Construct
  `BuildDescriptor`" without showing how `--distro`,
  `--os-version`, `--arch` map to the nested `BuildTarget`
  struct. State this explicitly.

- **`version_type` serialization.** The CLI flag is `--type`
  but the JSON field is `"version_type"` (not `"type"`).
  Note the mapping to prevent a bug.

- **`BuildRecord` field names undocumented.** The list display
  uses `state`, `user`, `submitted timestamp` — confirm the
  actual field names (`user_email`, `submitted_at`, etc.).

## Strengths

- `SubmitBuildBody` wrapper correctly documented with example.
- `build components` now shows `versions`.
- `--all` correctly omits the user filter.
- `--type` values match `VersionType` enum.
- `build get` documents the 404-not-403 ownership behavior.
