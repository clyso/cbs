# Design Review: 02 — Build Operations

**Document:** `docs/cbc/design/02-20260318T1802-builds.md`

---

## Summary

Two blockers must be fixed before implementation: the POST body
shape is wrong, and the `build components` output references a
`repo` field that doesn't exist in the API response. Several
additional mismatches between the design and the server API need
correction.

**Verdict: Revise and re-review.**

---

## Blockers

### B1 — POST /api/builds/ body shape is wrong

The design says: "construct `BuildDescriptor` and POST to
`/api/builds/`." The server's `SubmitBuildBody` is:

```rust
struct SubmitBuildBody {
    descriptor: BuildDescriptor,
    priority: Priority,
}
```

The client must POST `{"descriptor": {...}, "priority": "normal"}`
— not a bare `BuildDescriptor`. A bare descriptor POST will
deserialize incorrectly, causing a 422.

**Fix:** Document that the POST body wraps the descriptor:
`{"descriptor": <BuildDescriptor>, "priority": <string>}`.

### B2 — `build components` shows `repo` but API returns `versions`

The design shows:

```
ceph
  repo: https://github.com/ceph/ceph.git
```

The server's `ComponentInfo` has `name: String` and
`versions: Vec<String>`. There is no `repo` field. The
component's default repo URL is not exposed by the API.

**Fix:** Change the display to show `name` and available
`versions`, not `repo`.

---

## Major Concerns

### M1 — `--all` flag sends non-existent query parameter

The server's `ListBuildsQuery` has `user: Option<String>` and
`state: Option<String>`. There is no `all` parameter. The
`--all` flag must simply omit the `?user=` filter — the server
decides based on the user's capabilities whether to return all
builds.

**Fix:** Document that `--all` omits the user filter rather
than sending a query parameter.

### M2 — `--limit` is purely client-side (no server pagination)

The server has no `limit` or `offset` query parameter. The
`-n, --limit` option must truncate client-side after fetching
all builds. This is acceptable at current scale but should be
documented as a known limitation.

### M3 — `--type rc` does not exist in the server's enum

`cbsd_proto::VersionType` has `Release`, `Dev`, `Test`, `Ci`.
There is no `Rc` variant. The `--type` values should be
`release`, `dev`, `test`, `ci`.

**Fix:** Replace `rc` with the correct enum variants.

### M4 — `build new` output shows fields not in the 202 response

The server's `SubmitBuildResponse` has only `id`, `state`, and
optional `warning`. The version, channel, image, etc. shown in
the design output must be echoed from the local descriptor, not
parsed from the response. State this explicitly.

---

## Minor Issues

- **`build get` capability requirement undocumented.** Requires
  `builds:list:any` or `builds:list:own`. With `:own` only,
  requesting another user's build returns 404 (not 403) to
  prevent enumeration. Document this.

- **`build revoke` returns 409 for terminal-state builds, not
  404.** The error handler should distinguish these status codes.

- **`--component name@ref` — the `ref` field uses serde rename.**
  Ensure `BuildComponent.git_ref` serializes as `"ref"` in JSON.

---

## Strengths

- Fetching `whoami` before submit to populate `signed_off_by`
  is correct.
- Shared `BuildDescriptorArgs` for reuse with `periodic new`.
- Error handling delegates to clap for missing required options.
