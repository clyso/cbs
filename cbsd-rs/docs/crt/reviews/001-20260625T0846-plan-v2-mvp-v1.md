## Verdict

**GO-WITH-CHANGES.** The overall architecture is sound: the reuse base
from M1–M3 is real, commit sequencing (producer → portable verifier →
internal audit) follows the design, and the LOC estimates are plausible.
However, two issues must be resolved before coding: (F1) decision 3 has a
latent byte-domain flaw that silently breaks the primary trust path in
production while every synthetic test stays green; and (F2) the plan drops
design §11 leg 1 for `MaterializationRecord`, which is additionally
blocked by a missing schema version on the new signed structure. The
remaining findings are improvements but do not block proceeding once F1
and F2 are addressed.

## 1. Findings — Blockers

### F1 — Decision 3: `source_tree_digest` byte domain is underspecified; the worktree≡tarball fixture gives false confidence

- **Severity:** Blocker
- **Confidence:** 88
- **Plan section:** §Decisions §3; §4.2 tests; §4.3
- **Design clause:** §8 ("Exact rules pinned in M4 (§14)");
  §14 (`source_tree_digest` algorithm)

**Problem.** Decision 3 pins the algorithm as `sha256(raw file bytes)` over
a worktree walk on the producer side and an extracted tarball on the
verifier side, and asserts the two must be "byte-identical." The plan's
load-bearing test fixture asserts this equivalence against a *synthetic*
repo with no `.gitattributes`. This is insufficient: the two byte domains
can diverge under real Ceph sources in at least four ways the fixture never
exercises:

1. **Git content filters.** A `.gitattributes` `text=auto`, `eol=lf`,
   `working-tree-encoding`, `ident`, `export-subst`, or a custom `smudge`
   filter can silently transform checkout bytes relative to stored blob
   bytes. The Ceph repository uses `.gitattributes` (it is a large C/C++
   project with `.gitattributes` used for linguist and conflict-marker
   settings). Producing over a worktree reflects any smudge transforms;
   `git archive` and most tarball extractors reflect only the stored bytes.
2. **`core.autocrlf`.** On Windows or in an environment with
   `core.autocrlf = true`, line endings in the worktree differ from stored
   blob bytes. A verifier on any platform extracting a tarball gets
   stored bytes; a producer on a Windows or `autocrlf` machine gets
   transformed bytes.
3. **ZIP consumer + executable bit + symlinks.** The design (§8) names
   "tarball/ZIP/UI download" as the primary verifier use-case. A ZIP
   archive produced by `git archive` or GitHub's "Download ZIP" strips the
   executable bit and resolves symlinks into their content. Decision 3
   records the executable bit per-entry and symlinks by target — but a ZIP
   consumer extracting the archive will read file bytes where a symlink
   existed, not the symlink target string, and will see all files as
   non-executable. The producer (worktree) and ZIP verifier therefore
   enumerate the same path but hash different content for symlinks.
4. **`export-subst` in `000-RELEASE/` itself.** If any future `.gitattributes`
   applies `export-subst` to `record.json` or `record.json.asc`, the
   signed bytes would be mutated on export — `$Format:...$` tokens
   expanded, breaking the signature. The plan states no
   `.gitattributes export-ignore` on `000-RELEASE/`, but also does not
   mandate a `000-RELEASE/-text` guard that would prevent eol-mangling
   of the signed files in transit. Design §3 lines 133-134 mentions
   `.gitattributes` in `bundle_digests`, implying it is expected to be
   present; the plan's 4.2 file list omits it entirely.

A synthetic fixture that passes on a toy repo is not evidence the
algorithm is correct for the real Ceph checkout. The failure mode is
silent: every test stays green, but `crt verify --tree` fails for real
consumers.

**Recommendation.** Before ratifying decision 3, the plan must do two
things:

a) **Check the real base repo.** Grep the configured base repo's
   `.gitattributes` for `text`, `eol`, `filter`, `export-subst`, and
   `working-tree-encoding` entries. If any transforming rule is present,
   the "raw file bytes" algorithm is broken as stated and must be replaced
   with a filter-independent source — the canonical choice is `git archive
   --format=tar` (which applies `export-subst` but not smudge) or
   `git cat-file blob <path>:<sha>` (stored blob bytes, no filter), with
   the algorithm definition and the fixture reflecting whichever canonical
   source is chosen.

b) **Add a `.gitattributes -text` entry to `000-RELEASE/`.** Explicitly
   record `000-RELEASE/* -text` in the `000-RELEASE/` commit's
   `.gitattributes` to prevent any eol/encoding filter from mutating the
   signed `record.json` and `record.json.asc` in a `git archive` or
   checkout. Include this file in `bundle_digests` (as design §3 implies).
   Add it to the 4.2 file list.

c) **Expand the fixture.** The worktree≡tarball determinism test must
   exercise: a file with LF and CRLF content, a file with an executable
   bit, and a symlink — asserting the algorithm produces the same hash on
   both sides (or explicitly documenting the exclusion, if symlinks are
   excluded).

Until decision 3 is corrected with this analysis, the fixture is proving
the algorithm works on the specific synthetic input, not that it is safe
for production use.

### F2 — Leg 1 is dropped for `MaterializationRecord`, and the signed structure has no schema version to gate on

- **Severity:** Blocker
- **Confidence:** 90
- **Plan section:** §4.4 ("Leg 0 extend, Leg 2 extend, Leg 3 new,
  Leg 4 new")
- **Design clause:** §11 leg 1 (lines 421-422: "`ReleaseRecord` /
  `Manifest` / `MaterializationRecord` and each referenced `PatchMeta`
  deserialize and validate")

**Problem.** Design §11 leg 1 names `MaterializationRecord` explicitly in
its "deserialize and validate" set. The plan's 4.4 enumerates the legs it
activates as 0/2/3/4 only — leg 1 for `MaterializationRecord` is absent.
This is not an intentional deferral: leg 1 is already activated for
`Manifest`/`PatchMeta` in M2 (`verify.rs:98-143`); the plan implicitly
treats legs 1–2 as already done, but leg 1 for the new in-tree structure
is genuinely new work in M4.

This gap is compounded by a structural design omission: the
`MaterializationRecord` model (design §3, lines 127-137) has **no
`schema_version` field**. The existing leg-1 implementation for `Manifest`
checks `schema_version != SCHEMA_VERSION` (`verify.rs:98`). Without a
version field on `MaterializationRecord`, leg 1 has nothing to gate on —
and a consumer receiving a `record.json` from a future schema revision
will silently accept it or fail with an unguided deserialization error
rather than a clean "version mismatch" diagnostic.

`MaterializationRecord` is a signed, externally-shipped structure — it is
the root of trust for external consumers of the git artifact. Skipping the
version mechanism here contradicts the precision applied to `Manifest`.

**Recommendation.** Before coding 4.4:

a) Add a `schema_version` field to `MaterializationRecord` in `crt-core`
   (a companion to `SCHEMA_VERSION`; a separate constant
   `MATERIALIZATION_RECORD_VERSION` avoids coupling to the manifest
   schema). The `000-RELEASE/` bundle commit (4.2) stamps it; `verify
   --tree` (4.3) checks it; and leg 1 in 4.4 validates it.

b) In 4.4, explicitly enumerate leg 1 for `MaterializationRecord` (checks
   `schema_version == MATERIALIZATION_RECORD_VERSION`) and add a test: a
   `record.json` with a future schema version fails leg 1 with the
   version-mismatch diagnostic, not a deserialization error.

## 2. Findings — Major Concerns

### F3 — `provenance.json` public-safety is enforced by convention, not by construction

- **Severity:** Major
- **Confidence:** 85
- **Plan section:** §4.2 ("`provenance.json` (public fields)")
- **Design clause:** §8 (bundle contents: `provenance.json` — "public
  patch list (provenance / risk / public justification)"); §3
  ("no `visibility`, no `justification.internal` — public-safe by
  construction")

**Problem.** The plan lists `provenance.json` as one of the `000-RELEASE/`
bundle files to be assembled in 4.2 from "public fields." But the plan
does not define what type drives the assembly. The natural source is a
projection from `ManifestEntry`, which carries `justification.internal`,
the per-patch `visibility` flag, `risk` (blast/conflict/coverage —
internal operational posture), and `Provenance::Other{description}` (which
may contain sensitive downstream patch descriptions). If `provenance.json`
is assembled by serializing a subset of `ManifestEntry` fields by caller
convention, the plan reproduces the exact pattern flagged as v5-F4
("convention not construction"): a future author adding a field to the
projection type accidentally leaks internal fields to the public artifact.

By contrast, `MaterializationRecord` is safe by construction: its type
definition (design §3 lines 127-137) structurally excludes `visibility`
and `justification.internal` — there are no such fields to accidentally
include. `provenance.json` deserves the same treatment.

**Recommendation.** Define a dedicated `ProvenanceEntry` struct (in
`crt-core`, visibility-neutral like `MaterializedPatch`) that contains only
the public fields — `blob_hash`, `patch_id`, `order`, `public_summary`,
`category`, `provenance` (public only), and optionally `risk_band`. The
4.2 assembly populates `ProvenanceEntry` from `ManifestEntry` via an
explicit projection (just as `MaterializedPatch` is a public projection of
`ManifestEntry`). This makes information-safety structural, not a "I
remembered to omit internal" at call site. Add a test asserting
`provenance.json` contains none of `visibility`, `justification.internal`,
or `Provenance::Other` descriptions.

### F4 — Decision 1 invents a third config concept without reconciling the design

- **Severity:** Major
- **Confidence:** 82
- **Plan section:** §Decisions §1 ("`materialize.repo` location")
- **Design clause:** §9 (`destination_repo: clyso/ceph`); §9 channel
  `upstream.repo`

**Problem.** The design already defines `destination_repo` at the
top-level config (§9 line 354: `destination_repo: clyso/ceph`) and
per-channel `upstream.repo` fields for the destination. Decision 1
introduces a new `materialize.repo` key without explaining how it
relates to the existing `destination_repo` field, whether it replaces
it, or how it interacts with per-namespace/per-channel overrides. This
is a schema design that will be serialized to `crt.config.yaml` and
then cannot easily be changed.

Additionally, decision 1 does not address whether the configured path
is a *local clone* of the downstream repo (from which `base_ref` must
already be fetched) or a *remote* pointing to it (requiring a fetch).
The plan's network-free test strategy (`--push` is `#[ignore]`d) only
works if `base_ref` is already present in the configured local repo;
a misconfigured remote URL passes the test but fails in production.

**Recommendation.** Reconcile decision 1 with the existing `destination_repo`
field before coding:
- If `materialize.repo` is a *local filesystem path* to a clone (the
  simplest testable form), say so explicitly and add a pre-flight check
  that `base_ref` exists in it (`git rev-parse <base_ref>`).
- If the intent is that `destination_repo` from §9 names the remote and
  `materialize.repo` is a separate local working path, document both
  fields in the `crt.config.yaml` schema update and confirm there is no
  ambiguity with the per-channel `upstream.repo`.
- Consider whether the channel-level `upstream.repo` (§9) is the right
  per-namespace/channel override, making `materialize.repo` a
  global fallback only.

## 3. Findings — Minor Issues

### F5 — Dead code at the 4.1→4.2 boundary: commit SHAs captured but not consumed

- **Severity:** Minor
- **Confidence:** 88
- **Plan section:** §4.1 ("capture each patch's commit SHA for 4.2's
  BOM")

**Problem.** Plan 4.1 ends with the patched branch built and the SHA of
each commit captured "for 4.2's BOM." If 4.1 compiles and is independently
revertable — as required by the git-commits smell test — that captured
SHA data has no consumer within 4.1: not returned from any function,
not written to any observable output, not asserted by any test. The commit
boundary smell test for the project (D6/confidence-scoring criterion) flags
code with no callers within the commit. This is the exact pattern the
project's no-dead-code invariant is designed to prevent.

**Recommendation.** One of two framings, both correct:

a) Change the framing so 4.1's SHA capture is consumed within 4.1: have
   `materialize_branch()` return `Vec<MaterializedPatch>` (with
   `git_commit` populated), which the 4.1 test asserts (assert the SHA
   is non-empty and matches the commit on the materialized branch). 4.2
   then receives that return value from the caller — no dead code in
   either commit.

b) Explicitly state that 4.1 returns `Vec<MaterializedPatch>` from a new
   function, with the 4.1 test asserting the shape, and 4.2 feeds it
   into `MaterializationRecord.patches`. This is the same as (a) but
   written as a function signature commitment.

### F6 — Leg 3 re-materialization: the plan underclarifies what "re-materialize and compare" adds beyond the patch-id walk

- **Severity:** Minor
- **Confidence:** 78
- **Plan section:** §4.4 ("Leg 3 new: … re-materialize and compare the
  pre-`000-RELEASE` source tree via `source_tree_digest`")
- **Design clause:** §11 leg 3 ("re-materializing reproduces the same
  pre-`000-RELEASE` source tree")

**Problem.** The plan correctly frames leg 3 as comparing the source tree
via `source_tree_digest` rather than commit SHAs (which are
non-deterministic across runs due to author/committer timestamps in `git
am`). This is right. But the plan does not clarify what "re-materialize"
means operationally: does it `git am` all patches onto a fresh worktree
checkout, hash the result, and compare? That involves cloning/checking
out the destination repo again — network, disk, and potentially minutes
of `git am` for a real Ceph release. Or does it hash the existing
materialized branch's pre-bundle tree directly (without re-applying
patches) and compare that hash to `source_tree_digest`?

The two approaches have different costs and subtly different semantics:
re-hashing the existing tree verifies *current tree state* (could detect
partial modifications after materialization); re-applying patches verifies
*patch application is reproducible* (but is expensive and requires a clean
checkout). The design says "re-materializing" — but the patch_id walk
already verifies patch-by-patch anchoring; it's unclear what a full
`git am` re-run adds beyond what hashing the existing tree provides.

**Recommendation.** Clarify in the plan whether "re-materialize" in leg 3
means:
- Hash the *existing* branch's pre-`000-RELEASE` tree (cheap, hashes the
  live state), or
- Apply all patches from scratch onto a new `base_ref` checkout (expensive,
  proves apply-reproducibility).

If the cheaper option is sufficient (it likely is given the patch_id walk
already anchors content), document it and remove the re-apply step. If the
expensive option is intended, flag the runtime cost in the plan's test
strategy and note the network/disk implications for decision 2's
network-free test goal.

### F7 — `sign_manifest` generalization is already done; 4.2's LOC attribution overstates the work

- **Severity:** Minor
- **Confidence:** 92
- **Plan section:** §4.2 (`crt-core`: "generalize `sign_manifest` → detached-sign
  over arbitrary bytes")
- **Code:** `cbsd-rs/crt-core/src/seal.rs:24`

**Problem.** Plan 4.2 lists "generalize `sign_manifest` → detached-sign
over arbitrary bytes" as a `crt-core` deliverable. The existing
`sign_manifest` (`seal.rs:24`) already takes `canonical_bytes: &[u8]` —
it is *already* a generic "detached-sign over arbitrary bytes" function.
`verify_manifest` likewise (`seal.rs:51`) takes `canonical_bytes: &[u8]`.
Signing `record.json` bytes (4.2) and verifying them (4.3/4.4) requires
no API change, only a new call site with the `record.json` bytes as input.
Similarly, `sign_manifest` already accepts injected key bytes and RNG,
so there is no new abstraction to build.

This inflates the 4.2 LOC estimate if the plan's author budgeted real
design effort here. If the plan means "the existing API is sufficient and
`record.json` signing is a call-site addition only," that should be stated
explicitly.

**Recommendation.** Remove "generalize `sign_manifest`" from 4.2's
`crt-core` work list. Replace with: "call `sign_manifest(rng, record_json_bytes,
vault_key, passphrase)` — no API change needed; `seal.rs` API already
accepts arbitrary bytes." Adjust the LOC estimate for 4.2 downward
accordingly.

### F8 — The `import.rs:239` patch-id reuse claim is partial; the commit→diff-bytes derivation for leg 3 is new code

- **Severity:** Minor
- **Confidence:** 85
- **Plan section:** §Reflected design invariants ("Leg 3 recomputes
  `git patch-id --stable` exactly as import did
  (`git::git_with_stdin(repo, ["patch-id","--stable"], blob)`,
  `import.rs:239`)")

**Problem.** The plan is partially correct: the `git_with_stdin` call form
and `--stable` mode are indeed reused from `import.rs:239`. However,
`import.rs` feeds `git patch-id` the raw *stored blob* (the
`git format-patch` bytes retrieved from the store). Leg 3's input is
different: it must derive the diff bytes *from a commit already on the
materialized branch* — either by fetching the blob from the store (if
still available) or by running `git format-patch -1 <commit_sha>` on the
destination repo. The commit→blob derivation step is new code, not a
literal reuse of import's path. If the plan intends to fetch the blob
from the store (referenced by the `Crt-Patch` trailer's `blob_hash`)
that is the cleaner approach and aligns with the content-addressing
contract.

**Recommendation.** Clarify in the plan: leg 3 reads the patch blob from
the store (via `entry.blob_hash`) and feeds *those stored bytes* to
`git patch-id --stable` — matching exactly what import did, and proving
the stored bytes round-trip through `git am` to the same `patch_id`.
This is clean and avoids the ambiguity of re-deriving bytes from the
materialized commit. Make this explicit in the 4.4 description.

### F9 — Exit-code mapping for 4.4's new leg failures is unspecified

- **Severity:** Minor
- **Confidence:** 80
- **Plan section:** §4.4; §4.3 ("Distinct exit codes (signature vs.
  verify vs. operational)")
- **Code:** `cbsd-rs/crt/src/verify.rs:31-44` (`VerifyVerdict`)

**Problem.** Plan 4.3 correctly specifies distinct exit codes for
`verify --tree`. Plan 4.4 activates four legs (0/2/3/4) in the
store-backed `verify` path but does not specify: (a) which `VerifyVerdict`
variant a leg-3 or leg-4 failure maps to (the existing
`verify.rs:VerifyVerdict` has `SignatureFailed`, `VerifyFailed`, but no
`LegState::Failed` — only `Passed` and `Skipped`), and (b) whether the
extended leg 0 (bundle signature check) maps to `SignatureFailed` (exit 2)
or `VerifyFailed` (exit 3).

The `LegState` enum currently has only `Passed` and `Skipped`
(`verify.rs:42-44`). Activating leg failures requires a `Failed` arm
(or the verdict short-circuits at `VerifyFailed`). The plan should say
which path it takes.

**Recommendation.** Add a `Failed(String)` arm to `LegState` (or confirm
that each leg failure early-returns a `VerifyVerdict` variant without going
through `LegStatus`). State explicitly: leg-3/leg-4 failures map to
`VerifyFailed` (exit 3); the extended leg-0 bundle signature failure maps
to `SignatureFailed` (exit 2). Add a test asserting each failure code.

## 4. Findings — Suggestions

### F10 — Verifier-first contingency should be the plan, not the fallback

- **Severity:** Suggestion
- **Confidence:** 80
- **Plan section:** §4.2 ("If it projects over ~800 LOC … I will pause
  and propose splitting")

The plan names the verifier-first split (landing 4.3 before part of 4.2)
as an escape hatch. But the worktree≡tarball determinism fixture in 4.2
cannot be fully exercised until 4.3's tarball-extraction reader exists —
the fixture needs both producer and verifier code to be meaningful. This
makes the verifier-first order logically cleaner regardless of LOC: write
the canonical `source_tree_digest` walk once and exercise it from both
sides (4.2 produces, 4.3 verifies) from the start. Consider making the
verifier-first ordering the default plan, not the fallback.

### F11 — `MaterializationRecord` should carry a `created` timestamp

- **Severity:** Suggestion
- **Confidence:** 72
- **Plan section:** §4.2 (`crt-core`: `MaterializationRecord` types)
- **Design clause:** §3 (`MaterializationRecord` field list)

`MaterializationRecord` (design §3) has no `created` timestamp. Since it
is a signed, externally-verifiable document, consumers have no indication
of *when* materialization occurred without it. This is a schema decision
that cannot be added without a version bump after the first real release is
materialized. The `MATERIALIZATION_RECORD_VERSION` recommended in F2 makes
this addition zero-cost now.
