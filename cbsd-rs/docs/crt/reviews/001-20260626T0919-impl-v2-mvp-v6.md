## Verdict

**GO (accept with findings; no blocker).** M4 delivers exactly the capability it
claims and closes the MVP verification model. Every load-bearing contract in the
review brief was checked against the code and holds (see "Verification" below).
The external-attacker threat model — the one the tool exists to defend — is
**closed**: a tampered source file, a mutated bundle file, a stripped or
wrong-key signature, and an unsigned extra bundle file each fail `verify --tree`
with the correct verdict, all proven by tests.

The story is **test coverage of the two newest internal-audit legs, not a broken
invariant**. Leg 4 (artifact faithfulness) and the in-tree leg-2b (BOM/back-ref
faithfulness) have **positive** coverage but **no negative (tamper) test** — a
verification leg that has never been shown to _fail_ is the core risk a
release-verification tool must not ship with unproven. The implementer
pre-justified the deferral ("needs a forged-but-signed record / committed
bundle"); that rationale is weak — the tests already hold a working keypair and
can forge-then-resign. These cap the score and are the required actions before
the milestone is closed; they are not a proven defect and do not gate merge,
because leg 0 + leg-2 exhaustiveness independently catch the realistic attacker.

## Verification of the load-bearing contracts

Each contract from the review brief was checked against the code, not the prose.

- **`source_tree_digest` portability (4.2 walker excludes top-level `.git`
  whether dir or gitlink-file).** `verify.rs:741` `walk_into` tests
  `rel == ".git" || rel == BUNDLE_DIR` **before** inspecting the node type
  (`verify.rs:752`), so a linked-worktree `.git` _file_ is excluded exactly as a
  clone's `.git` _directory_ is. The top-level-name match (`rel == ".git"`) does
  not exclude a nested file named `.git`. This is the "As built (c)" fix from a
  prior self-review and is now guarded by the real-worktree end-to-end test (the
  verify-tree leg checks out a detached worktree, whose `.git` is a file).
  **Holds.**

- **Producer ≡ verifier bytes (checkout, not `git archive`).** Producer:
  `materialize_branch` (`git.rs:159`) creates the worktree with
  `-c core.autocrlf=false` and `git am` runs with the same; the source blobs
  land byte-faithfully. Verifier: `checkout_detached` (`git.rs:294`) uses
  `-c core.autocrlf=false` and **never** `git archive` (which would re-apply
  `.gitattributes`). Both walk the same `walk_source_tree` over raw on-disk
  bytes. The digest is content-only (mode ignored), and symlinks are recorded by
  target — so a ZIP that drops the exec bit or a tar that preserves links both
  agree. **Symmetric; holds.**

- **Signed-by-construction (`record.json.asc` transitively covers the bundle).**
  `bundle.rs` computes `bundle_digests` over every non-record file
  (`bundle.rs:119`), assembles the record, serializes **once** (`to_vec_pretty`,
  `bundle.rs:139`), signs **those exact bytes** (`bundle.rs:140`), and commits
  the whole `000-RELEASE/` in **one** commit with `record.json.asc` already
  present (`bundle.rs:153`). The verifier reads `record.json` **verbatim** and
  verifies the `.asc` over those raw bytes (`verify.rs:593`) — no
  re-serialization, so no byte-drift risk. **Holds.**

- **Exhaustiveness both directions (no unsigned extra, no missing).**
  `verify_bundle_digests` (`verify.rs:661`) walks the on-disk `000-RELEASE/`,
  rejects any file not in `bundle_digests` ("unsigned content",
  `verify.rs:693`), rejects any digest mismatch, **and** rejects any
  `bundle_digests` key absent from disk (`verify.rs:710`). It also rejects a
  non-file node and a non-UTF-8 name. The set must equal `bundle_digests`
  exactly. Proven by `verify_tree_fails_on_an_unsigned_bundle_file`. **Holds.**

- **Leg 3 offset-invariance, non-tautological (N1-overridden).**
  `anchor_history` (`verify.rs:427`) recomputes `patch_id` from each
  **commit's** diff (`format-patch -1 --stdout <sha>` → `patch-id --stable`,
  `verify.rs:455`), never from the stored blob, and compares to the sealed
  entry's `patch_id`. The test `verify_leg3_anchors_through_an_apply_offset`
  constructs a real offset (base prepended above the hunk) and **asserts the
  materialized commit's hunk headers differ from the stored mailbox**
  (`release.rs:2236`) yet `patch-id --stable` matches — so the leg is provably
  not tautological. `verify_leg3_fails_on_a_mismatched_patch_id` proves the fail
  path. **Holds.**

- **`.gitattributes` is slash-free `* -text`.** `release.rs:622` writes
  `b"* -text\n"`; the end-to-end test asserts the checked-out bundle file is
  exactly `* -text\n` (`release.rs:1846`). A slash pattern there would resolve
  to `000-RELEASE/000-RELEASE/*` and match nothing — correctly avoided.
  **Holds.**

- **Public-safety (no `visibility` / `justification.internal`).**
  `MaterializationRecord` and `PublicProvenance` (`materialize.rs:42`, `:67`)
  carry only public fields; `PublicProvenance::from_entries` is field-explicit
  (copies four fields), so a future classification field on `ManifestEntry`
  cannot silently flow through. Proven structurally
  (`record_round_trips_through_json_and_is_public_safe`,
  `public_provenance_projects_public_fields_only_…`) and end-to-end: the
  materialize test seeds a `Private` entry with a `DO-NOT-LEAK-internal` note
  and asserts the canary appears in **no** bundle file — `RELEASE-NOTES.md` and
  `sbom.cdx.json` included (`release.rs:1832`). The branch carries no
  `Crt-Visibility` trailer (`git.rs` test `release.rs:425`). **Holds.**

- **Fail-loud / write-once.** `materialize_branch` aborts the in-progress `am`,
  removes the worktree, and deletes the partial branch on apply failure
  (`git.rs:198`, proven by `an_apply_failure_aborts_and_leaves_no_branch`).
  `write_bundle` calls `cleanup_failed(Some(tag))` on any post-build failure
  (`bundle.rs:94`), deleting branch **and** tag. The tag is refused up front if
  it pre-exists (`release.rs:657`), **before** anything is built — so a later
  `cleanup_failed` can never delete a tag this run did not create (proven by
  `materialize_refuses_a_preexisting_tag`, which also asserts no branch was
  created). **Holds.**

- **Read-only verify teardown does not mask the verdict.** `verify_ref_legs`
  captures the result, then on `wt.remove()` failure only `eprintln!`s a warning
  and returns the captured result (`verify.rs:315`). The success-path
  `write_bundle` does the same for its scratch worktree (`bundle.rs:85`).
  **Holds.**

- **`crt-core` purity.** `materialize.rs` imports only `std::collections` and
  `serde`; `source_tree_digest` is a pure combine over an in-memory `BTreeMap`.
  All IO (the directory walk, subprocess git, signing-key fetch, public-key
  fetch) lives in `crt`. **Holds.**

- **Skipped is never silently passed.** With no `--repo`, legs 3–4 are pushed as
  explicit `Skipped` (`verify.rs:240`); with `--repo` but no tag, same
  (`verify.rs:303`). `render_report` prints `[skip]` distinctly from `[pass]`
  (`verify.rs:487`). Proven by
  `a_sealed_release_verifies_and_reports_skipped_legs`. **Holds.**

## Findings

### Important (80–89)

**F1 — Leg 4 (artifact faithfulness) has no negative/tamper test.**
Confidence 85. `verify_ref_legs_in_tree` byte-compares the committed
`sbom.cdx.json` / `RELEASE-NOTES.md` against a re-derivation (`verify.rs:396`,
`byte_match`). There is **positive** coverage
(`verify_release_runs_ref_legs_on_a_materialized_release` asserts
`[pass] leg 4`), but **no** test mutates the committed bundle SBOM/notes and
asserts leg 4 reports `[FAIL] leg 4`. The existing
`verify_tree_fails_on_a_mutated_bundle_file` is **not** leg-4 coverage — it
trips leg 2's digest/exhaustiveness check and never reaches the leg-4
byte-compare. For a release-**verification** tool, a faithfulness leg never
shown to fail is the central risk. The plan's "As built (d)" admits this
deferral; the cited reason (needs a "committed bundle" forgery) is weak — the
test harness already builds a full materialized tree and holds the keypair, so
it can mutate `000-RELEASE/RELEASE-NOTES.md` post-materialize, re-sign
`record.json` over the consistent-but-unfaithful bundle, and assert
`VerifyFailed` with `[FAIL] leg 4`. (D5 partial: success path asserted; only the
tamper branch untested.)

**F2 — In-tree leg-2b (BOM + back-ref faithfulness) has no negative test.**
Confidence 82. `verify_ref_legs_in_tree` checks the in-tree record's
`s3_manifest_digest` against the sealed digest (`verify.rs:348`) and its BOM
against the sealed entries on `(order, blob_hash, patch_id)` + count
(`verify.rs:363`). These two failure branches — the guard against a
tampered-but-validly-signed `record.json` — are **never exercised**. (The shared
`verify_tree` tests cover legs 0–2 of the offline path, but not these two
ref-only extension branches.) Same forge-then-resign construction closes it. (D5
partial: success path asserted; only the failure branch untested.)

### Minor / observations (below the report threshold; recorded for completeness)

- **O1 — `.gitattributes` symmetry not proven against a hostile root
  attribute.** Confidence 60 (not counted). The tar-roundtrip fixture
  (`source_tree_digest_survives_a_tar_roundtrip_with_attrs_exec_symlink`)
  validates content hashing but bypasses git's smudge/clean path entirely, and
  the end-to-end repos carry no _transforming_ root `.gitattributes`
  (`text=auto`, `eol=`). So "checkout bytes == signed bytes under a transforming
  root attribute" is reasoned-correct (`core.autocrlf=false` + the `* -text`
  bundle override) but not test-proven. Design §14 explicitly defers the real
  `destination_repo` `.gitattributes` audit, so this is **accepted residual risk
  for the MVP**, not an M4 defect.

- **O2 — `record.json` is not reproducible across materializations.** The signed
  `created` timestamp (`materialize.rs:49`) is wall-clock at materialize time,
  so two runs produce different signed records. This is by design (the record is
  a materialize-time artifact; leg 4 re-derives only the _pure_ SBOM/notes, not
  `record.json`), and verify reads `record.json` verbatim, so determinism is
  unaffected. Recorded only to confirm it was considered. Not a finding.

- **O3 — design-ratified residual risks (not M4 defects).** Content-only hashing
  (exec bit ignored) and the clone-under-`core.autocrlf=true` divergence are
  decision-3 tradeoffs the design ratified; `verify_manifest` validating against
  the primary key only is a pre-existing M2 concern (real Clyso keys signing via
  a subkey would need attention then, not now). None are in M4 scope.
