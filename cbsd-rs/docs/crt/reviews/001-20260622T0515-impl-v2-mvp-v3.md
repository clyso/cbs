## Design-fidelity confirmations (verified, on the record)

These are the contract-critical claims the task asked to confirm; each was
checked in code, not taken on trust:

- **Digest excludes itself.** `Manifest` carries no `digest`/`signature` fields;
  both live in the `ReleaseRecord` envelope around it. `digest()` hashes
  `canonical_json(manifest)` only. Correct (design §6).
- **No float canonicalization hazard.** Risk weights/totals are `u8`; the only
  numbers serialized in a `Manifest` are `schema_version` and `order` (`u32`).
  `created` is stored as a string. No floats, no date types — no RFC 8785
  number/date-canonicalization edge can shift the signed bytes.
- **`risk_total`/`risk_band` are computed, not stored** — methods on
  `ManifestEntry`, absent from the serialized struct, so a later band
  re-calibration cannot alter a signed manifest (design §3/§6).
- **Risk rubric matches concept §6.1.** Three authored axes (blast/conflict/
  coverage, 1→3) plus the derived `upstream` axis from provenance; total 4–12;
  bands 4–6/7–9/10–12. `Band::from_total` and `upstream_weight` are exercised at
  all boundaries by tests. The `Declined → 3` mapping (M1 addition) is
  defensible: a patch closed-unmerged upstream is effectively a downstream-only
  carry, which the rubric already weights 3.
- **`crt-core` purity holds.** `sign_manifest` takes caller-supplied armored key
  bytes and an injected `R: Rng + CryptoRng`; the crate sources neither keys nor
  entropy. No `tokio`/IO. `pgp` is `default-features = false` (no C dep).
- **Sign/verify failure modes covered.** Tests assert tampered bytes, wrong key,
  and a malformed signature all return `Err`. Errors map `pgp` failures into
  `CrtCoreError::Pgp(String)` — the message stringifies the `pgp` error, which
  describes the operation, not key material; no key bytes are interpolated.
- **Write-once is atomic.** `put_release` uses `PutMode::Create` and maps
  `Error::AlreadyExists → StoreError::ReleaseExists`. There is no `head`-then-
  `put` path; no TOCTOU window. `put_draft` uses plain `put` (freely
  overwritable), and `delete_draft` is a store primitive only — seal-time
  deletion is 2.5, correctly out of scope here.
- **Envelope storage vs. signed bytes are decoupled.** The store persists the
  `ReleaseRecord` via `to_vec_pretty`, but the digest and signature are computed
  over `canonical_json(manifest)`. The pretty-printed stored bytes never feed
  verification, so the storage encoding cannot perturb the cryptographic
  contract. Sound.

## Deliberate design deviations (judged intentional, not defects)

The plan records two deviations from the design snapshot; both are sound and
consistently implemented:

1. **Drafts are store-backed and mutable** (`drafts/` prefix), not local files.
   Rationale (no work stranded on one operator's machine) is sound; the
   implementation cleanly separates the mutable `put_draft` (plain `put`) from
   the write-once `put_release` (`PutMode::Create`). Concurrency is documented
   last-writer-wins (serial handoff) — acceptable for the MVP and flagged in the
   plan, not mistaken for safe concurrent editing.
2. **`list` enumerates the prefix** as the source of truth;
   `index/releases.json` is deferred as a re-derivable cache. Avoids the
   write-amplification and read-modify-write race a maintained index would add.
   Consistently applied via the shared `list_keys(prefix)` helper for both
   drafts and releases.

These are **library** commits inside the M2 phase; the design's "a crt-core
crate with no caller is dead code" is a **milestone-level** statement (M1–M4 as
capabilities), and the approved plan places the CLI consumers in 2.4/2.5/2.6.

same-commit unit tests, and clippy is clean with **no `#[allow(dead_code)]`**.
Each commit compiles alone (deps track usage: `serde_json_canonicalizer` lands
with `manifest.rs` in 2.1; `pgp`+`rand` with `seal.rs` in 2.2; `object_store`
draft/release ops with their tests in 2.3), is single-concern, and is

## Findings

### F1 — No `PrefixStore` round-trip test (Important)

**Confidence 82.** The single stated correctness concern for 2.3 — that `list`
round-trips under an S3 prefix — is verified here only by reading `object_store`
source. The repo has **no** test that wraps `InMemory` in a `PrefixStore` and
asserts `list_drafts`/`list_releases` still recover the keys; the only prefix
exercise is the `#[ignore]`d real-S3 test, which never runs in `cargo test`. The
behavior is correct today (`strip_meta` strips the prefix), so this is a
coverage gap on a load-bearing path, not a bug. **Fix:** a ~10-line test —
`ObjectBackedStore::new(Arc::new(PrefixStore::new(InMemory::new(), StorePath::from("crt/"))))`,
put a draft + a release, assert both `list_*` return the unprefixed
`ReleaseKey`s — locks what currently requires source reading to trust. (D5:
untested critical path — the prefix-aware list seam.)

### F2 — "Byte-level golden" is a digest golden, not a byte-string golden (Low)

**Confidence 80.** `manifest.rs` comments the test as a "byte-level golden"
locking the "canonical-digest contract," but the assertion is
`assert_eq!(d1.to_hex(), DIGEST_GOLDEN)` — it pins the SHA-256 of the canonical
bytes, never the canonical JSON **string**. This is cryptographically sufficient
to catch any canonicalizer or schema drift (so it is **not** a tautology and not
a defect), and the digest const is a fixed literal, so the lock is real.
However: (a) the comment overstates it ("byte-level" implies the bytes are
asserted), and (b) on failure you see two differing 64-char hex strings with no
indication of _what_ changed. **Fix (optional):** add an
`assert_eq!(String::from_utf8(canonical_json(&m)?)?, GOLDEN_JSON)` alongside the
digest assertion — matches the comment and makes drift diagnosable. (D11:
comment slightly misdescribes the test.)

### F3 — `list` silently drops keys with `/` or empty segments (Low)

**Confidence 80.** `parse_release_key` requires exactly 4 path parts and the
right prefix. A `ReleaseKey` whose `name`/`channel`/`namespace` contains a `/`
splits into >4 parts (→ `None`); one with an empty segment collapses under
`Path`'s empty-segment skip (→ <4 parts → `None`). In both cases the object is
still **written** by `put_draft`/`put_release` and still **retrievable** by
`get_*` (which build the path identically), but is **invisible to `list`** — a
write/list consistency gap. There is no validation on `ReleaseKey` fields in
this layer. This is **latent, not active**: release names are prefix-resolved
(`ces-v18.2.0`, which round-trips correctly — there is a test) and validated
upstream at `release new` (commit 2.4, not yet landed). **Fix:** validate
`ReleaseKey` segments (non-empty, no `/`) at construction, or document the
invariant where keys are minted in 2.4. Note for the 2.4 reviewer, not a blocker
here. (D7-adjacent: unvalidated input feeding a key path; downgraded — no public
endpoint, no live caller yet.)

### F4 — 2.2 below the size floor (Note, no deduction)

**Confidence n/a.** ~70 authored lines, under the ~200 floor. Intentional and
plan-justified (separately-revertable integrity vs. authenticity seam). Recorded
for completeness; passes the smell test. No deduction.

## Forward notes (not findings)

- **Vault keys & signing subkeys (2.5).** `sign_manifest` signs with the secret
  **primary** key. Real-world OpenPGP keys often carry a sign-only **subkey**
  and keep the primary for certification. This matches the design as written,
  but the 2.5 reviewer should confirm the Vault-provisioned key signs from the
  primary (or extend `sign_manifest` to select a signing subkey) before a real
  key is wired.
- **Canonicalizer choice.** `serde_json_canonicalizer` 0.3 was selected over the
  §14-flagged-under-maintained `serde_jcs`. The 2.1 golden test guards a future
  swap; the choice is sound.

## Verdict

**GO.** Commits 2.1–2.3 faithfully implement their M2 plan slice: the
canonical-JSON digest contract is locked, the digest excludes itself, risk

prefix-stripping `list` that round-trips correctly under an S3 prefix. The two
deliberate deviations from the design (store-backed mutable drafts;
enumerate-the-prefix listing) are sound and consistently implemented. Tests pass
and clippy is clean with no suppressions. The findings are one Important
coverage gap (F1) and two Low precision/robustness items (F2, F3); none blocks
proceeding to 2.4.
