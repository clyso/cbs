## Design-invariant audit (the three load-bearing invariants)

All three hold ‚ÄĒ stated explicitly so findings are not manufactured:

1. **`blob_hash` ‚Č† `patch_id`, distinct identities.**
   `blob_hash = sha256(raw format-patch bytes)` (`crt-core::blob_hash`),
   `patch_id = git patch-id --stable` (`import.rs:202`). Distinct types, stored
   under distinct keys (`patches/blobs|meta/sha256/‚Ä¶` vs
   `patches/by-patch-id/‚Ä¶`). ‚úď
2. **Matching `patch_id`, different `blob_hash` ‚áí STORE + WARN, never skip.**
   `put_blob`/`put_meta` run unconditionally (`import.rs:236‚Äď238`);
   `equivalent_to` is set via `existing.filter(|e| *e != hash)`
   (`import.rs:219`) and `main.rs:133‚Äď138` warns to stderr. Re-importing the
   **same** blob yields `equivalent_to == None` (the `filter` drops the equal
   hash), so an exact re-import is correctly not flagged. ‚úď The design ¬ß4
   phrasing ("references it rather than storing") reads more loosely than the
   task's authoritative invariant 2; the **code follows invariant 2** (store +
   warn), which is the safe behavior.
3. **Bytes always from local `git format-patch`; octocrab metadata-only.**
   `import_pr` calls octocrab only for PR metadata, fetches head/base into the
   local repo, then `import_commit` runs `git format-patch ‚Ä¶ --stdout`
   (`import.rs:182‚Äď196`). GitHub's `.patch`/`.diff` endpoint is never touched. ‚úď

The `format-patch` invocation pins `format.signoff=false`,
`format.subjectPrefix=PATCH`, `--no-signature`, `-N` against ambient git config,
and a test asserts the version trailer is stripped ‚ÄĒ `blob_hash` reproducibility
is genuinely defended, not assumed.

## Findings (ordered by severity)

### Important ‚ÄĒ F1. The merged-PR `base..head` range is an unverified, untested external-API assumption on the tool's primary path

`import_pr` derives the commit set as
`rev-list --reverse --no-merges {base_sha}..{head_sha}` (`import.rs:114`,
`162`). The whole merged-PR path rests on the comment's claim
(`import.rs:108‚Äď113`) that GitHub "freezes a PR's `base.sha` ‚Ä¶ it does not
advance to the post-merge tip." Three problems:

1. **The claim is not authoritatively established.** GitHub's REST "Get a pull
   request" reference does **not** document `base.sha` post-merge behavior.
   Independent sources conflict: one widely-cited technical writeup states
   `base.sha` is frozen at PR-creation time (supports the code); GitHub
   community discussions (#39880, #59677) report observing it track the base tip
   and request that it be frozen (contradicts the code). A load-bearing
   correctness assumption rests on contested behavior.

2. **The dominant failure mode is plausibly the _primary_ path.** The tool's
   stated target is **Ceph** PRs, and Ceph integrates via **merge commits**, not
   squash/rebase. If `base.sha` is (or becomes) an ancestor of `head_sha` ‚ÄĒ
   exactly what a merge-commit merge produces once the base advances ‚ÄĒ then
   `rev-list base..head` yields **empty**, and the import silently produces zero
   patches. The code's own warning text (`main.rs:121‚Äď124`, "the commits are
   already on the base branch") admits this case exists.

3. **A second, distinct wrong-result edge even if `base.sha` is frozen.** If the
   PR author merged the base branch _into_ the feature branch (a common
   conflict-resolution move), `base.sha..head` pulls in those merged-in base
   commits as if they were PR patches. `--no-merges` drops only the merge commit
   itself, not the base commits it introduced ‚ÄĒ so unrelated commits are
   imported and content-addressed as part of the PR. This is a wrong-result, not
   an error.

**Coverage gap (D5).** The only test that exercises `import_pr` is
`import_pr_real`, which is `#[ignore]`d, needs network + a hand-supplied
`CRT_TEST_PR_URL`, and does not systematically cover a merge-commit-merged PR. A
single passing `--ignored` run against one PR is not coverage for this class of
failure. The non-ignored tests (`equivalent_patch_id_is_flagged`,
`imports_a_range_into_the_store`) exercise only `import_range`.

**Recommended fix (robust, still invariant-3-compliant):** enumerate the PR's
commits via the GitHub API (`pulls().list_commits()` ‚ÄĒ metadata only) for their
SHAs, then `git format-patch` each one locally. This sidesteps the range
entirely and closes both the empty-result and merged-in-base edges.
Alternatively: keep `base..head` but (a) verify and cite the `base.sha`
semantics in the comment, and (b) add a **non-ignored** fixture that constructs
a local merge-commit-merged topology and asserts the PR commit set is recovered
exactly. Until one of these lands, treat the merged-PR path as unproven.

### Minor ‚ÄĒ F2. A closed-but-unmerged PR is mislabeled `OpenInReview`

`pr_state` (`import.rs:145‚Äď151`) maps `(merged, base_ref)` and falls through to
`OpenInReview` for `merged == false`. A PR that was **closed without merging**
(declined) is therefore recorded as in-review provenance, which is inaccurate
and could mislead the M2 risk rubric (design ¬ß6.1 derives the `upstream` risk
axis from `UpstreamPrState`). This is bounded by the design: `UpstreamPrState`
(design ¬ß3 / `meta.rs:22`) has no closed/declined variant, so it is a **design
limitation surfaced**, not purely a code bug. Worth a one-line note in the doc
comment, and a candidate enum addition for M2. (`ApprovedOpen` is likewise never
produced, but the doc comment explicitly defers approval-state fetching to
post-M1 ‚ÄĒ acknowledged, not dead code.)

### Minor ‚ÄĒ F3. The GitHub token is not threaded to the `git fetch`, so private-PR support is absent and the doc implies otherwise

`github_token` is passed only to `octocrab.personal_token` (`import.rs:61‚Äď62`).
The subsequent `git fetch https://github.com/owner/ name.git ‚Ä¶`
(`import.rs:90‚Äď101`) is **unauthenticated**. For a **private** PR, metadata
fetch succeeds (token) but the `git fetch` then fails ‚ÄĒ even with a valid token.
The CLI help and doc comment ("public PRs work without one") imply private PRs
work _with_ one; they will not. Note the tension: embedding the token in the
fetch URL would leak it via process args (`ps`/`/proc`), so _not_ threading it
is a defensible secret-hygiene choice ‚ÄĒ but then private-PR support should be
stated as an explicit M1 limitation rather than left implicit. For a public Ceph
target this is acceptable for M1 **if documented**. (Properly, use a credential
helper or `http.extraheader` via env, not the URL.)

### Observations (no deduction)

- **`patch_id_path` interpolates the raw `patch_id` into the object key**
  (`crt-store/src/lib.rs:68‚Äď70`). `git patch-id --stable` output is hex and the
  empty case is rejected (`import.rs:205` `.context(...)`), so there is no real
  injection surface; the value is locally computed, never remote-controlled.
  Nitpick at most.
- **Partial-write window is documented and accepted** (`import.rs:233‚Äď235`):
  blob-then-meta can leave a blob without meta on crash; a re-import heals it.
  Reasonable for the CLI; the future service makes it transactional.
- **`import_commits_in_range` is a clean shared seam** for the range and PR
  paths ‚ÄĒ no duplication between them (no D2). The `Provenance` is built once
  per path and threaded through, replacing the prior per-commit
  `format!("{source_repo} {range}")`.
