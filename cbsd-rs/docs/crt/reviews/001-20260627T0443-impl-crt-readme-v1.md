## Findings (by severity)

### F1 — "a signed tag" is a false trust claim (Important)

- **README line 8:** "…materializes it into a linear git branch plus a
  self-verifying `000-RELEASE/` bundle **and a signed tag**."
- **Contradicted by the code:** `bundle.rs:169–178` creates an **annotated** tag
  with `-c tag.gpgsign=false`, and the code comment states "a signed tag over
  the git graph is §8 future work."
- **Contradicted by the README itself:** the out-of-scope list at **line 376**
  says "Signed git tags / cosign… not yet built," and every other mention (lines
  53–54, 264–266) correctly says **annotated** tag carrying the manifest digest.
- **Why it matters:** this is in the trust-laden opening paragraph. It claims a
  cryptographic property (a _signed_ tag) the tool deliberately does not
  provide, and the README internally contradicts itself. A reader calibrating
  trust on the intro would over-credit the git tag.
- **Fix:** one word — change "a signed tag" to "an annotated tag" (matching
  lines 54 and 266).
- **Confidence the claim is wrong:** 96.

### F2 — exit-code table is incomplete and slightly imprecise (Minor)

- **README lines 307–312** document exit codes `0`, `2` ("signature verification
  failed (leg 0)"), and `3`. The table covers only the verify-verdict codes
  (`EXIT_SIGNATURE = 2`, `EXIT_VERIFY = 3` in `main.rs:30–32`).
- **Missing / imprecise:**
  - **Code 1** — operational failures (no such release, store/Vault/git error,
    missing public key, re-seal refusal) return `Err` from
    `main() -> Result<()>`, which `anyhow` renders and exits **1**. Not in the
    table.
  - **Code 2 is also clap's usage-error code** — a bad/missing argument exits 2
    via clap, not only a signature failure. The table's "code 2 = signature
    failed" reads as exhaustive when it is not.
- **Why it matters:** anyone scripting `crt` (CI gating on verify) will treat
  the table as the contract; conflating operational failure (1) and usage error
  (2) with the verify verdicts can mis-gate a pipeline.
- **Fix:** add a row for `1` ("Operational/usage error — bad arguments, no such
  release, store/Vault/git failure"), and scope the `2`/`3` rows to the verify
  verdict ("verify run reached a verdict; signature leg failed", etc.).
- **Confidence:** 88.

### F3 — "warns on startup" overstates when the secrets warning fires (Minor)

- **README lines 105–106:** "`crt` warns on startup if the secrets file is
  group- or world-readable."
- **Actual behavior:** the permission warning lives in `secrets::load`
  (`secrets.rs:44–63`), which is only called for an **S3 store**
  (`main.rs:236`), `release seal` (`main.rs:390`), and `release materialize`
  (`main.rs:436`). A local-store invocation of `release list` / `info` /
  `verify --tree` never reads the secrets file, so no warning fires "on
  startup."
- **Fix:** reword to "`crt` warns when it loads the secrets file (for an S3
  store, `release seal`, or `release materialize`) if it is group- or
  world-readable."
- **Confidence:** 82.
