## Design-invariant audit (unchanged — re-verified, not assumed)

Because `import_commit` was not touched, the three trust invariants from v1
still hold; re-checked against the current source:

1. **`blob_hash` ≠ `patch_id`.** `blob_hash = sha256(format-patch bytes)`
   (`import.rs:236`), `patch_id = git patch-id --stable` (`import.rs:238`),
   distinct types, distinct store keys. ✓
2. **Matching `patch_id`, different bytes ⇒ STORE + WARN, never skip.**
   `put_blob`/`put_meta` run unconditionally (`import.rs:273–274`);
   `equivalent_to = existing.filter(|e| *e != hash)` (`import.rs:255`) drops the
   equal hash, so an exact same-blob re-import is **not** flagged. The
   `equivalent_patch_id_is_flagged` and `imports_a_range_into_the_store`
   (re-import) tests still pass. ✓
3. **Bytes always from local `git format-patch`, never GitHub `.patch`.**
   `import_commit` runs `git format-patch -1 … --stdout` (`import.rs:218–232`).
   octocrab is now used for **two** metadata reads (PR `get` + `pr_commits`) and
   zero byte reads — the `.patch`/`.diff` endpoint is still never touched. ✓

## Finding resolution (F1 / F2 / F3)

### F1 (was Important / D5) — RESOLVED (correctness) / PARTIALLY RESOLVED (coverage)

The fragile `base..head` range is **gone**. `import_pr` now enumerates the PR's
commits authoritatively from the API (`pulls().pr_commits(n).send()` +
`all_pages`, `import.rs:106–115`), drops merge commits by `parents.len() <= 1`
(`import.rs:124–128`), and **bails loudly** on an empty set
(`bail!("… has no non-merge commits to import")`, `import.rs:129–131`) —
converting the v1 silent-empty-result failure into a hard error. The base-branch
`git fetch` is removed; only `fetch pull/<n>/head` remains
(`import.rs:135–144`). This is **sound**: every non-merge PR commit and its git
parent (including the branch point) is an ancestor of head, so
`format-patch -1 <sha>` resolves locally for each — the removed base fetch was
never needed once the SHA set comes from the API.

Both v1 wrong-result edges are closed: the empty-range case now bails, and the
"base merged into feature branch" case can no longer pull in base commits
because the commit set is the PR's own commit list, not a range.

**Commit ordering is not load-bearing for M1.** GitHub's `/pulls/{n}/commits`
returns commits oldest-first (the PR "Commits" tab order) and the new fixture's
`rev-list --reverse` matches that shape. But even if the order were reversed, M1
output would be byte-identical: `format-patch -1` addresses each commit
independently against its own git parent, and provenance records only
`head_sha`, never the enumerated list — nothing in the store persists sequence.
The only order-sensitive effect is **which** of two same-`patch_id` commits in
one PR becomes the index representative, which is cosmetic (both blobs are
stored). Apply-order correctness is an M2 concern, and M2 must re-derive order
regardless since M1 does not store it.

**Why coverage is only PARTIALLY resolved (the residual D5).** The two new tests
exercise _adjacent_ paths, not the API path itself:

- `range_skips_merge_commits` drives `import_range` → tests git's `--no-merges`.
  It genuinely constructs a real 2-parent merge commit (`git merge --no-ff` of
  `side` into `mainline`, `import.rs:496–501`) and asserts the two single-parent
  commits import while the merge is skipped — a correct, valuable fixture. But
  it exercises the **git** filter, not the octocrab `parents.len() <= 1` filter.
- `import_shas_records_pr_provenance` drives `import_shas` directly with
  hand-built SHAs → tests the shared tail and `UpstreamPr` provenance
  round-trip, not the enumeration.

The actual new F1 code path — `get` → `pr_commits` → `all_pages` → 250-warn →
`parents.len()` filter → empty-`bail!` → head fetch — has **zero non-ignored
coverage**; the single most load-bearing new line (the API merge filter) is
asserted by inspection only. `import_pr_real` remains `#[ignore]`d and is not
systematic. This is defensible (octocrab cannot be unit-tested without a network
mock or an HTTP fixture, which M1 does not set up), but it is a real residual
gap: one D5 deduction, down from v1's full weight because the **correctness**
root cause is eliminated and the empty case is now loud rather than silent.

### F2 (was Minor / D8) — RESOLVED

`UpstreamPrState::Declined` is added to `crt-core` (`meta.rs:22–30`, with a doc
comment and `#[serde(rename_all = "kebab-case")]` so it serializes as
`"declined"`) and to the design §3 type block (`design …:91–92`). `pr_state` now
takes `closed` and maps `(false, true, _) ⇒ Declined` (`import.rs:182–189`). A
closed-but-unmerged PR is no longer mislabeled `OpenInReview`.

The merged-PR regression risk is handled correctly: GitHub reports merged PRs as
`state == "closed"`, but `pr_state` checks `merged` first, so
`pr_state(true, true, "main") ⇒ MergedMain`. This exact case is asserted by the
updated `pr_state_mapping` test (`import.rs:438`). Design and code agree.

### F3 (was Minor / D11) — RESOLVED

The `--github-token` help text now reads "GitHub token for the API (raises rate
limits). The git fetch is anonymous, so private-repo PRs are unsupported"
(`main.rs:67–68`), and the `import_pr` doc comment states the token
"authenticates the GitHub **API** calls … The local `git fetch` is anonymous, so
PRs on **private** repositories are not supported in M1" (`import.rs:62–64`).
The misleading "public PRs work without one" phrasing is gone; the private-PR
limitation is now explicit. Token hygiene still holds: the token reaches only
`builder.personal_token` (`import.rs:74–76`) and never the fetch URL or any log
line.

## New problems introduced by the remediation

Adversarial hunt across the diff. Nothing rises to a deduction beyond the
residual F1/D5; the items below are observations or minor notes.

- **`>= 250` is correct, not a flaw (no deduction).** GitHub caps the endpoint
  at 250, so `> 250` could never fire. A PR with exactly 250 commits is not
  truncated yet still triggers the warning — a conservative false positive that
  is the only sane trade-off without a second API call to count commits. The
  warning text correctly directs the operator to `--range`. (`--range` itself
  has no commit cap — it is a local `rev-list` — so the fallback is genuine.)
- **`per_page` is unset (minor efficiency).** `pr_commits` is sent without
  `.per_page(100)`, so it defaults to GitHub's 30/page; a 250-commit PR costs ~9
  round-trips and burns the rate limit faster. Functionally correct (`all_pages`
  collects them all); a one-liner `.per_page(100)` would cut it to 3. Note, not
  a deduction.
- **`Declined` has no dedicated round-trip test (minor).** The variant rides the
  existing derive and `patch_meta_round_trips_through_json`, but no test asserts
  `Declined` specifically serializes/deserializes as `"declined"`. Low-risk (the
  derive is mechanical); worth a line in M2.
- **Plan prose is now mildly stale (observation, out of scope).** The plan
  `001-…-01-store-and-import.md` was not touched and still describes 1.3 in
  terms of the original approach. Plans are point-in-time snapshots, so this is
  acceptable; if the remediation is squashed into 1.3 the plan's 1.3 narrative
  could optionally be refreshed, but it is not a current-state contract.
- **`Vec::with_capacity(shas.len())` in `import_shas`** (`import.rs:200`) and
  the `head_sha` move instead of `.clone()` (`import.rs:148`) are correct,
  idiomatic 2024-edition cleanups enabled by the rewrite. No issue.
- **No panic on remote-controlled input.** Every PR-field access is
  `.as_ref().map(..).context(..)` or `matches!`; `c.sha` is non-optional in the
  model; the empty set bails. No `.unwrap()`/`.expect()` outside `#[cfg(test)]`.
