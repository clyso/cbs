> **Maintainer disposition (added 2026-06-25, post-review; plan author and
> maintainer in agreement).** **N1 is overridden — its remedy is not adopted.**
> N1 recommends feeding leg 3's stored _blob_ bytes to `git patch-id --stable`
> instead of the materialized commit's diff. That contradicts design §11 leg 3,
> which requires the **commit's** recomputed `git patch-id --stable` to equal
> the entry's `patch_id`, **offset-invariant by design**. Computing from the
> stored blob would merely re-derive the import-time `patch_id` (trivially equal
> — that is how it was computed) and verify nothing about what `git am` actually
> landed on the branch, defeating leg 3's purpose; `patch-id --stable` is
> offset-invariant precisely so the materialized commit's diff (applied at an
> offset) still matches. The kernel of truth in N1 _was_ adopted: the
> v1-revision's offset-fixture wording was wrong ("blob_hash differs" — in fact
> the blob is unchanged; it is the commit's diff bytes that differ at an
> offset), so the plan's leg-3 fixture wording is corrected and a "compute from
> the commit" rationale added. The binary/rename/mode `patch-id` instability N1
> gestures at is the existing §14 M1 open item, not a new M4 blocker. **N2
> (Major)** and **F11** were accepted and folded into the plan.

## Verdict

**GO-WITH-CHANGES.**

The two v1 blockers (F1, F2) are genuinely resolved in the revised plan, not
papered over. The verifier-first reordering (4.1 branch → 4.2 digest and
`verify --tree` → 4.3 bundle → 4.4 legs) is architecturally sound. One new
blocker (N1) is introduced by the revision's handling of the F8 clarification:
the plan still misspecifies the input to leg 3's `git patch-id`, creating an
incoherence that would cause leg 3 to fail silently on offset-applied patches in
production. One new major concern (N2) is that the verifier-first 4.2 has a
latent dead-code risk at the 4.2→4.3 boundary that the plan needs to pre-empt.
The remaining v1 findings (F3–F9/F11) are folded or resolved at an acceptable
level; F10 is adopted as the default ordering (its recommendation was accepted).

## 2. New Findings

### N1 — Leg 3 patch-id input source is incoherent; the offset-fixture

       will pass by accident, not by proof

- **Severity:** Blocker
- **Confidence:** 87
- **Plan section:** §Reflected design invariants ("Two hashes, two jobs;
  patch-id reuse is partial (F8)"); §4.4 ("Leg 3 new")
- **Design clause:** §11 leg 3 ("recomputed `git patch-id --stable` equals that
  entry's `patch_id`")

**Problem.** The revised plan says leg 3 "derives the diff bytes from each
materialized **commit** (`git show`/`format-patch -1`) rather than a stored
blob, then pipes that to `patch-id`." It simultaneously claims this is "partial
reuse" of `import.rs:239`.

`import.rs:239` feeds the raw stored blob (the exact bytes produced by
`git format-patch -1 --no-signature -N --stdout <sha>` at import time, from the
_source_ repo, with fixed options) to `git patch-id --stable`. That is the
canonical input that defines the stored `patch_id`.

If leg 3 re-derives the diff from the _destination_ repo's materialized commit
via `git format-patch -1` (or `git show`), the byte sequence produced may differ
from the stored blob because:

1. `git format-patch` output includes the From-header date field
   (`From <sha> Mon Sep 17 00:00:00 2001`) — constant for the stored blob but
   regenerated at leg-3 run time from the destination commit's timestamp.
2. `git am` may apply the patch with an offset hunk; the resulting commit's diff
   differs in line numbers from the original blob diff. This is exactly the
   scenario the plan's "offset fixture" is supposed to validate.
3. The destination repo at leg-3 time may have commits on `base_ref` not present
   at import time, altering the context lines in `format-patch` output.

`git patch-id --stable` is offset-invariant over _hunk content_ but not over
`format-patch` headers or context-line differences introduced by running
`format-patch` against a different commit object. The plan's offset fixture will
pass only if git's `patch-id` happens to absorb these differences — which it
does for hunk offsets, but not necessarily for header timestamps or context
divergence.

The clean resolution (already hinted at in v1 F8 but not adopted) is for leg 3
to fetch the stored blob from the store via `entry.blob_hash` and feed _those
exact bytes_ to `git patch-id --stable` — matching import's input precisely and
proving the stored bytes survive `git am` to the same `patch_id`. This avoids
re-deriving bytes from the destination repo entirely.

The plan as written says leg 3 works from the materialized commit's diff, not
the stored blob. That is the incoherence.

**Recommendation.** Revise the 4.4 description to read: leg 3 reads each entry's
blob from the store (via `entry.blob_hash`, already in the `Crt-Patch` trailer),
feeds those stored bytes to `git patch-id --stable`, and compares the result to
the entry's `patch_id` — the same operation import performed. The `Crt-Patch`
trailer identifies which blob; the store provides the bytes. The offset fixture
then proves that `git am`-applied-with-offset still yields the same `patch_id`
as the stored blob, which is the meaningful claim. The `git_commit` in
`MaterializedPatch` is used only to _find_ the commit, not as the diff source.

### N2 — 4.2's `MaterializationRecord` is dead code until 4.3; the

       plan must pre-empt this at the commit boundary

- **Severity:** Major
- **Confidence:** 83
- **Plan section:** §4.2 ("verifier-first per F10: this lands before the
  producer 4.3 that emits what it verifies"); §4.2 `crt-core` deliverables
- **Design clause:** §12 testing ("crt (integration): …materialize; assert…")

**Problem.** 4.2 adds `MaterializationRecord`, `MaterializedPatch`,
`schema_version`, `MATERIALIZATION_RECORD_VERSION`, and `PublicProvenance` to
`crt-core`, and adds `crt verify --tree` to `crt`. Until 4.3 exists, no code
calls `MaterializationRecord::new()` or serializes it to `record.json`. In the
4.2 commit, the types are declared and consumed only by:

- The `verify --tree` deserializer (reads `record.json` from disk).
- The hand-built `record.json` fixture in the test.

This is the same dead-code shape that v1 F5 flagged at the 4.1→4.2 boundary (and
the plan resolved F5 by removing the dead carry-forward). The revised plan's
verifier-first ordering reintroduces the structurally same pattern at the
4.2→4.3 boundary.

The difference from F5 is that 4.2's "dead code" is intentional (the
verifier-first ordering is correct) and the hand-built fixture consumes the
types. But the fixture must be carefully constructed: if the `record.json`
fixture is hand-authored with only the fields the plan names now, and 4.3
discovers it needs an additional field or a structural change, the fixture will
need a fixup, and the 4.2 commit will not have compiled correctly with the final
4.3 types.

More concretely: `MaterializationRecord.bundle_digests` is a
`BTreeMap<String, Sha256>` populated in 4.3 from every `000-RELEASE/` file. If
4.2's hand-authored fixture omits a file that 4.3 adds, the leg-2
cross-reference in 4.4 (which checks `bundle_digests`) will fail on a real
bundle. The fixture is testing a schema, not a real bundle, so it cannot prove
exhaustiveness.

Additionally, `source_tree_digest` is computed by the 4.2 directory walker.
Until 4.3 exists, there is no end-to-end proof that the walker's output matches
the value embedded in a real `record.json`. The fixture proves the walker is
self-consistent; the end-to-end proof only exists after 4.3 lands.

This is not a reason to reject the verifier-first ordering — it is correct. But
the plan must explicitly acknowledge:

a) The 4.2 fixture is a _schema smoke-test_, not a complete correctness proof;
the end-to-end proof is in 4.3's test. b) The `bundle_digests` set in the 4.2
fixture must match the exact file list 4.3 commits to; any 4.3-added file that
is not in the fixture should cause a test failure in 4.3 (not silently pass). c)
The `crt-core` types added in 4.2 may not compile cleanly if 4.3 adds fields —
track the type as a commitment, not a draft.

**Recommendation.** Add a note in 4.2: the hand-built fixture pins the
`bundle_digests` file list; 4.3 must assert its actual file list exactly matches
the set in the fixture (or the fixture is updated as part of 4.3, in the same
commit). This turns the 4.2→4.3 boundary into a checked invariant rather than an
unchecked assumption.

## 3. Open Questions

**OQ1.** F1's decision 3 defers exec-bit and symlink treatment for ZIP consumers
to 4.2. The design (§8) names ZIP/UI-download as the primary recipient path. If
the real destination repo contains symlinks (unlikely but possible for a large
C/C++ project), the "hash content only (ignore mode) and record symlinks by
target" framing produces a `source_tree_digest` that a ZIP recipient (which
collapses symlinks into their content) cannot reproduce. The plan leans toward
content-hash-only and symlink-by-target but does not commit. This must be
settled in 4.2 before 4.3 emits any `record.json` — it is not deferrable past
4.2.

**OQ2.** Leg 3's `source_tree_digest` recomputation in 4.4 walks "the
checked-out pre-`000-RELEASE` tree" on the materialized branch. This requires
the destination repo worktree to be checked out at that point. Does
`crt release verify` assume the destination repo is available and checked out,
or does it fetch the tree from git object storage? The plan does not say. For an
operator running `verify` on a CI system without the full destination repo
checked out, this could fail silently with an operational error rather than a
verify failure.

**OQ3.** 4.3 walks the branch's `Crt-Patch` trailers to rebuild the patch BOM.
This walk is new code not described in `git.rs` today. The plan names it in 4.3
but does not describe its error handling: what happens if a `Crt-Patch` trailer
is missing on a commit (e.g., a commit that was manually added to the branch)?
Should `materialize` prevent this, or should leg 3 detect and fail on it?
