## Design / plan fidelity

- **¬ß11 legs 0‚Äì2** are implemented in `verify_release` in the documented order:
  leg 0 signature fail-fast (`verify.rs:91‚Äì95`), leg 1 `schema_version` check
  (`98‚Äì104`), leg 2 recomputed-digest + blob-presence + denormalized-`patch_id`
  agreement (`106‚Äì143`). Legs 3‚Äì4 are emitted as `skip` rows
  (`verify.rs:159‚Äì163`) and `render_report` labels them `[skip]` so "not run"
  cannot read as "passed." Matches ¬ß11 and the plan's leg split.
- **Distinct exit codes** (¬ß11): signature ‚áí 2, verify ‚áí 3, operational ‚áí 1
  (anyhow), success ‚áí 0 (`main.rs:28‚Äì31`, `349‚Äì366`). Matches.
- **Seal write ordering** (¬ß6, plan 2.5): canonicalize ‚Üí digest ‚Üí sign ‚Üí
  `put_template` ‚Üí `put_release` (write-once) ‚Üí `delete_draft` last
  (`release.rs:399‚Äì416`). The ordering invariant (a sign failure never burns the
  write-once key; a failed seal leaves the draft for retry) holds.
- **`crt-core` purity.** Signing/verifying take in-memory key bytes; Vault and
  the public-key fetch are `crt`-only edge shims (`vault.rs`, `verify.rs:186`).
  The whole seal/verify pipeline is unit-testable without Vault or network.
  Matches the design's two-crate split.
- **Visibility inert / internal never leaked.** `render_info` emits
  `public_summary` only and never `justification.internal`

  asserts a `DO-NOT-LEAK` marker is absent. The in-tree `MaterializationRecord`
  is M4, so the S3-only fields are correctly not yet projected anywhere public.
- **Config additions land with their first consumer** (plan): `namespaces`/
  `channels`/`branding` + `risk_components` in 2.4; `vault` in 2.5;
  `public_key_url` in 2.6. Verified against the per-commit diffs.
- **Documented deviations** (store-backed drafts; required `--base-ref`;

## Findings (by severity)

No critical or important findings. All four are low-severity / hardening.

### F1 ‚Äî Seal pre-check fires _after_ the Vault round-trip (claim not implemented as described)

`seal_release`'s comment (`release.rs:354‚Äì359`) states the cheap `get_release`
pre-check "refuses before any Vault round-trip ‚Ä¶ before the Vault round-trip,"
and the plan's 2.5 "Decisions, recorded as landed" repeats this. But the wiring
in `main.rs`'s `Seal` arm calls `vault::fetch_signing_key(&vault).await?`
(`main.rs:324`) **before** entering `seal_release` (`main.rs:325`), and the
pre-check is the first statement _inside_ `seal_release` (`release.rs:357`). So
re-sealing an already-sealed release still performs the full Vault read, then
bails. The functional impact is minor (a wasted Vault call on the rare
re-seal-of-sealed path; the `put_release` write-once guard is still
authoritative), but the code comment and the plan claim a behavior the code does
not exhibit ‚Äî precisely the "documented-but-not-implemented deviation" class.
**Fix:** either run a `get_release` existence pre-check in `main` before
`fetch_signing_key`, or correct the comment and the plan to say the pre-check
fires after the key fetch. Confidence: high.

### F2 ‚Äî A corrupt manifest / `ReleaseRecord` surfaces as exit 1, not a verify failure (undocumented)

The plan documents one narrow ¬ß11 leg-1 deviation: a referenced `PatchMeta` that
is _present but unreadable_ surfaces as an operational error (exit 1) rather
than a verify failure (exit 3) ‚Äî and the code matches (`verify.rs:127‚Äì136`). But
the same gap exists one level up and is _not_ documented: `verify_release` first
calls `store.get_release(&key)` (`verify.rs:84‚Äì87`), whose JSON deserialize
failure becomes `StoreError::Json` ‚Üí propagated as `Err` ‚Üí exit 1. A
`ReleaseRecord`/`Manifest` that is present but corrupt therefore never reaches
leg 1, where ¬ß11 frames a manifest "deserialize and validate" failure as a
verify failure (exit 3). This is a consistent extension of the documented
PatchMeta simplification, but it is undocumented and applies to the most
security-relevant object (the signed record itself). **Fix:** extend the plan's
recorded deviation to cover the record/manifest case, or (better, post-MVP) map
present-but-corrupt store objects to `VerifyFailed` so they exit 3. Confidence:
high.

### F3 ‚Äî `add` silently discards `--behavior-change` / `--upgrade-notes` when `--public-summary` is omitted

In `main.rs:282‚Äì287`, when `--public-summary` is absent the `else` branch
returns `release::compose_via_editor()?`, whose three-tuple shadows the
`behavior_change` / `upgrade_notes` bindings parsed from the flags. A user who
runs `crt release add ‚Ä¶ --behavior-change "X"` (without `--public-summary`)
loses `"X"`: the editor session's (possibly empty) values win, with no warning.
The flags and the editor are mutually exclusive by design, but the silent drop
is surprising. **Fix:** if `--public-summary` is omitted but
`--behavior-change`/`--upgrade-notes` are present, either error ("compose all
narrative fields in `$EDITOR`, or pass `--public-summary`") or seed the editor
template with the flag values. Confidence: high.

### F4 ‚Äî `load_public_key` accepts plaintext `http://`

`load_public_key` (`verify.rs:186‚Äì199`) branches on both `http://` and
`https://`, and `crt.config.yaml.example:22` documents an `http(s)` URL. The
public key is the root of trust for leg 0; fetching it over plaintext HTTP
invites a trivial MITM key substitution. Note: fingerprint pinning is explicitly
design-deferred (¬ß6.1 "pin its fingerprint via a trusted channel," ¬ß14 key
management), so the absence of pinning is **not** a phase miss and is not scored
as a security gap. The plaintext-`http` acceptance is the one in-scope hardening
item: prefer rejecting `http://` (or warning loudly) so the key is at least
transport-authenticated. Confidence: medium; non-blocking.

### Optional notes (no deduction)

- **Stale-draft-after-seal.** `seal_release` deletes the draft last
  (`release.rs:415`) and propagates a `delete_draft` failure as `Err`. If the
  `put_release` succeeds but `delete_draft` fails, the operator sees "seal
  failed" while the release actually landed, and `show_info` (which prefers a
  draft over a sealed release, `release.rs:300‚Äì316`) would then render the stale
  draft. Benign for the MVP (idempotent retry hits the write-once guard and the
  caller can re-`delete`), but worth a comment.
- **Leg-1 "validate" is shallow.** Leg 1 checks only `schema_version` equality;
  the structural "validate" ¬ß11 implies is whatever serde enforced during
  `get_release`. Acceptable while the schema is flat, but if future schema
  versions add cross-field invariants, leg 1 should grow an explicit check.
