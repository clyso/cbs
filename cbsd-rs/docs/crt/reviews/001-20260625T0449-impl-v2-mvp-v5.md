## Verdict

**GO (accept with findings; no blockers).** Both commits deliver exactly the M3
capability they claim: a pure-`crt-core` notes renderer pinned by the sealed
`RenderSpec` (branding snapshot + template-by-digest + exact-semver minijinja
gate), a hand-rolled deterministic CycloneDX 1.6 SBOM (digest-derived
`serialNumber`, `release.created` timestamp, fixed field order, `Vec` never a
map), and `release notes` / `release materialize --out` over a **sealed-only**,
**version-gated** read path. The hard invariants hold: `crt-core` stays IO-free
and tokio-free (minijinja resolves to `memo-map` + `serde` only), no
`.unwrap()`/`.expect()` appears outside `#[cfg(test)]` in the new code, the
information-leak guard holds, and the new `CrtCoreError` variants are
well-formed `thiserror`.

The determinism contract — load-bearing for M4 verify leg 4 — is **correct in
the code** but **under-tested**: the SBOM ships with neither a committed byte
golden (the plan named one) nor a renderer-version gate equivalent to the notes
side, so it is the one projection whose cross-build byte-stability nothing
currently pins. That is the lead finding (F1). None of the findings block M3 —
no signed bytes depend on the SBOM yet — but F1 is a cheap, high-leverage
should-fix-now that matches the crate's own existing `CANONICAL_GOLDEN` pattern.

## Verification of claimed work

Each implementer/plan claim was checked against the code, not the prose.

- **minijinja linked in `crt-core`, exact pin, single source of truth.**
  `crt-core/Cargo.toml` pins `minijinja = "=2.21.0"`;
  `RENDER_MINIJINJA_VERSION = "2.21.0"` (`notes.rs:26`) sits next to the pin;
  `seal_release` now stamps `crt_core::RENDER_MINIJINJA_VERSION`
  (`release.rs:384`), the provisional local `"2.5.0"` is deleted. `Cargo.lock`
  confirms `minijinja 2.21.0` with deps `memo-map` + `serde` only.
  **Confirmed.**
- **Notes render `public_summary`, never `justification.internal`.** The default
  asset and the inline `TEST_TEMPLATE` reference only `public_summary`;
  `never_renders_the_internal_note` (`notes.rs:160`) and
  `render_sealed_notes_renders_..._and_hides_internal` (`release.rs`) assert a
  `DO-NOT-LEAK` marker is absent from rendered output. **Confirmed** (see F4 for
  the structural caveat).
- **SBOM `serialNumber` digest-derived, `timestamp` = `release.created`, never
  random/wall-clock.** `serial_number` (`sbom.rs:64`) formats the first 32
  nibbles of `digest(manifest)` as a UUID URN; `metadata.timestamp` is
  `release.created.clone()` (`sbom.rs:36`).
  `serial_number_is_derived_from_the_digest_not_random` and
  `has_the_expected_cyclonedx_shape` cover both. **Confirmed.**
- **SBOM never emits the internal note.** `patch_for_entry` reads only
  `public_summary` + `refs`; `never_emits_the_internal_note` (`sbom.rs:312`)
  asserts it. Structurally safe — the internal field is never referenced.
  **Confirmed.**
- **`release notes` / `materialize` are sealed-only + version-gated.** Both
  route through `sealed_record_and_notes` (`release.rs:434`), which calls
  `store.get_release` (sealed envelope only — drafts use `get_draft`) then
  `crt_core::check_render_version` before rendering;
  `render_sealed_notes_refuses_a_minijinja_version_mismatch` proves the gate.
  **Confirmed.**
- **Emitted notes == `release notes` output.**
  `materialize_writes_both_artifacts_deterministically` asserts the written
  `RELEASE-NOTES.md` equals `render_sealed_notes(...)`. **Confirmed.**

## Design / plan fidelity

- **§7.2 three pinned inputs** (template-by-digest, minijinja version, branding
  snapshot) are all enforced on the render path (`release.rs:434–470`,
  `notes.rs`). Render uses `trim_blocks` + `lstrip_blocks`, deterministic
  regardless. **Matches.**
- **§7.1 SBOM shape** — one Ceph `component`, each entry under
  `pedigree.patches[]` as `{ type: "backport", diff, resolves }`. The wire
  structs (`sbom.rs:107–177`) match CycloneDX 1.6 key names
  (`bomFormat`/`specVersion`/`serialNumber`/`bom-ref`/`type`). **Matches.**
- **Determinism mechanics** — verified at the source: minijinja `groupby` sorts
  by the grouper via `safe_sort` → `slice::sort_by`, which is **stable**, so
  within-group order is the manifest entry order (deterministic); the comparator
  defaults to case-insensitive but, because the `unicode` feature is **off** in
  minijinja's defaults, it is `to_ascii_lowercase().cmp(...)` — pure std, no
  `unicase` transitive-version drift. SBOM iterates `entries.iter()` in sealed
  manifest order and uses `Vec` throughout. The notes/within-group/SBOM ordering
  is genuinely deterministic across runs and platforms. **Matches** (the one
  unpinned link is `serde_json`'s pretty-printer — F1/F6).
- **`crt-core` purity.** notes/sbom take in-memory inputs; the `crt` binary owns
  all store IO and the filesystem writes. minijinja pulls in no IO/tokio.
  **Matches.**
- **§13 / §8 deferrals.** Git ref/tag and the signed `000-RELEASE/` bundle, plus
  verify leg 4's byte-compare, are correctly deferred to M4 and the command help
  says so (`main.rs` `Materialize` doc, `materialize_artifacts` doc).
  **Matches.**

## Findings (by severity)

No critical findings. One important (test-contract) finding; the rest are
low-severity hardening / documentation.

### F1 — SBOM ships with no byte-golden and no renderer gate: the one projection whose cross-build determinism is unpinned

- **Severity:** Important · **Confidence:** 88 · **Commit:** `2dc6fa3` ·
  `crt-core/src/sbom.rs:255–270` (the test), `sbom.rs:54–58` (the gap).

The plan's commit-3.2 test list explicitly names a "**golden SBOM for a sample
manifest**" (plan line 118). It was **not** implemented:
`is_deterministic_and_pure` only asserts `build_sbom(m) == build_sbom(m)`
**within a single process**. That catches RNG/clock/map-iteration
nondeterminism, but it is strictly weaker than the leg-4 contract M3 exists to
establish — **byte-identical across builds and platforms**. The realistic threat
the SBOM faces (a `serde_json` pretty-printer change, which the code itself
flags at `sbom.rs:54–58`) is invisible to a rebuild-equality test, because both
rebuilds in one process link the identical `serde_json`. A committed golden, by
contrast, fails in CI the moment serialization drifts.

The asymmetry is the sharp point. The notes side has the **minijinja exact-pin +
`check_render_version` gate**: any cross-build renderer change is a hard error,
not a silent byte shift. The SBOM has **no equivalent `serde_json` gate** —
`serde_json` is an unpinned `^` transitive dep — **and** no golden. So the SBOM
is the single projection with neither a version gate nor a byte fixture guarding
the exact contract leg 4 will rely on. The plan's stated reason for dropping the
golden ("the SBOM is unsigned, unlike the manifest", plan line 146) is a
non-sequitur: leg 4 byte-compares regardless of whether the bytes are signed.
The crate already establishes the pattern — `manifest.rs` has `CANONICAL_GOLDEN`
pinning exact bytes — so the omission is an inconsistency, not just a gap.

**Does not block M3** (nothing signs or commits the SBOM bytes yet, no
production release is sealed). **Fix:** add a committed byte-golden SBOM for a
fixed sample manifest (mirroring `CANONICAL_GOLDEN`), asserting full-string
equality. Cheap, and it converts the determinism contract from "asserted in
prose" to "enforced in CI."

### F2 — `serde_json` pin should be a _hard_ M4 deliverable, not conditional on observing drift

- **Severity:** Low · **Confidence:** 80 · **Commit:** `2dc6fa3` ·
  `crt-core/src/sbom.rs:54–58`.

The deferrals the plan records (real CycloneDX validator; issue
`references`/`id` refinement; `serde_json` pin) are all **safe for M3** — in M3
nothing signs or commits the SBOM, so changing SBOM derivation in M4 invalidates
no signature. That tie-breaker is sound and worth stating. But one deferral is
mis-framed: the `build_sbom` comment and the plan defer pinning
`serde_json "if M4's leg 4 ever observes pretty-printer drift"`. By the time leg
4 _observes_ drift, the confusing failure has already shipped — leg 4 would
start failing against committed bytes that were faithful when written. **Fix:**
make "pin `serde_json` (exact)" a hard M4 deliverable that lands **with** leg 4,
alongside the byte-golden of F1, rather than a contingency. (The validator and
`references` refinements are genuinely fine to defer.)

### F3 — `materialize_artifacts` does blocking `std::fs` IO inside an `async fn`

- **Severity:** Low · **Confidence:** 82 · **Commit:** `2dc6fa3` ·
  `crt/src/release.rs:485–500`.

`materialize_artifacts` is `async` (it must `.await` the store) but performs
`std::fs::create_dir_all` and two `std::fs::write` calls directly, blocking the
tokio executor for the duration of the writes. The rust-2024 checklist flags
blocking IO in async context. Practical impact is negligible — this is a
single-shot CLI command on a multi-thread runtime writing two small files — so
this is low severity, but it is a real idiom violation worth recording while the
artifact set is still two files (M4 adds a whole `000-RELEASE/` bundle).
**Fix:** wrap the filesystem block in `tokio::task::spawn_blocking`, or use
`tokio::fs`. (`crt` already depends on tokio.)

### F4 — Notes-rendering safety is enforced by caller convention, not by construction

- **Severity:** Low · **Confidence:** 80 · **Commit:** `772f350` ·
  `crt-core/src/notes.rs:31–40`.

Contrast the two projections. The SBOM is **safe by construction**:
`patch_for_entry` never reads `justification.internal`, so no template/caller
mistake can leak it. `render_notes`, however, calls
`Value::from_serialize(manifest)` — exposing the **entire** manifest, including
`justification.internal`, to the template context. Both M3 invariants therefore
hold only by **convention**: (a) internal-hiding holds only because the
_default_ template happens not to reference `.internal` (guarded by an output
`contains` test, not by the renderer); (b) the version gate holds only because
the two current callers remember to call `check_render_version` before
`render_notes` — `render_notes` itself does not gate. This is acceptable for the
MVP (one renderer, one default template, two disciplined callers) and is
partially design-acknowledged (a custom/preview template is explicitly a
"preview, not canonical notes", §7.2). But it is a latent footgun: a future
caller that skips the gate, or a future custom template, silently breaks an
invariant the type system could enforce. **Fix (post-MVP, note only):** consider
rendering from a purpose-built public projection of the manifest (internal
stripped) rather than the whole manifest, so internal-hiding is structural like
the SBOM's; and/or fold `check_render_version` into the render entry point so
the gate cannot be bypassed.

### F5 — `materialize --out` defaults to `.` with fixed filenames: silent clobber across releases, and overwrite-into-non-empty-dir is untested

- **Severity:** Low · **Confidence:** 80 · **Commit:** `2dc6fa3` ·
  `crt/src/main.rs` (`Materialize { out, default_value = "." }`),
  `crt/src/release.rs:475–502`.

`materialize` writes fixed names `RELEASE-NOTES.md` and `sbom.cdx.json` into
`--out` (default the current directory). `crt release materialize A` followed by
`crt release materialize B` in the same cwd silently overwrites A's artifacts
with B's — no per-release namespacing, no "file exists" guard. M4 reorganizes
materialization around the git ref, so this is low severity and arguably
intended for M3's "just emit the files" scope, but it is a sharp edge for an
operator emitting several releases into one directory. Relatedly, the
determinism test re-materializes into a **fresh** `tempdir` (`dir2`), so
**in-place overwrite of a non-empty directory is never exercised** — the
overwrite path (`create_dir_all` on an existing dir + `write` over an existing
file) has no coverage. **Fix:** at minimum, note the clobber behavior in the
command help; consider namespacing the output under `--out/<release-name>/`; and
add a test that materializes twice into the _same_ dir and asserts identical
bytes (covering the overwrite path and the determinism contract together).

### F6 — Minor coverage gaps: empty-entries SBOM and the `references` form

- **Severity:** Low · **Confidence:** 80 · **Commit:** `2dc6fa3` ·
  `crt-core/src/sbom.rs`.

Two small omissions. (a) No test exercises an **empty-entries** manifest, where
`pedigree.patches` serializes to `[]` — a plausible edge (a release with only
known-issues) and a useful determinism/shape anchor. (b) The plan flags, and
this confirms, that issue `references` carries bare strings like
`"CVE-2026-0001"` (`is_deterministic_and_pure` fixture) whereas CycloneDX 1.6
`issue.references` expects `iri-reference` form (the bare CVE arguably belongs
in the issue `id` instead). This is correctly deferred to M4's validator pass
and is **not** a determinism problem, but it is a known spec-shape deviation
worth keeping visible. **Fix:** add an empty-entries SBOM test now; resolve the
`references`/`id` shape under M4's CycloneDX-validator deferral (F2).

## Positives worth recording

- **Purity invariant held under a new C-free dependency.** Linking a template
  engine into the no-IO core is exactly the kind of change that usually breaks
  purity; minijinja with default features stays pure (`memo-map` + `serde`), and
  the exact pin + adjacent constant make a version bump a deliberate two-line
  edit.
- **The determinism _mechanics_ are right.** Stable `groupby` sort, ASCII-only
  comparator (no `unicase` drift), digest-derived serial, `release.created`
  timestamp, fixed serde field order, `Vec` not map — every determinism lever
  the design asks for is pulled correctly. The gap is in _guarding_ it (F1), not
  in _achieving_ it.
- **Error handling is clean.** No `.unwrap()`/`.expect()` outside `#[cfg(test)]`
  in the new code; `?` propagation throughout; `with_context` on every IO edge
  in `crt`; new `CrtCoreError::{Render, RenderVersionMismatch, Sbom}` variants
  are well-formed and carry actionable detail (the mismatch names both
  versions).
- **The version gate is unbypassable on the two shipped paths** and emits a hard
  error, not a silent re-render — exactly the §7.2 "errors on mismatch"
  contract.
