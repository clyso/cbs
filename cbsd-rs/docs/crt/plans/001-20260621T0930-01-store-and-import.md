## Scope

Stand up the workspace and deliver patch ingestion: an operator can import a
patch (local git range or GitHub PR) and have it land **content-addressed** in
the store (local-fs for dev, S3 for prod) with visibility-neutral `PatchMeta`.

**Reflected decisions** (design §2/§5/§9): one `object_store`-backed `Store`
impl (`LocalFileSystem` / `InMemory` / `AmazonS3`); S3 endpoint/bucket in
`crt.config.yaml`, access/secret keys in `crt.secrets.yaml`; **both files
git-ignored, only `*.example` committed**; `--config` / `--secrets` paths
CLI-configurable (defaults, env-overridable), never hardcoded; **no minio
dependency** (tests use `InMemory`/`LocalFileSystem`; real-S3 is opt-in
standalone).

Out of M1: manifest/seal/sign (M2), SBOM/notes (M3), git materialization (M4).

## Commits

### 1.1 — `crt: import patches from a local git range into a content-addressed store`

**After this:** `crt patch import --repo R --range A..B` writes each patch
content-addressed (`blob_hash`) with `PatchMeta` into a **local-fs** store,
selected via a minimal `crt.config.yaml`.

- **Crates created** (flat workspace members, edition 2024, `GPL-3.0-or-later`):
  - `crt-core` — `Sha256` newtype, `CrtCoreError`, `PatchMeta`, `Provenance` /
    `UpstreamPrState`, `blob_hash(raw) = sha256(bytes)`, and a
    `git format-patch` parser (author / subject / body). No IO.
  - `crt-store` — `Store` trait (`put_blob`/`get_blob`/`has_blob`/`put_meta`/
    `get_meta`) + a single `ObjectStore`-backed impl; M1.1 constructs it over
    `object_store::LocalFileSystem`.
  - `crt` — `clap` CLI skeleton, global `--config`/`--secrets` (defaults + env),
    minimal config loader (`component`, local store path), and
    `patch import --repo --range` (runs `git format-patch` +
    `git patch-id --stable` as subprocesses → `PatchMeta` + blob → store).
- **`.gitignore`**: add `crt.config.yaml` + `crt.secrets.yaml`; commit
  `crt.config.yaml.example`.
- **Tests:** `crt-core` unit (blob_hash determinism, format-patch parse,
  patch_id capture); `crt-store` round-trip via `LocalFileSystem` + `InMemory`;
  `crt` integration (import from a fixture git range → assert blob+meta
  present).
- **Smell test:** one feature (import), spans three crates because the CLI can't
  compile without the core types + store and they aren't dead (import + tests
  exercise them); compiles + testable end-to-end; revertable. **~750 LOC**
  (justified-larger if it overruns — the first capability is irreducibly
  cross-crate; lockfile excluded).

### 1.2 — `crt: configure the store backend from config + a secrets file`

**After this:** point the store at **S3** — `crt.config.yaml` carries
`store.s3.{endpoint,region,bucket,prefix}`; `crt.secrets.yaml` carries
`s3.{access_key_id,secret_access_key}`; `import` writes to S3.

- `crt` — extend the config loader (S3 block) + add the secrets loader
  (`crt.secrets.yaml`, mode-0600 check, path from `--secrets`/env/default);
  construct `object_store::AmazonS3` from the merged config+secrets and select
  the backend. Commit `crt.secrets.yaml.example`; add `crt.secrets.yaml` to
  `.gitignore`.
- **Tests:** config+secrets parse/precedence; backend selection; store logic
  re-run against `object_store::InMemory` (no minio). Real-S3 wiring →
  `#[ignore]`d standalone test (env-gated, `CRT_TEST_S3_*`) against a test
  bucket; **never in `cargo test`**.
- **Smell test:** delivers a user-visible capability (production S3 target);
  caller = the existing `import` via the trait + tests. **~450 LOC.**

### 1.3 — `crt: import patches from a GitHub PR with equivalence detection`

**After this:** `crt patch import <pr-url>` imports from a PR (octocrab metadata
plus a local `git format-patch`); a `patch_id`-equivalent existing blob triggers
a **warning** so the operator reuses it rather than storing a near-duplicate.

- `crt` — octocrab PR lookup (metadata + head ref), fetch the ref locally,
  `git format-patch` for the canonical bytes (never GitHub's `.patch` endpoint —
  design §4), and the `patch_id` dedup-warning path.
- **Tests:** PR-URL parsing; equivalence-warning logic (unit, with two patches
  sharing a `patch_id`); integration behind a recorded/fixture PR where
  feasible.
- **Smell test:** capability increment on 1.1; **~500 LOC.**

## Verification (M1 end-to-end)

1. `cargo fmt --all`, `cargo clippy --workspace`, `cargo check --workspace` pass
   (per `cbsd-rs/CLAUDE.md`); `cargo test --workspace` green **without** minio
   or network S3.
2. With a local-fs `crt.config.yaml`:
   `crt patch import --repo <ceph-clone> --range <A>..<B>` → the blob +
   `patches/meta/sha256/<hash>.json` exist; re-import is idempotent.
3. With an S3 `crt.config.yaml` + `crt.secrets.yaml` against a **test bucket**
   (standalone, opt-in): the same import populates `patches/blobs|meta/...` in
   S3.
4. `crt patch import <ceph-pr-url>` imports and **warns** when re-importing an
   equivalent patch.

## Notes / risks

- **Build deps**: M1 pulls `object_store` (S3 feature), `clap`, `octocrab`/
  `reqwest`, `sha2`, a YAML serde crate. First `cargo` run must fetch these — if
  the environment is offline, surface it before proceeding rather than working
  around it.
- **YAML crate**: `serde_yaml` is deprecated — use `serde_yml` (or
  `serde_norway`); pin in 1.1.
- `serde_jcs` / `pgp` / `vaultrs` are **not** M1 deps (they arrive in M2).
- `patch_id` is computed via `git patch-id --stable`; import and (later) verify
  must use the same mode (design §4).
