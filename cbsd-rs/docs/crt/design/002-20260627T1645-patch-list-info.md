# crt — `patch list` and `patch info`

- **Status:** proposed
- **Date:** 2026-06-27
- **Component:** `crt` (CLI) + `crt-store`
- **Builds on:** `001-20260620T1318-v2-mvp.md` (§3–§5 store & patch model, §4
  patch identity)

## Context

`crt patch import` ingests patches into a content-addressed store, but there is
no way to see what has been imported. An operator who imported a batch yesterday
has to read the S3/filesystem layout by hand to find a `blob_hash`, and has no
way to inspect a single patch's recorded metadata. This design adds two
**read-only** introspection commands:

- `crt patch list` — enumerate the imported patches.
- `crt patch info <blob_hash>` — show one patch's full `PatchMeta`.

A third command, `crt patch delete`, was assessed alongside these but is **out
of scope here** (see [Non-goals](#non-goals)): it is the first _removal_
operation on an otherwise accumulate-only store and needs a reference-safety
scan, equivalence-index cleanup, and a design-corpus position of its own.

## Goals / Non-goals

### Goals

- List every imported patch with enough context to identify it (hash, subject).
- Inspect one patch's complete recorded metadata by `blob_hash` or a unique
  short prefix of it.
- Offer machine-readable `--json` output on both commands for scripting.
- Reuse the existing store reads and the `release list` / `release info` CLI
  pattern; add the minimum new store surface.
- Read-only: no mutation of blobs, metadata, the patch-id index, drafts, or
  releases.

### Non-goals

- `patch delete` (separate design + commit; see below).
- Filtering / search on `list` (possible follow-up).

## Existing machinery (reused)

Confirmed against the code:

- `Store::get_meta(&Sha256) -> PatchMeta` (`crt-store/src/lib.rs`) — already
  returns the full record `patch info` needs. **No new store method for info.**
- `PatchMeta` (`crt-core/src/meta.rs`): `blob_hash`, `patch_id`, `author`
  (`Identity`), `authored`, `subject`, `body`, `cherry_picked_from`,
  `provenance` (`Provenance::{UpstreamPr,Other}`), `source_repo`.
- `Sha256` (`crt-core/src/lib.rs`): `TryFrom<String>` (lowercase-hex, validates
  32 bytes) + `to_hex()`/`Display` — used to parse the `info` argument and the
  meta object key.
- `StoreError::is_not_found()` — the not-found branch, exactly as `release info`
  uses it.
- CLI pattern: `ReleaseCmd::{List,Info}` + their `main.rs` dispatch arms, backed
  by `release::list_releases` (returns data; `main` formats) and
  `release::show_info` (returns a formatted `String`).
- Store layout: patches live at `patches/meta/sha256/<hex>.json` and
  `patches/blobs/sha256/<hex>` (`crt-store/src/lib.rs`).

## New store surface

One new trait method, mirroring `list_releases`:

```rust
/// List the content addresses of all imported patches.
async fn list_patches(&self) -> Result<Vec<Sha256>>;
```

`ObjectBackedStore` implements it by listing the `patches/meta/sha256/` prefix
via `object_store::list` and parsing each object's `<hex>.json` location into a
`Sha256`. This needs a small new path parser — the existing `parse_release_key`
assumes a 4-part `prefix/ns/channel/name.json` layout and returns a
`ReleaseKey`, so it cannot be reused as-is; a `parse_meta_hash(location)` helper
strips the `patches/meta/sha256/` prefix and the `.json` suffix and calls
`Sha256::try_from`. Metadata is the canonical record of a patch (import always
writes blob then meta), so enumerating the meta prefix is the authoritative
list. It is the only new store method and backs both `patch list` and
`patch info`'s short-prefix resolution (below).

No delete/remove method is added (info and list are read-only).

## `crt patch list`

- **CLI:** `PatchCmd::List { json: bool }` (`--json`).
- **Backing fn:** `patch::list(store) -> Result<Vec<PatchMeta>>` — calls
  `store.list_patches()`, reads each patch's `get_meta`, and sorts
  deterministically by the key `(subject, blob_hash.to_hex())`. (`Sha256`
  derives no `Ord`, so the hash is ordered as its hex string; `PatchMeta`
  already carries `blob_hash`, so a separate tuple element is unnecessary.)
- **Cost note:** `object_store::list` yields only object locations, so enriching
  the listing with subjects is one `get_meta` read per patch (1 list + N gets) —
  acceptable for expected patch counts. A `get_meta` that fails (a corrupt or
  unreadable meta) aborts the whole listing with that error rather than silently
  skipping the patch — fail-loud, matching the rest of the tool.
- **Text output (stdout):** one line per patch, `<blob_hash>  <subject>` — full
  hex hash (matches `patch import`'s output, stays greppable). Summary to
  stderr: `N patch(es)` (mirrors `release list`). Empty store → no stdout,
  `0 patch(es)`.
- **`--json` output (stdout):** a single pretty-printed JSON array of the full
  `PatchMeta` objects (each already includes `blob_hash`) and nothing else — no
  stderr summary — so the result is a clean, parseable document.

```text
$ crt patch list
3f1c…(64 hex)  rgw: fix a crash in the frontend under load
9a02…(64 hex)  mds: tighten cap revocation
2 patch(es)
```

## `crt patch info`

- **CLI:** `PatchCmd::Info { blob_hash: String, json: bool }` — `blob_hash` is a
  full 64-char hex address **or a unique short prefix** of one; `--json` selects
  machine output.
- **Resolution (`patch::find(store, &arg) -> Result<PatchMeta>`):** validate the
  arg **first** with an explicit lowercase-hex predicate (`0-9a-f` only — note
  `char::is_ascii_hexdigit` would wrongly accept uppercase, and
  `Sha256::try_from` validates only the full 64-char form), then branch on
  length. The valid length range is 4–64: `<4` is rejected as too short and
  `>64` as too long / not a hash.
  - **Full (64 hex):** `Sha256::try_from` + `store.get_meta` directly — O(1), no
    enumeration. Not-found → `no patch with blob hash <hash>`.
  - **Prefix (4–63 hex):** enumerate via `store.list_patches()` and match by hex
    prefix — zero matches → not-found error; exactly one → use it; more than one
    → ambiguous error listing the matching hashes.
- **Text output (stdout):** a labeled block over the full `PatchMeta`:

```text
blob_hash:    <64-hex>
patch_id:     <git patch-id --stable>
subject:      <subject>
author:       <name> <<email>>
authored:     <iso-8601>
source_repo:  <repo>
provenance:   upstream-pr [merged-stable]
                prs:     <url>[, …]
                commits: <sha>[, …]
              # or — other: <description>
cherry-picked-from: <sha>[, …]   # omitted when empty

<body>
```

- **Equivalence cross-reference (nice-to-have):** if
  `store.get_patch_id(meta.patch_id)` resolves to a _different_ `blob_hash`,
  append a line noting that an equivalent patch is the indexed blob for this
  `patch_id` (design §4). Read-only; informational. The index records only the
  first-seen blob per `patch_id`, so this is **one-way**: inspecting that
  representative blob shows no line (there is nothing more canonical to point
  at) — matching how `import` reports equivalence.
- **`--json` output (stdout):** the resolved `PatchMeta` as a single
  pretty-printed JSON object — faithful to the stored record (the equivalence
  note is a text-mode-only annotation, not a synthetic JSON field).

## Module structure

A new `crt/src/patch.rs` module (`mod patch;` in `main.rs`), mirroring
`release.rs`: data-returning `pub async fn list(...) -> Vec<PatchMeta>` and
`pub async fn find(...) -> PatchMeta`, plus text renderers (`render_list`,
`render_info`). The `PatchCmd::{List, Info}` dispatch arms open the store once
(`open_store`), then either serialize the data with
`serde_json::to_string_pretty` (`--json`) or print the text renderer's output —
keeping data and presentation separate so both output modes share one path.

## Testing

Against `ObjectBackedStore::in_memory()` (the existing test backend), seeded by
importing real patches or writing `put_blob` + `put_meta` directly:

- **store** `list_patches`: write two patches → returns both hashes; empty store
  → empty vec.
- **`patch::list`**: returns patches sorted by `(subject, blob_hash.to_hex())`;
  empty store → empty. `--json` round-trips — the output parses back to the same
  `Vec<PatchMeta>`.
- **`patch::find`**: a full hash resolves; a unique short prefix resolves to the
  same patch; an ambiguous prefix → error; a no-match prefix → error; a
  too-short (<4), too-long (>64), uppercase, or non-hex arg → error.
- **info `--json`**: the output parses back to the resolved `PatchMeta`.
- The equivalence cross-reference line appears (text mode) when two blobs share
  a `patch_id`.

## Implementation / commit plan

Two `crt:` commits, split by capability and ordered `list` → `info`. Each
delivers a distinct operator capability (enumerate the store vs. inspect one
patch) and touches different components: `list` carries the only `crt-store`
change (`list_patches`), while `info` is `crt`-side, consuming the existing
`get_meta` plus the now-landed `list_patches` for prefix resolution. The new
store method ships in commit 1 with a caller, and commit 2 adds a second caller
— no dead-code intermediate either way; each passes the smell test on its own.

1. **`crt: list imported patches`** — the `list_patches` store method +
   `parse_meta_hash` helper and its store test (`crt-store`); the new
   `crt/src/patch.rs` module (`mod patch;` + `patch::list` + `render_list`);
   `PatchCmd::List { json }` + dispatch (`main.rs`); the list / empty / sorting
   / `--json` tests. (~150 LOC)
2. **`crt: inspect an imported patch`** — `patch::find` (full hash + unique
   short-prefix resolution, reusing `list_patches`) + `render_info`;
   `PatchCmd::Info { blob_hash, json }` + dispatch (`main.rs`); the full-hash /
   short-prefix / ambiguous / no-match / malformed-hex / `--json` / equivalence
   tests. No new `crt-store` change. (~130 LOC)

`list` lands first because `info`'s short-prefix resolution reuses
`list_patches`; commit 2 then adds no `crt-store` change, keeping the
library-layer change isolated in commit 1. `info` is split out rather than
bundled because it is a distinct capability that passes the smell test alone —
not a trivial move/rename.

## Invariants (do not regress)

- **Read-only.** No new mutation path; the patch-id index, blobs, meta, drafts,
  and releases are untouched.
- `crt-core` stays pure (no IO); the new logic is the `crt-store` enumeration +
  the `crt` CLI module only.
- No new third-party dependencies (`object_store::list`, `serde_json`, and
  `Sha256` already in tree).
- Output goes to stdout; summaries/diagnostics to stderr — matching the existing
  commands so the tool stays scriptable.
