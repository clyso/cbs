# CRT v2 — MVP Design (store, manifest, materialization, CLI)

> **Status:** Design. Elaborates [`../000-concept.md`](../000-concept.md) into a
> buildable specification for the MVP (milestones M1–M4). The concept fixes the
> _why_ and the locked decisions; this document fixes the _what_ and _how_:
> concrete types, the on-S3 schema, content-addressing, sealing + signing,
> projection, git materialization, the CLI, config, and verification. Rationale
> already in the concept is referenced, not repeated.

## 1. Scope

In scope (MVP): the `crt` CLI and the `crt-core` / `crt-store` crates delivering
M1–M4 — import patches → compose a **sealed and GPG-signed** release → project a
**deterministic** SBOM + release notes → materialize a **signed,
self-verifying** linear git ref. Deferred (concept §11): the GitHub-Issues
workflow, build-system wiring, `crt-server` + web UI, **cosign/sigstore
signing** (§6.2), and migration from the Python `crt`. (GPG manifest-signing is
now **in** scope; only the sigstore variant is deferred.)

**Decisions settled here** (closing concept §13 + the signing follow-ups):

| Question                 | Resolution                                                                                                                                                                                                                                                                      |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CLI + crate name         | `crt` (binary); crates `crt`, `crt-core`, `crt-store`.                                                                                                                                                                                                                          |
| License                  | **GPL-3.0-or-later** for all crt crates (incl. future `crt-server`).                                                                                                                                                                                                            |
| Commit prefix            | `crt:` (e.g. `crt: add content-addressed patch store`).                                                                                                                                                                                                                         |
| Integrity + authenticity | RFC 8785 canonical JSON + `sha256` **digest** (integrity/identity) **and** a detached **OpenPGP signature** over the canonical manifest (authenticity). No/invalid signature ⇒ invalid release.                                                                                 |
| Signing library          | **`pgp` (rPGP) 0.19.0** — pure-Rust OpenPGP (RFC 9580), detached sigs, no C deps. Chosen over `sequoia-openpgp` 2.3.0 (default Nettle C backend; its pure-Rust backend is "not recommended").                                                                                   |
| Signing keys             | Private key fetched at sign time from **HashiCorp Vault** via **`vaultrs` 0.8.0** (KV v2), never persisted by `crt`; the Vault addr/token + the signing-key vault path live in `crt.secrets.yaml` (§9). Public key published at a **known Clyso-owned location** for consumers. |
| Patch visibility         | Recorded per-patch (`public`/`private`) but **inert in the MVP** — no effect on SBOM, notes, verify, or materialization. The flag (and `justification.internal`) live **only in the S3 manifest**, never in the in-tree record. Enforcement deferred to the service gateway.    |
| Canonical JSON           | RFC 8785 (JCS) via a named crate (`serde_jcs`; see §14 re: maintenance). Sorted-key `serde_json` is **not** equivalent and is not used.                                                                                                                                         |
| S3 credentials           | S3 endpoint/bucket in `crt.config.yaml`; access/secret keys in `crt.secrets.yaml`. **Both git-ignored** (only `*.example` committed); paths CLI-configurable (§9). No minio in tests (`InMemory`/`LocalFileSystem`; real-S3 opt-in standalone).                                 |
| Hard WORM                | Tamper-evidence by signature; S3 Object Lock optional, off by default.                                                                                                                                                                                                          |
| Risk `component`         | A **configurable list** in `crt.config.yaml`, not a hardcoded enum. Axes/weights/bands from relenor as-is.                                                                                                                                                                      |

| Git ops                  | Subprocess (`git`), as `cbsd-worker` does; `gix`/`git2` later.                                                                                                                                                                                                                  |

### The two trust anchors

1. **Content-addressing** — patches keyed by `sha256(raw blob)`; the manifest
   references them by hash, so it transitively commits to every patch's bytes.
2. **Signatures** — a detached OpenPGP signature over the canonical manifest (in
   S3) and over the in-tree materialization record (in the git ref). The
   signature is the authenticity anchor and, because it ships _beside_ the file
   it signs, it verifies a **detached** tree (tarball / ZIP download) that a git
   tag cannot reach (§8).

## 2. Crates & dependencies

Flat workspace members under `cbsd-rs/`, edition 2024, resolver 2, license
`GPL-3.0-or-later`.

| Crate       | Kind                 | Key deps                                                                                                                                 |
| ----------- | -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `crt-core`  | lib, **no IO/tokio** | `serde`, `serde_json`, `serde_jcs`, `sha2`, `pgp` (sign/verify over provided key bytes — pure), `minijinja`, `thiserror`                 |
| `crt-store` | lib (IO)             | `crt-core`, `object_store` (S3 + local-fs), `serde_json`, `thiserror`, `tokio`                                                           |
| `crt`       | bin (CLI)            | `crt-core`, `crt-store`, `clap`, `anyhow`, `tokio`, `vaultrs` (private-key fetch), `octocrab`/`reqwest`, `tracing`, `tracing-subscriber` |

- `crt-core` stays runtime-agnostic: `pgp` signing/verification operates on
  in-memory key bytes (pure compute, no IO); `minijinja` renders from in-memory
  strings.
- `crt-store` owns S3/fs IO; `tokio` only because `object_store` is async.
- `crt` owns `tokio`, `clap`, tracing, the `git` subprocess driver, and the
  Vault fetch (`vaultrs`); it hands key bytes to `crt-core` to sign/verify.

## 3. Domain model (`crt-core`)

Two record kinds plus the on-disk artifacts:

- **`PatchMeta`** — patch-**intrinsic**, visibility-neutral facts. Stored once
  per blob (§4).
- **`ManifestEntry`** — **per-release** fields; self-contained (denormalizes
  rendering-relevant provenance at seal time). `visibility` and
  `justification.internal` live **only here / in S3** — they are never copied to
  the in-tree materialization record (§8).

```rust
// ---- patch-intrinsic (crt-store: patches/meta/sha256/<blob_hash>.json) ----
struct PatchMeta {
    blob_hash: Sha256,            // artifact address (§4); sha256(raw blob)
    patch_id: String,             // `git patch-id --stable`; equivalence + verify (§11)
    author: Identity, authored: DateTime,
    subject: String, body: String,
    cherry_picked_from: Vec<String>,
    provenance: Provenance, source_repo: String,
}
enum Provenance {
    UpstreamPr { prs: Vec<Url>, commits: Vec<Sha1>, state: UpstreamPrState },
    Other { description: String },        // downstream-only
}
enum UpstreamPrState { MergedStable, MergedMain, ApprovedOpen, OpenInReview }

// ---- per-release (inside the sealed manifest) ----
struct ManifestEntry {
    blob_hash: Sha256,            // artifact address: ref into the blob store
    patch_id: String,             // offset-invariant identity; verify (§11)
    order: u32,
    visibility: Visibility,       // recorded; inert (§7); S3-only, never in-tree
    category: Category,           // configurable list (§9); notes grouping
    risk: Risk,                   // blast/conflict/coverage (+ derived total/band)
    justification: Justification, // public_summary rendered; `internal` S3-only
    behavior_change: Option<String>, upgrade_notes: Option<String>,
    lifecycle: Lifecycle, data_structure_change: Option<DataStructureChange>,
    provenance: Provenance,       // denormalized snapshot
}
struct Justification { kind: JustificationKind, refs: Vec<String>,
                       public_summary: String, internal: Option<String> }

// ---- release (canonical, in S3) ----
struct Manifest {                 // the body that gets sealed + signed
    schema_version: u32,
    release: ReleaseHeader,       // product, ns, channel, name, base ref, created, author
    entries: Vec<ManifestEntry>,
    known_issues: Vec<KnownIssue>, upgrade_notes: Option<String>,
    branding: Branding,           // SNAPSHOT of resolved channel branding at seal (§7.2)
    render: RenderSpec,           // pins notes rendering so verify reproduces (§7.2/§11)
}
struct RenderSpec { minijinja_version: String, template_digest: Sha256 } // bytes at templates/sha256/<digest> (§5)
struct ReleaseRecord {            // stored at releases/<ns>/<channel>/<name>.json
    manifest: Manifest,
    digest: Sha256,               // sha256(canonical_json(manifest)) — see §6
    signature: ArmoredSignature,  // detached OpenPGP over canonical_json(manifest) — see §6
}

// ---- in-tree, public-safe (000-RELEASE/record.json + record.json.asc) ----
struct MaterializationRecord {
    s3_manifest_digest: Sha256,   // back-ref to the sealed release (opaque to outside consumers)
    base_ref: String,
    render: RenderSpec,
    source_tree_digest: Sha256,   // hash over materialized source files, EXCL 000-RELEASE/ (§8)
    bundle_digests: BTreeMap<String, Sha256>, // sha256 of EVERY other 000-RELEASE/ file
                                  //   (sbom, notes, provenance, README, .gitattributes) —
                                  //   so the signed record covers the whole bundle, by construction
    patches: Vec<MaterializedPatch>,  // { order, blob_hash, patch_id, git_commit }
    // NOTE: no `visibility`, no `justification.internal` — public-safe by construction
}
```

Risk `total`/`band` are computed by `crt-core`, not stored, to avoid drift.

## 4. Content addressing

Two hashes, each one job (kept separate is load-bearing):

- **`blob_hash = sha256(raw blob)`** — a plain artifact content-address over the
  exact stored `git format-patch` bytes. Byte-exact; it does not survive rebase.
- **`patch_id` (`git patch-id --stable`)** — offset/whitespace-invariant logical
  identity; survives rebase/`git am` offset/re-export. Anchors (a) dedup /
  equivalence at import and (b) materialized-git anchoring at verify (§11).
  **Invariant**: import (stores it) and verify (recomputes it) use the same
  `--stable` mode.

Store keys: `patches/blobs/sha256/<blob_hash>` (raw patch text) and
`patches/meta/sha256/<blob_hash>.json` (`PatchMeta`).

**Canonical byte source.** Import produces the blob from a **local checkout**
via `git format-patch` with fixed options (no signature, fixed numbering) —
**never** GitHub's `.patch`/`.diff` endpoint, whose bytes differ. `octocrab`
supplies PR _metadata_ only. (The Python predecessor computes
`patch_id --stable` for the git-range import path — `git_utils.py:754`, stored
in `patch.py` — but stubs it in the PR-import path; we compute it uniformly. So
this is partly new code.)

**Reuse across releases** works at both levels: one stored blob referenced by
many manifests via `blob_hash` (exact reuse); on import a matching `patch_id`
**warns** that an equivalent change already exists, so the operator references
it rather than storing a near-duplicate. `ManifestEntry` carries both hashes.

## 5. Store layer (`crt-store`)

A trait abstracts the backend; the seam the future service swaps. Two impls:
`object_store`-backed S3 and a local filesystem backend for tests.

```rust
#[async_trait]
trait Store {
    async fn put_blob(&self, hash: &Sha256, raw: &[u8]) -> Result<()>;   // idempotent
    async fn get_blob(&self, hash: &Sha256) -> Result<Bytes>;
    async fn has_blob(&self, hash: &Sha256) -> Result<bool>;
    async fn put_meta(&self, hash: &Sha256, meta: &PatchMeta) -> Result<()>;
    async fn get_meta(&self, hash: &Sha256) -> Result<PatchMeta>;
    async fn put_template(&self, digest: &Sha256, bytes: &[u8]) -> Result<()>;
    async fn get_template(&self, digest: &Sha256) -> Result<Bytes>;
    async fn put_release(&self, key: &ReleaseKey, rec: &ReleaseRecord) -> Result<()>;
    async fn get_release(&self, key: &ReleaseKey) -> Result<ReleaseRecord>;
    async fn list_releases(&self) -> Result<Vec<ReleaseKey>>;            // from index/releases.json
}
struct ReleaseKey { namespace: String, channel: String, name: String }
```

- **Blobs / meta / templates** are content-addressed ⇒ idempotent writes.
- **`ReleaseRecord`** carries the `digest` + detached `signature` (§6); a
  consumer recomputes the digest and verifies the signature. `put_release` of an
  existing key is refused by default (guard); hard WORM is optional S3 Object
  Lock on `releases/`.
- **Single `object_store`-backed impl.** One `Store` over `dyn ObjectStore`; the
  backend is whichever `ObjectStore` is constructed — `LocalFileSystem` (dev),
  `InMemory` (tests), or `AmazonS3` (prod). No second hand-rolled backend.
- **S3 connection**: endpoint/region/bucket/prefix from `crt.config.yaml`;
  access/secret keys from the git-ignored `crt.secrets.yaml` (§9), built into
  `object_store::AmazonS3`. **No minio dependency** — `cargo test` exercises the
  store logic against `InMemory`/`LocalFileSystem`; real-S3 is an opt-in,
  `#[ignore]`d standalone test against a test bucket, never in `cargo test`.
- **Keys are not stored here**: the private signing key comes from Vault at sign
  time (§6.1); the public key is published at a known Clyso-owned location.
- `index/releases.json` is a re-derivable cache for `list`/`info`.

## 6. Sealing, canonicalization & signing

- **Canonical JSON**: RFC 8785 (JCS) via `serde_jcs` — deterministic key order +
  number/string canonicalization. The same `Manifest` always serializes
  identically. (Sorted-key `serde_json` is not RFC 8785 and is not used.)
- **Digest**: `digest = sha256(canonical_json(manifest))`, stored in the
  `ReleaseRecord` envelope **around** the manifest — never inside the hashed
  body (digest-excludes-itself).
- **Signature**: `seal` fetches Clyso's OpenPGP private key from Vault (§6.1)
  and produces a **detached** signature over `canonical_json(manifest)` with
  `pgp` (rPGP), stored in the `ReleaseRecord`. The signature is the
  **authenticity** anchor (the digest is integrity/identity); **no signature or
  an invalid signature ⇒ invalid release**. The signature is necessarily
  _outside_ the hashed bytes — detached, no self-reference.
- **Seal precedes materialize**: the sealed manifest keys patches by
  `blob_hash`; commit SHAs do not exist yet (the git↔hash binding is the
  post-seal `Crt-Patch` trailers and the in-tree record, §8).

### 6.1 Signing keys & Vault

The private OpenPGP key lives in **HashiCorp Vault** (KV v2), fetched at sign
time via `vaultrs` 0.8.0 and never persisted by `crt` — mirroring how cbscore
sources secrets from Vault (`SecretsMgr.gpg_signing_key(id)`), but in Rust. The
Vault address + token (or AppRole) **and the vault path of the signing key**
live in the git-ignored `crt.secrets.yaml` (§9), so everything needed to fetch
the key sits in one secret-bearing file. The corresponding **public key** is
published at a known, Clyso-owned location (a URL in `crt.config.yaml`);
consumers fetch it once, pin its fingerprint via a trusted channel, and verify
releases against it. Key rotation / revocation / distribution is an operational
open item (§14).

### 6.2 Future: cosign / sigstore (deferred)

The signing seam is _detached signature over canonical bytes_, which is
scheme-agnostic — so GPG can later be swapped or complemented by **sigstore**
(`sigstore` Rust crate). Keyless sigstore signs with a short-lived **Fulcio**
certificate bound to an **OIDC** workload identity and records the signature in
the **Rekor** transparency log; verification checks the cert chain + log
inclusion instead of a long-lived public key. Benefits: no long-lived private
key to custody, third-party-auditable transparency log, revocation via short
cert lifetimes. Costs: online Fulcio/Rekor + an OIDC identity provider, more
moving parts. Deferred to its own design doc; only the signing/verification
adapter changes.

## 7. Materialization & projection (`crt-core`)

**One tier in the MVP.** The per-patch `visibility` flag is recorded but
**inert**: nothing filters or redacts on it. Rationale: the full patched git
branch is itself a deliverable and contains every patch, so redacting derived
artifacts while shipping the whole tree would be theater. Visibility enforcement
(and a genuinely reduced public artifact) is deferred to the service gateway.
Projection is the identity over the sealed manifest; it remains the single seam
the gateway later extends.

### 7.1 SBOM — CycloneDX JSON (deterministic)

One CycloneDX document: the Ceph component is one `component`; each patch is an
entry under `pedigree.patches[]`
(`{ type: "backport", diff: {...}, resolves: [...] }`). It is emitted
**deterministically** so it is a pure function of the manifest (required for the
faithfulness check, §11): `metadata.timestamp` is omitted or derived from
`release.created`, and `serialNumber` is omitted or derived from the manifest
digest — never a random UUID or wall-clock time. Granularity confirmed against a
CycloneDX validator in M3 (§14). Emitted as `sbom.cdx.json`.

### 7.2 Release notes — minijinja (deterministic)

- Rendered in `crt-core` (no IO). Three inputs are pinned/sealed so rendering is
  reproducible: (a) the **template**, recorded by digest in `RenderSpec` with
  its **bytes stored content-addressed** (§5) and fetched by digest at
  render/verify; (b) the **minijinja version** in `RenderSpec`, which `verify`
  compares against its own linked version and **errors on mismatch** (distinct
  exit code) rather than silently re-rendering; (c) the **branding snapshot**
  sealed into the manifest (`Manifest.branding`) — notes render from that
  snapshot, **not** from live `crt.config.yaml`, so a later config edit cannot
  change a sealed release's notes.
- The default template **ports with adaptation** from relenor's
  `RELEASE-NOTES.md.j2` (grouped by `category`; per-patch sections;
  active-patches table; known issues; coordination follow-ups). Not literally
  verbatim — relenor registers a custom `groupby` _global_ and uses
  `selectattr`/`rejectattr`/`sort(attribute=)`, whereas minijinja exposes
  `groupby` as a _filter_ (case-insensitive by default — sort stability pinned);
  these are re-expressed and validated against minijinja in M3 (§14).
- Notes render `public_summary` (never `justification.internal`). Emitted as
  `RELEASE-NOTES.md`. `crt release notes` re-renders from the sealed manifest +
  pinned `RenderSpec`; rendering with a _different_ template is a **preview**,
  not the canonical notes.

## 8. Git materialization & the signed in-tree record

Driven by a sealed release, produce **one** `release/<name>` branch + annotated
tag in the configured destination repo. Subprocess `git`.

1. Clean checkout/worktree at the base ref.
2. For each entry in `order`, `git am` the blob, then amend to append a trailer
   `Crt-Patch: sha256:<blob_hash>`. **No `Crt-Visibility` trailer** — visibility
   is S3-only (so the branch leaks no classification). A patch that fails to
   apply aborts the run (`git am --abort`) and **fails loud**.
3. Append the final **`000-RELEASE/`** commit, the portable verification bundle:
   - `record.json` — the `MaterializationRecord` (public-safe BOM, §3): base
     ref, `source_tree_digest`, `bundle_digests` (a sha256 of **every other**
     `000-RELEASE/` file — so nothing in the bundle is unsigned, by
     construction), the patch BOM (`blob_hash`/`patch_id`/`git_commit`),
     `RenderSpec`, and the `s3_manifest_digest` back-ref.
   - `record.json.asc` — the **detached OpenPGP signature over `record.json`**
     (the portable root of trust; same key as the sealed manifest, §6.1).
   - `sbom.cdx.json`, `RELEASE-NOTES.md`, `provenance.json` (public fields),
     `README.md` (what this is + how to verify).
   - **No `.gitattributes export-ignore`** on `000-RELEASE/`: the bundle must
     **travel with** the tree so a tarball/ZIP recipient can verify offline. It
     is inert for builds (an unreferenced top-level dir); the deferred build
     path (concept §11) excludes it from the Ceph build tarball.
4. Create an **annotated tag** carrying the manifest `digest` (a future _signed_
   tag covers the git graph cryptographically); push branch + tag.

**`source_tree_digest`** is a deterministic hash over the materialized source —
`sha256` of a canonical serialization of sorted `(path, sha256(content))` pairs
for every file **except** under `000-RELEASE/`. Exact rules pinned in M4 (§14).

**Integrity — three legs:**

- **Portable / detached-tree (primary).** From a tarball, ZIP, or UI download —
  no git, no store — a recipient: (1) fetches Clyso's public key from the known
  location and verifies `record.json.asc` over `record.json` ⇒ record authentic;
  (2) recomputes `source_tree_digest` and **every** `bundle_digests` entry over
  the extracted files and compares ⇒ all of the source and the whole bundle are
  authentic. This is what the git tag cannot do (a tag is lost the moment the
  tree leaves git).
- **Git-native (defense-in-depth).** The protected annotated tag covers the tip
  tree via git's Merkle hashing and cannot be force-moved; a future signed tag
  adds cryptographic authenticity for clones.
- **Internal faithfulness (Clyso, with S3).** `verify` confirms the in-tree
  record is a faithful projection of the sealed manifest and re-derives the SBOM
  / notes from it (§11).

## 9. Configuration & secrets

Two YAML files, **both git-ignored** — only their `*.example` versions are
committed; the config is **not** shareable. Both default to a known location and
are overridable via CLI (`--config` / `--secrets`) or env, never hardcoded.

`crt.config.yaml` — non-secret:

```yaml
component: ceph
destination_repo: clyso/ceph
store:
  s3: { endpoint: <url>, region: <region>, bucket: <bucket>, prefix: crt/ }
  object_lock: false
risk_components: [rgw, rgw-multisite, dashboard, build, docs, other]
public_key_url: https://<clyso-known-location>/crt-pubkey.asc
secrets_file: /etc/crt/secrets.yaml # default; overridable by --secrets / $CRT_SECRETS
namespaces:
  clyso-enterprise:
    channels:
      ces: # prefix-based resolution (ces-v18.2.0 → here)
        upstream: { repo: ceph/ceph }
        branding:
          {
            display_name: "Clyso Enterprise Storage",
            blurb: "...",
            footer: "...",
          }
# issues_repo: clyso/ceph   # post-MVP GitHub-Issues workflow (concept §8)
```

`crt.secrets.yaml` — secrets only (git-ignored, mode 0600):

```yaml
s3: { access_key_id: <id>, secret_access_key: <key> }
vault:
  addr: <vault-addr>
  token: <token> # or AppRole role_id / secret_id
  keys: { gpg_signing_private: secret/data/crt/openpgp-signing-key }
```

Endpoint/bucket are non-secret config; only the access/secret keys and the Vault
token are secret (matching cbscore's split — `S3StorageConfig.url` in config,
creds via `SecretsMgr`). The `vault.keys` map names the vault path(s) where the
GPG signing key(s) live. Namespace/channel resolution is prefix-based on the
release name; the `branding` block is snapshotted into the sealed manifest at
seal time (§7.2).

## 10. CLI surface (`crt`)

`clap` derive; `anyhow` at the top; non-zero exit on any failure. Global options
`--config <path>` / `--secrets <path>` (each with a default location,
env-overridable) select the config and secrets files (§9); their paths are never
hardcoded.

| Command                                            | Behavior                                                                                                                                                                                               |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `crt patch import <pr-url \| --repo --range A..B>` | PR metadata via `octocrab`; patch bytes from a local `git format-patch` (§4); write blob + `PatchMeta`. Warns on `patch_id` equivalence.                                                               |
| `crt release new <name>`                           | Resolve `name` → namespace/channel; create a draft release record.                                                                                                                                     |
| `crt release add <name> <blob_hash…>`              | Add entries; flags `--category --visibility` (recorded, inert) `--justification --ref`; `$EDITOR` for `public_summary`/`behavior_change`/`upgrade_notes`.                                              |
| `crt release seal <name>`                          | Compute risk bands; snapshot branding + record `RenderSpec` (store the template); canonicalize + digest; **fetch the private key from Vault and sign**; write the `ReleaseRecord`.                     |
| `crt release materialize <name>`                   | Emit `sbom.cdx.json` + `RELEASE-NOTES.md`; materialize the linear git ref/tag with the signed `000-RELEASE/` bundle (§8).                                                                              |
| `crt release notes <name>`                         | Re-render notes from the sealed manifest (pinned `RenderSpec`); no re-seal.                                                                                                                            |
| `crt release list \| info <name> \| verify <name>` | Query; `verify` runs the layered audit (§11), including signature checks.                                                                                                                              |
| `crt verify --tree <dir>`                          | **Offline / detached-tree** verification of an extracted tarball or clone: verify `record.json.asc` with the public key, then recompute `source_tree_digest` + all `bundle_digests`. No store, no git. |

Draft releases live locally until `seal` writes them to the store. Authoring
uses flags + `$EDITOR`. `crt release supersede` is post-MVP.

## 11. Verification & error model

`crt release verify <name>` (full, store-backed):

0. **Signature** — fetch Clyso's public key (known location); verify the
   `ReleaseRecord.signature` over the canonical manifest, and (when a ref
   exists) the `000-RELEASE/record.json.asc` over `record.json`. No/invalid ⇒
   fail fast.
1. **Schema** — `ReleaseRecord`/`Manifest`/`MaterializationRecord` and each
   referenced `PatchMeta` deserialize and validate.
2. **Cross-reference** — recomputed digest == stored digest; entry `blob_hash`
   set == referenced blobs; every referenced blob exists; the in-tree record's
   `s3_manifest_digest` == the sealed digest, and its patch BOM is a faithful
   projection of the sealed entries.
3. **Materialized-git anchoring** (ref exists) — walk the linear history; each
   commit's `Crt-Patch` trailer names its entry's `blob_hash`, and the commit's
   recomputed `git patch-id --stable` equals that entry's `patch_id`
   (offset-invariant — robust to `git am` applying with **offset**; `git am`
   does not fuzz by default). Re-materialize and compare the **pre-`000-RELEASE`
   source tree** (deterministic).
4. **Artifact faithfulness** (ref exists) — re-derive `sbom.cdx.json` (a pure
   function of the manifest) and re-render `RELEASE-NOTES.md` (sealed
   `RenderSpec`
   - branding snapshot + version-gated minijinja, template fetched by digest
     from §5), then byte-compare to the committed copies. **This leg is new
     work** — it is _not_ from relenor (relenor's `verify_release.py` has three
     layers and no re-render/SBOM); the protected tag (§8) already delivers
     post-publish tamper-evidence, so this leg only adds materialize-time
     faithfulness.

Note: **external consumers do not run leg 4** — they verify by signature +
