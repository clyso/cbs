# 017 ‚Äî Robot Accounts: Implementation Plan

**Design:** `docs/cbsd-rs/design/017-20260417T1130-robot-accounts.md` (v3,
approved for implementation)

**Reviews:**

- `docs/cbsd-rs/reviews/017-20260417T1841-design-robot-accounts-v1.md`
- `docs/cbsd-rs/reviews/017-20260419T0905-design-robot-accounts-v2.md`

## Scope

Admin-managed service identities (robot accounts) that authenticate via
long-lived `cbrk_` bearer tokens. All robot permissions flow through role
assignments; no direct per-robot caps. Plaintext tokens shown once; Argon2id
hashes persisted. Tombstone + revive semantics preserve `builds.user_email`
history across identity rotation.

Three preparatory refactors (P1, P2, P3) land first. Each is independently
valuable: the `cbc` `--yes-i-really-mean-it` flag on irreversible commands, the
admin URL reshape under `/api/admin/entity`, and token usage tracking on all
token types.

### Commit 2: `cbc: require --yes-i-really-mean-it on irreversible commands` (P1)

**~100 authored lines**

Establish the confirmation-flag pattern on destructive, non-reversible `cbc`
commands. Scope is limited to commands that erase data or identity without an
automatic path back. Reversible operations (`admin users deactivate`,
`build revoke`) are explicitly excluded ‚Äî deactivate can be paired with
activate, and build revocation flips a state flag.

**In scope (currently missing the flag):**

- `cbc admin roles delete` ‚Äî currently has `--force`; replace with
  `--yes-i-really-mean-it` (same semantics, renamed).
- `cbc admin workers deregister` ‚Äî no flag today; add it.
- `cbc periodic delete` ‚Äî no flag today; add it.

**Out of scope (soft-delete or reversible):**

- `cbc admin channels delete`, `cbc admin channels type delete` ‚Äî soft-delete,
  reversible.
- `cbc admin users deactivate` ‚Äî reversible via `activate`.
- `cbc build revoke` ‚Äî flips a state flag, not data loss.

**Files:**

| File                                    | Change                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `cbc/src/admin/roles.rs`                | Replace `force: bool` on `DeleteArgs` with `yes_i_really_mean_it: bool` (`#[arg(long = "yes-i-really-mean-it")]`). Rename the associated help text. Reject the command when the flag is absent with a clear error; do not pass a `?force=` query any more ‚Äî the server-side role-delete endpoint must be updated to remove the `force` parameter.                                        |
| `cbc/src/worker.rs`                     | `DeregisterArgs` gains `yes_i_really_mean_it: bool`. Client-side enforcement; no server change.                                                                                                                                                                                                                                                                                          |
| `cbc/src/periodic.rs`                   | `PeriodicDeleteArgs` gains the same flag; client-side enforcement.                                                                                                                                                                                                                                                                                                                       |
| `cbsd-server/src/routes/permissions.rs` | `DELETE /api/permissions/roles/{name}` ‚Äî remove the `force` query parameter handling entirely; the endpoint always cascades (prior behaviour of `?force=true`). **This is a breaking server API change** ‚Äî any external caller passing `?force=` will see the parameter ignored or rejected. The `cbc` client in this same commit stops sending it, so internal callers stay consistent. |

**Key details:**

- The `cbc` flag rename is client-side UX. The server-side `?force` parameter
  removal is a separate concern bundled here because it is a direct consequence:
  `--force` was the only production caller of `?force=true` and there is no
  longer a client-side way to set it.
- **Breaking API change ‚Äî commit message must call this out explicitly.** The
  server-side removal of the `?force` query parameter on
  `DELETE /api/permissions/roles/{name}` is a contract change for any external
  tooling calling that endpoint directly. No known external consumers exist for
  the admin/permissions surface, but the notice must be visible in the commit
  log so downstream users can audit.
- Error message when the `cbc` flag is missing:
  `"this is a destructive operation; pass --yes-i-really-mean-it to confirm"`.

### Commit 5: `cbsd-rs/server: track first_used_at and last_used_at for tokens and api_keys` (P3)

**~150 authored lines**

Symmetrical usage visibility across all three token types. The robot work
(commit 6) creates `robot_tokens` with these columns from the start; P3 adds
them to the two existing tables so all auth paths write them uniformly.

**Files:**

| File                                                                   | Change                                                                                                                                                                                                                       |
| ---------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `migrations/006_token_usage_tracking.sql`                              | New. `ALTER TABLE tokens ADD COLUMN first_used_at INTEGER; ALTER TABLE tokens ADD COLUMN last_used_at INTEGER; ALTER TABLE api_keys ADD COLUMN first_used_at INTEGER; ALTER TABLE api_keys ADD COLUMN last_used_at INTEGER;` |
| `cbsd-server/src/auth/paseto.rs` (or wherever `validate_paseto` lives) | After successful verification, write: `UPDATE tokens SET last_used_at = unixepoch(), first_used_at = COALESCE(first_used_at, unixepoch()) WHERE id = ?`. No throttle.                                                        |
| `cbsd-server/src/auth/api_keys.rs`                                     | `verify_api_key` (or the post-verify path) writes the same update against `api_keys`.                                                                                                                                        |
| `cbsd-server/src/db/tokens.rs`, `db/api_keys.rs`                       | Add helper `mark_used(pool, id)` that performs the single-row UPDATE. Both auth sites call it.                                                                                                                               |
| `.sqlx/`                                                               | Regenerated for the new migration and queries.                                                                                                                                                                               |

**Key details:**

- Both fields are nullable. `first_used_at` uses `COALESCE` to become immutable
  once set.
- No throttle. SQLite WAL handles the write volume at the realistic auth rate.
- `mark_used` is invoked outside any request-path transaction. A failed
  usage-update must not fail the request ‚Äî the error is swallowed, but **every
  swallow emits `tracing::warn!` with the failing cause and the token id**. No
  bare `let _ = ...` without a log line.
- **Write-only columns until commit 6.** `first_used_at` and `last_used_at` are
  populated by this commit but no `GET` endpoint or `list` response exposes them
  here. Commit 6 adds the robot equivalents and the response surface that reads
  these columns; commit 7 may add a read path for the existing `tokens` and
  `api_keys` tables (not required by the design). This is intentional
  scaffolding, not dead-write ‚Äî flag for reviewers so the D11 deduction is not
  applied at commit 5's review.

**Validation:** migrations run cleanly on a fresh DB and on an existing DB;
`SQLX_OFFLINE=true cargo build --workspace` passes; unit test for `mark_used`
idempotence on `first_used_at`; unit test asserting the `tracing::warn!` is
emitted on a forced `mark_used` failure (e.g., against a closed pool).

### Commit 6: `cbsd-rs/server: provision robot accounts with authenticated bearer tokens` (R1)

**~900‚Äì1100 authored lines**

Delivers the core feature end-to-end on the server: an admin can create, list,
inspect, and tombstone robot accounts; the issued `cbrk_` token authenticates
API requests. Token rotation (issuing a new token without discarding the
account) and purely cosmetic coexistence guards are deferred to commit 7.

**Schema ‚Äî `migrations/007_robot_accounts.sql`:**

```sql
ALTER TABLE users ADD COLUMN is_robot INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN robot_description TEXT;

CREATE TABLE robot_tokens (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    robot_email   TEXT    NOT NULL
                  REFERENCES users(email) ON DELETE CASCADE,
    token_hash    TEXT    NOT NULL UNIQUE,
    token_prefix  TEXT    NOT NULL,
    expires_at    INTEGER,
    revoked       INTEGER NOT NULL DEFAULT 0,
    first_used_at INTEGER,
    last_used_at  INTEGER,
    created_at    INTEGER NOT NULL DEFAULT (unixepoch())
);
CREATE INDEX idx_robot_tokens_prefix ON robot_tokens(token_prefix);
CREATE INDEX idx_robot_tokens_robot  ON robot_tokens(robot_email);
CREATE UNIQUE INDEX idx_robot_tokens_active
    ON robot_tokens(robot_email) WHERE revoked = 0;
```

Both ALTERs and the new table land in a single migration file ‚Äî they are one
conceptual change (introducing robot identity).

`robot_tokens.expires_at` is explicitly nullable. **A `NULL` value means the
token never expires** (the `"infinity"` input from the REST API maps to `NULL`).
The auth extractor's expiry check is
`expires_at IS NULL OR expires_at > unixepoch()`; the `token_status.state`
reporter in `GET /api/admin/robots/{name}` treats `NULL` as never-expiring.

**Entity deactivate / activate ‚Äî `cbsd-server/src/routes/admin.rs`:**

The two entity handlers must distinguish _disable_ (reversible; tokens intact)
from _tombstone_ (permanent; tokens revoked) for robot targets. The design's
Disable-vs-Revoke-vs-Tombstone table (design v3 ¬ß Authentication Flow) is
authoritative: only the dedicated `DELETE /api/admin/robots/{name}` path
performs a tombstone, and only `POST /api/admin/robots` with a tombstoned name
performs a revive. The entity routes are pure disable/enable.

`deactivate_entity` ‚Äî after the last-admin guard transaction commits, branch on
`target.is_robot`:

- **Human (is_robot=0):** existing behaviour ‚Äî revoke PASETO tokens, revoke API
  keys, purge cache by owner.
- **Robot (is_robot=1):** set `users.active = 0`; purge cache entries by owner
  (both `ApiKey` and `RobotToken` kinds ‚Äî `remove_by_owner` covers both). **Do
  NOT touch `robot_tokens` rows.** The robot's credential material is preserved
  so that `activate_entity` can restore function without issuing a new token.
  This is the design's disable semantics.

Note on last-admin guard for robot targets: the guard's query has
`AND u.is_robot = 0`, so no robot is ever counted as a wildcard holder. The
guard runs unconditionally (before the branch) but is a no-op for robot targets
‚Äî the count never changes when a non-admin is deactivated. Retained for uniform
handling.

`activate_entity` ‚Äî after the target lookup, branch on `target.is_robot` and
`target.active`:

- **Human (is_robot=0), active=0:** existing behaviour ‚Äî set `active = 1`.
- **Human (is_robot=0), active=1:** idempotent success.
- **Robot (is_robot=1), active=1:** idempotent success.
- **Robot (is_robot=1), active=0, has at least one `robot_tokens` row with
  `revoked = 0`:** _disabled_ state ‚Äî set `users.active = 1`. Auth begins
  succeeding again with the preserved token. Purge cache by owner so the prior
  entries (staled while active=0) do not shadow the fresh lookup.
- **Robot (is_robot=1), active=0, no non-revoked `robot_tokens` rows:** either
  tombstoned or disabled-after-token-revoke. In both cases there is no
  credential material to restore; simply flipping `active = 1` would produce an
  authentication-less robot. Return **400** with a message pointing at
  `POST /api/admin/robots` (revive-via-create, which issues a fresh token) or at
  `POST /api/admin/robots/{name}/token` (commit 7, once available) for the
  disabled-after-revoke case.

The "has at least one `robot_tokens` row with `revoked = 0`" check is a single
`SELECT EXISTS(...)` query executed inside the handler, not under a long-held
transaction. A concurrent tombstone could in principle invalidate the check
between the existence query and the `active = 1` update, but the net result ‚Äî a
concurrently-tombstoned robot flipping briefly to `active = 1` ‚Äî is benign: the
next auth request sees `users.active = 1` but finds no non-revoked token and
fails (and a subsequent admin pass through the tombstone path will set
`active = 0` again).

**Key details:**

- `rotate_token` runs inside a transaction opened by the handler with
  `BEGIN IMMEDIATE`. This matches the serialisation strategy of `create_robot` /
  `revive_robot` and closes the concurrent-rotation race: two simultaneous
  `POST /api/admin/robots/{name}/token` calls are serialised at lock
  acquisition; the loser waits for the winner to commit, then re-reads and
  proceeds against the post-revoke state (where the only non-revoked row is the
  winner's new token). Relying on the partial unique index alone would surface a
  raw SQLite `SQLITE_CONSTRAINT_UNIQUE` (code 2067) to the loser, which sqlx
  would bubble as a 500 unless the handler explicitly mapped it ‚Äî
  `BEGIN IMMEDIATE` avoids that mapping entirely. No retry loop is required in
  the handler; SQLite's lock wait handles it transparently.
- `list_entities` with `?type=robot` is the server-side filter that backs the
  cbc `admin users list` client-side restriction added in commit 8.
- The `${username}` reject messages must tell the operator which channel and
  template are the source of the reject, not just "robot identity cannot use
  this channel".

**Tests (in this commit):**

Token rotation:

- `POST /api/admin/robots/{name}/token` without `renew=true` against a robot
  with an expired-but-not-revoked token: 409, no new row (the expired row stays
  exactly as it was ‚Äî same `id`, `revoked = 0`, same `expires_at`).
- `POST /api/admin/robots/{name}/token` with `renew=true` against an
  expired-but-not-revoked token: atomic rotation ‚Äî the **old row's `revoked`
  field is explicitly `1`** (not just "count of active rows is 1"; the partial
  unique index would satisfy count=1 even under bad orderings); a new row with
  `revoked = 0` and a different `token_hash` exists; the old token no longer
  authenticates; the new token does.
- `POST /api/admin/robots/{name}/token` with `renew=true` against a
  currently-active (non-expired) token: same rotation behaviour; confirms
  `renew` covers both expired and active cases per design.
- `POST /api/admin/robots/{name}/token` without `renew=true` against a robot
  whose only token row is revoked: create (no 409); exactly one non-revoked row
  exists after. This is the "no active credential, safe to create" path from the
  design's `token new` semantics table.
- Concurrent rotation of the same robot: two `POST .../token` requests with
  `renew=true` fire simultaneously. Both complete successfully with distinct
  `token_hash` values; after both commit, exactly one row has `revoked = 0` (the
  winner of the lock race) and the other two rows (the original plus the
  intermediate) have `revoked = 1`. Neither request returns 500. Confirms
  `BEGIN IMMEDIATE` serialises the transactions and no
  `SQLITE_CONSTRAINT_ UNIQUE` error surfaces.
- `POST /api/admin/robots/{name}/token` against a nonexistent robot name: 404
  with a descriptive error body. No partial mutation.

Standalone revoke:

- `DELETE /api/admin/robots/{name}/token` against a live robot: active row's
  `revoked = 1`; `users.active` unchanged; robot can no longer authenticate.

  succeeds (no non-revoked rows present).

  non-revoked tokens (already revoked or never issued): **idempotent 200 OK**

  "ensure no active token", which is already the state; a 404 here would force
  retries through status-code branching for a desired-state operation. Matches
  the idempotency of tombstone + revive.
- `DELETE /api/admin/robots/{name}/token` against a **nonexistent robot name**

  no-op prevents a typo in the name from masquerading as a successful revoke.

  on present row. Same policy applies to
  `PUT /api/admin/robots/{name}/description` and every other
  `/api/admin/robots/{name}/*` handler that targets a named robot.
- Tests: `GET /api/admin/robots/nonexistent` returns 404 (covers the read path ‚Äî
