- **v3 (2026-04-19)** — folds in clarifications requested by the v2 design

## Overview

Robot accounts are admin-managed service identities that allow automation
pipelines and CI systems to authenticate against the CBS service. Unlike human
users — who authenticate via Google OAuth and receive short-lived PASETO tokens
— robot accounts authenticate with a long-lived token issued at creation time
and managed exclusively by administrators.

### Goals

- Distinct, human-readable robot identity with explicit `robot:` prefix in all
  audit records.
- Admin-only lifecycle management: create (which also reactivates a tombstoned
  name), disable, enable, tombstone, and token rotation.
- All robot permissions flow through role assignments. There are no direct
  per-robot capability or scope grants.
- Plaintext token shown once (at creation — which covers both fresh create and
  revive-via-create — or at rotation); only an Argon2id hash is ever persisted.
- Robots are unambiguously distinguishable from human users in all API responses
  and logs.
- All destructive operations require an explicit `--yes-i-really-mean-it` flag.

### Non-Goals

- Self-service robot account creation (admin-only).
- Robot accounts creating their own API keys.
- Multiple simultaneous active tokens per robot.
- Permanent hard deletion of robot rows (build-history attribution must survive
  account removal).

## Preparatory Work

The robot-account feature depends on three preparatory refactors that must land
before any robot-specific code. Each is independently useful, independently
testable, and not on the critical path once complete. The implementation plan
must sequence these first.

### P1 — `--yes-i-really-mean-it` audit for `cbc`

Every existing destructive `cbc` command must be updated to require the
`--yes-i-really-mean-it` flag. Known candidates (non-exhaustive):

- `cbc admin users deactivate`
- `cbc admin roles delete`
- `cbc admin workers delete`
- Any build-revocation or periodic-task-delete commands currently lacking a
  confirmation flag

This establishes the pattern that all robot-specific destructive commands
(`cbc admin robots delete`, `cbc admin robots token revoke`, etc.) adopt.

### P2 — API entity refactor

**Scope: REST refactor only.** P2 introduces no robot-specific code, schema, or
logic. Its sole purpose is to reshape the admin API surface so that
account-level operations that apply to any row in the `users` table are grouped
under a single `/api/admin/entity/{email}/...` path rather than being
misleadingly lumped under `/api/admin/users/...` or
`/api/permissions/users/...`. At the end of P2 the system behaves identically to
before for every human user; the only observable change is the set of URLs the
handlers respond at. No `is_robot` column exists yet; no robot-awareness is
added anywhere.

Endpoint migration map (old → new):

- `PUT /api/admin/users/{email}/deactivate` →
  `PUT /api/admin/entity/{email}/deactivate`
- `PUT /api/admin/users/{email}/activate` →
  `PUT /api/admin/entity/{email}/activate`
- `PUT /api/admin/users/{email}/default-channel` →
  `PUT /api/admin/entity/{email}/default-channel`
- `GET /api/permissions/users/{email}/roles` →
  `GET /api/admin/entity/{email}/roles`
- `PUT /api/permissions/users/{email}/roles` →
  `PUT /api/admin/entity/{email}/roles`
- `POST /api/permissions/users/{email}/roles` →
  `POST /api/admin/entity/{email}/roles`
- `DELETE /api/permissions/users/{email}/roles/{role}` →
  `DELETE /api/admin/entity/{email}/roles/{role}`
- `GET /api/permissions/users` → `GET /api/admin/entities`

The `GET /api/admin/entities` endpoint in P2 returns every row in the `users`
table (all humans). The `?type=user|robot|all` query parameter is introduced
later by the robot-specific work when the `is_robot` column exists; in P2 the
endpoint accepts no filter.

Endpoints that remain untouched (not entity-scoped):

- `/api/permissions/roles*` — role definitions are global, not account-bound.
- `/api/admin/queue`, `/api/admin/workers*` — operational / worker-identity
  endpoints.
- `/api/auth/*` — per-user self-service (api-keys, token revoke, whoami).
- `/api/builds/*`, `/api/channels/*`, `/api/components/*`, `/api/periodic/*`.

**Unify the dual last-admin guard.** The current `deactivate_user` handler in
`routes/admin.rs` carries an inline SQL count of wildcard holders that does
**not** call `count_active_wildcard_holders`. P2 replaces the inline query with
a call to the function so that every last-admin check site runs through one code
path. No filter semantics change in P2 — the function remains the same as today;
the robot-specific `is_robot = 0` predicate is added later by the robot work.
This is purely a DRY cleanup that prepares the ground.

**CLI UX is preserved:** `cbc admin users` and `cbc admin roles` continue to
exist with the same user-facing commands; only the internal REST paths they call
change. This keeps the human-facing command tree stable while the server surface
reflects the data model.

### P3 — Token usage tracking

Add two nullable columns to each token table:

```sql
ALTER TABLE tokens    ADD COLUMN first_used_at INTEGER;
ALTER TABLE tokens    ADD COLUMN last_used_at  INTEGER;
ALTER TABLE api_keys  ADD COLUMN first_used_at INTEGER;
ALTER TABLE api_keys  ADD COLUMN last_used_at  INTEGER;
-- robot_tokens is created with these columns already present
-- (see Database Schema Changes).
```

Update all three auth paths (PASETO in `validate_paseto`, API key in
`verify_api_key`, robot token in the new path) to write `first_used_at` (if
NULL) and `last_used_at` on every successful authentication. No throttle. SQLite
WAL handles the write volume; the realistic auth rate is well below the
threshold where batching would be beneficial.

This gives admins symmetrical visibility across all token types and removes a
class of edge cases that per-token-type throttling would introduce.

## Identity Model

### Robots in the `users` Table

Robot accounts are stored in the existing `users` table alongside human
accounts. This preserves all existing FK chains — notably
`builds.user_email → users(email)` — without schema changes to the `builds`
table or any query that joins on it.

Two new columns are added to `users` as part of the robot-specific
implementation phase (not P2; P2 is pure REST refactor and introduces no schema
changes):

```sql
ALTER TABLE users ADD COLUMN is_robot INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN robot_description TEXT;
```

A robot entry uses a **synthetic email** as its primary key:

```
Synthetic email (stored): robot+<name>@robots
Display identity (shown):  robot:<name>
```

The `users.name` column stores the display identity `"robot:<name>"` directly.
There is no separate storage for the bare name; it is derived from the synthetic
email by stripping the `robot+` prefix and `@robots` suffix when required (URL
paths, `robot_description` lookup by bare name).

The `robot_description` column holds the optional description; it is `NULL` for
human accounts.

### Name Validation

Robot names (the bare form) are validated at creation time:

- Pattern: `^[a-zA-Z0-9_][a-zA-Z0-9_.-]*[a-zA-Z0-9_]$` (minimum 2 characters;
  must start and end with an alphanumeric character or underscore; interior may
  contain `.` and `-`).
- Maximum length: 64 characters.
- Must be unique (enforced by the primary key on the synthetic email).
- Immutable after creation. A tombstoned name may be reused via the revive path
  (see REST API).

The human-account creation path (SSO first-login) rejects any `users.name` value
that starts with `robot:`, preventing identity forgery via OAuth display-name
shenanigans.

### Audit Trail

All builds submitted and all admin operations performed by a robot appear in API
responses and server logs with the display identity `robot:<name>`. Because
`users.name` stores the display form directly, a
`display_identity(user: &AuthUser) -> &str` helper returns `user.name` for
robots and `user.email` for humans, selected via `user.is_robot`. Call sites
include the builds listing, build detail, and structured-log fields that record
the actor.

The `is_robot` flag is surfaced on the `AuthUser` extractor so route handlers
and audit-logging middleware can distinguish robot from human actors without
string-parsing the email.

### Channel Template Constraints for Robots

The server resolves the `${username}` template variable in channel prefix
configurations by taking the substring before `@` in the caller's email. For a
human user `alice@example.com`, this yields `alice` — the intended local-part
identity. For a robot with synthetic email `robot+ci-builder@robots`, the same
substring is `robot+ci-builder`, which embeds the internal `robot+` sentinel
into the resolved output. This is never meaningful in a user-facing channel
prefix.

**Design rule.** Robot accounts must not route builds through any channel whose
prefix template contains `${username}`. This is enforced at two points:

1. **Channel assignment.** `PUT /api/admin/entity/{email}/default-channel`
   rejects with 400 if the target email has `is_robot = 1` and the requested
   channel's prefix template contains the literal string `${username}`.
2. **Build submission.** On `POST /api/builds`, when the authenticated caller
   has `is_robot = true` and the resolved channel (either the explicit channel
   in the request or the caller's `default_channel_id`) has a prefix template
   containing `${username}`, the request is rejected with 400 and a message
   explaining that the `${username}` token is not defined for robot identities.

Robots are expected to use channels with static prefixes or prefixes that
reference other, well-defined template variables. Administrators who want
per-robot segmentation can create dedicated channels with static prefixes
(`ci-builder/…`, `nightly/…`, etc.) and point each robot's `default_channel_id`
at the appropriate one.

Note: no silent prefix-stripping transformation (e.g., "just strip `robot+` for
robots") is performed. The design prefers an explicit error at submission time
over an implicit rewrite that could mask configuration drift — a robot whose
default channel is misconfigured should fail loudly, not silently produce paths
that differ from what the operator expects to see.

### Coexistence with Existing User Endpoints

After the P2 entity refactor, the human-user-specific endpoints that remain
under `/api/admin/users/...` and `/api/permissions/users/...` are SSO-specific
and never take an arbitrary email as a path parameter for account-level
mutation. The coexistence concerns collapse to a small number of cases:

| Endpoint / command                 | Policy                             |
| ---------------------------------- | ---------------------------------- |
| `POST /api/auth/api-keys`          | 400 if target has `is_robot = 1`   |
| `POST /api/auth/tokens/revoke-all` | 400 if caller has `is_robot = 1`   |
| `GET  /api/auth/whoami`            | Response gains `is_robot: bool`    |
| `GET  /api/admin/entities`         | Filters by `type=user\|robot\|all` |
| `cbc admin users list`             | Internally filters `is_robot = 0`  |
| `cbc admin users get <email>`      | Errors if target is a robot email  |

Entity endpoints themselves accept any email and work uniformly for both row
types; the `is_robot` column drives any per-type handling inside the handler.

**Why `POST /api/auth/tokens/revoke-all` does not apply to robot callers.** This
endpoint is self-service: it revokes every **PASETO session token** owned by the
authenticated caller by issuing
`UPDATE tokens SET revoked = 1 WHERE user_email = ? AND revoked = 0` against the
`tokens` table. PASETO sessions exist only for human users — they are minted by
the OAuth callback after Google sign-in and stored server-side via the BFF
session cookie or returned as a bearer for CLI use. Robot accounts never acquire
a PASETO session; they authenticate exclusively via `cbrk_`-prefixed bearer
tokens stored in `robot_tokens`. A robot calling `tokens/revoke-all` would find
zero rows to update in `tokens` while its `robot_tokens` row remained active —
producing a misleading `200 OK` that implies all the robot's credentials were
revoked when in fact nothing happened. Returning `400 Bad Request` with an
explanatory message directs the caller to the correct endpoints:
`DELETE /api/admin/robots/{name}/token` to revoke the single active robot token
(requires `robots:manage`, so a robot cannot self-revoke anyway, but the correct
error surfaces the model).

## Token Design

### Format

Robot tokens use a new prefix distinct from user API keys (`cbsk_`):

```
cbrk_<64 hex chars>    (32 cryptographically random bytes)
```

`cbrk_` = "CBS Robot Key". The prefix is checked first by the auth extractor to
select the correct verification path.

### Storage

- **Argon2id hash** of the raw token stored in `robot_tokens` table.
- **First 12 hex chars** (`token_prefix`) used for fast candidate lookup before
  hash verification, mirroring the `api_keys` pattern.
- **Plaintext shown once** at creation (including revive-via-create) or
  rotation; never stored or logged.
- **One active token per robot** at any time, enforced by a partial unique index
  (see Database Schema Changes).

### Expiry

Token expiry is a property of each **token**, not of the robot account. Robot
accounts themselves never expire — they are active until an admin explicitly
disables or tombstones them.

Expiry is set when a token is created (whether at robot account creation or via
`token new`). Accepted values:

- An ISO calendar date `YYYY-MM-DD`. The token is valid through the end of that
  UTC day; stored as Unix epoch seconds of `00:00:00 UTC` on the day **after**
  the given date.
- `null` — stored as `NULL` in `robot_tokens.expires_at`. The token never
  expires.

The `expires` field is **required** on every create and rotate request body;
omitting it returns 400. This preserves the security property that a
never-expiring token is always an explicit caller opt-in (via
`"expires": null`), not a state reached by forgetting a field.

**Timezone semantics.** All date inputs are interpreted as UTC. An admin in a
non-UTC timezone who writes `--expires 2026-12-31` gets a token that expires at
`2027-01-01 00:00:00 UTC`, not at their local midnight. The `cbc` help text, API
documentation, and server-side error messages must state this explicitly so
operators are not surprised. If an admin needs the token to expire at a specific
local instant, they can compute the appropriate UTC calendar date themselves.

Example: `"2026-12-31"` → `expires_at = 1767225600` (epoch for
`2027-01-01T00:00:00Z`). The token authenticates successfully at any instant
before that epoch.

Duration arithmetic (`30d`, `6mo`, `1y`, …) is deferred. The REST API takes a
date; `cbc` may add a duration-to-date helper as a purely client-side sugar in a
follow-on change if users request it.

### Token States

A robot token can be in one of three states:

| State   | Condition                    | Effect on auth                 |
| ------- | ---------------------------- | ------------------------------ |
| Active  | `revoked=0`, not yet expired | Authenticates successfully     |
| Expired | `revoked=0`, past expiry     | Auth fails; row is not revoked |
| Revoked | `revoked=1`                  | Auth fails; row is a tombstone |

A revoked token row is a **tombstone**: it signals that the admin deliberately
removed that credential (vs. expiry, which is passive). This distinction drives
`token new` semantics (see below). Revoked rows accumulate; no cleanup policy is
defined for this iteration — volumes are expected to be small and inspection
value outweighs storage cost.

### `token new` Semantics

The `token new` command creates a token for a robot. Its behaviour depends on
the current token state:

| Current token state     | `token new`           | `token new --renew` |
| ----------------------- | --------------------- | ------------------- |
| No row exists           | Create ✓              | Create ✓            |
| Only revoked rows       | Create ✓              | Create ✓            |
| Active or expired token | **Refuse** with error | Rotate ✓            |

The `--renew` flag is the explicit acknowledgement that an existing credential
(whether still valid or merely expired) will be replaced. Without it,
`token new` is safe to call on a fresh or previously-revoked robot without risk
of accidental token rotation.

### Usage Tracking

The `robot_tokens` table records `first_used_at` and `last_used_at` with the
same semantics as the new columns on `tokens` and `api_keys` (see P3). Both
fields are written on every successful authentication. No throttle.

### Token Rotation

`token new --renew` follows the crash-safe pattern used for worker token
rotation:

1. In a single transaction:
   - Mark the old non-revoked row `revoked = 1`.
   - Insert the new `robot_tokens` row.
2. On commit, invalidate the old token's cache entry.
3. Return the new plaintext in the response body once.

The partial unique index on `robot_tokens(robot_email) WHERE revoked = 0`
guarantees no window where two active rows coexist: if the insert were attempted
before the revoke, the constraint would reject the transaction. Performing the
revoke first is therefore required.

## Database Schema Changes

All schema changes for this feature land as part of the robot-specific
implementation phase. Preparatory phases P1, P2, and P3 do **not** add or modify
schema for robot accounts (P3 adds `first_used_at` / `last_used_at` to `tokens`
and `api_keys`, which is a separately-scoped change that benefits all token
types, not a robot-specific column).

Two migrations are required, both authored together during the robot-specific
work:

### Migration A (users table extensions)

```sql
ALTER TABLE users ADD COLUMN is_robot          INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN robot_description TEXT;
```

### Migration B (robot token storage)

```sql
CREATE TABLE robot_tokens (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    robot_email   TEXT    NOT NULL
                  REFERENCES users(email) ON DELETE CASCADE,
    token_hash    TEXT    NOT NULL UNIQUE,  -- Argon2id hash
    token_prefix  TEXT    NOT NULL,         -- first 12 hex chars
    expires_at    INTEGER,                  -- NULL = no expiry
    revoked       INTEGER NOT NULL DEFAULT 0,
    first_used_at INTEGER,
    last_used_at  INTEGER,
    created_at    INTEGER NOT NULL DEFAULT (unixepoch())
);

CREATE INDEX idx_robot_tokens_prefix
    ON robot_tokens(token_prefix);
CREATE INDEX idx_robot_tokens_robot
    ON robot_tokens(robot_email);

-- At most one active (non-revoked) row per robot. SQLite supports
-- partial indexes; this makes the "single active token" invariant
-- structural rather than application-enforced, and closes the
-- concurrent-create race.
CREATE UNIQUE INDEX idx_robot_tokens_active
    ON robot_tokens(robot_email)
    WHERE revoked = 0;
```

**Notes:**

- Robot accounts do not have an `expires_at` on the `users` row. Expiry is a
  property of the token only.
- Role assignments reuse the existing `user_roles` table; no new table is
  needed, and the P2 entity endpoints already manage assignments uniformly for
  both row types.
- The `builds.user_email` FK requires no change — the robot's synthetic email
  satisfies it and survives tombstone/revive cycles.
- Because tombstoning preserves the `users` row (`active = 0`, `is_robot = 1`),
  no `ON DELETE` clause is ever exercised for `builds.user_email` in robot
  workflows.

## Permissions Model

### New Capabilities

| Capability      | Purpose                                            |
| --------------- | -------------------------------------------------- |
| `robots:manage` | Create (incl. revive), tombstone; manage tokens    |
| `robots:view`   | List and inspect robot accounts (no token content) |

These are added to the known-capabilities list in `routes/permissions.rs`. The
builtin `admin` role carries `*` (wildcard), which subsumes both new
capabilities automatically.

### Role-Based Permissions Only

A robot's effective permissions come exclusively from the union of capabilities
and scopes contributed by its assigned roles. There is no per-robot direct
cap/scope assignment mechanism. This preserves the invariant documented in
design 003 that `require_scopes_all` evaluates AND-within-assignment,
OR-across-assignments; direct assignments would have broken this invariant
because they produce no `AssignmentWithScopes` record.

If a robot needs a combination of capabilities not expressible through existing
roles, the admin workflow is:

1. Create a new named role with the exact caps and scopes needed.
2. Assign that role to the robot.

This keeps all permission evaluation through one code path
(`get_user_assignments_with_scopes`) and produces inspectable, auditable role
names rather than ad-hoc per-identity grants.

### Forbidden Capabilities

The following capabilities **must not** appear in a robot's effective set.
Enforcement is applied at **auth time** by stripping them from the computed
effective caps, making the restriction robust against role updates that later
add a forbidden cap to an assigned role. Validation at role-assignment time also
rejects assignments of roles containing these caps to robots.

| Capability           | Reason                                      |
| -------------------- | ------------------------------------------- |
| `*`                  | Full admin — privilege escalation           |
| `permissions:manage` | Robots must not manage users or roles       |
| `robots:manage`      | Robots must not create or manage robots     |
| `apikeys:create:own` | Robot accounts do not use personal API keys |

### Role Updates and Forbidden Caps

Forbidden-cap enforcement has two layers:

1. **Auth-time strip.** On every token verification, the computed effective cap
   set is filtered to remove any forbidden cap before `AuthUser` is returned.
   This is the primary guard.
2. **Assignment-time reject.** `POST /api/admin/entity/{email}/roles` refuses to
   assign a role to a robot target when the role's current `role_caps` contain
   any forbidden cap.

These two layers do not cover a third mutation path: an admin editing an
already-assigned role via `PUT /api/permissions/roles/{name}/caps` to add a
forbidden cap. The assignment-time check runs on assignment, not on role edits,
and the design deliberately does not cascade role edits to re-validate all
existing assignees (that would require scanning `user_roles` on every role
mutation).

This is **not** a security gap: the auth-time strip always runs before any
permission check, so the forbidden cap is never present in a robot's effective
set at decision time. It is an **observability** consideration only: an operator
looking at raw `role_caps` for a role and then at the set of robots assigned to
that role might infer that those robots hold the forbidden cap. They do not.

`GET /api/admin/robots/{name}` returns `effective_caps` computed through the
same forbidden-cap strip as the auth path, so the visible cap set in the API
response is always accurate.

### Zero-Cap Baseline

A robot with no role assignments has an empty effective cap set. Every
permission check fails. This is the correct zero-trust baseline and requires no
special handling in handlers or middleware — the existing `AuthUser::has_cap`
returns `false` for every capability, and the normal `403 Forbidden` path
applies.

### Last-Admin Guard

`db::roles::count_active_wildcard_holders` is updated by the robot-specific work
to include `AND u.is_robot = 0` in its query. A robot cannot serve as the "last
admin" anchor. This is defense-in-depth: the forbidden-cap stripping is the
primary protection; the query-level filter is a secondary guarantee that holds
even if the stripping logic has a bug.

The call-site consolidation that makes this filter effective everywhere happens
in P2, before any robot code exists. The current `deactivate_user` handler in
`routes/admin.rs` carries an inline SQL query that counts wildcard holders
independently of `count_active_wildcard_holders`. P2 replaces the inline query
with a call to the function, so there is exactly one last-admin check code path.
P2 does not change the function's query semantics — it only eliminates the
duplicate. When the robot work adds `AND u.is_robot = 0` to the function, every
call site (including the former inline one) inherits the filter for free.

## Authentication Flow

### Unified Token Cache

The existing `ApiKeyCache` (storing `CachedApiKey` entries) is renamed and
generalized to serve both API keys and robot tokens:

- Struct renamed: `ApiKeyCache` → `TokenCache`.
- Entry renamed: `CachedApiKey` → `CachedToken`.
- New `kind: TokenKind { ApiKey, RobotToken }` discriminator on `CachedToken` to
  route invalidation and verification to the correct DB table.

The cache is keyed by SHA-256 of the raw bearer token, unchanged. Reverse
indexes by `owner_email` and by `prefix` continue to exist and work uniformly
for both token types because:

- Robot synthetic emails (`robot+<name>@robots`) cannot collide with real user
  emails due to the `@robots` suffix.
- Token prefixes are globally unique across both tables (different bearer-prefix
  header discriminates, and within each table the 12-hex prefix collision
  probability is negligible).

The `AppState.api_key_cache` field is renamed to `AppState.token_cache`.
Capacity is unchanged.

### Extractor Changes

`AuthUser` gains an `is_robot: bool` field:

```rust
pub struct AuthUser {
    pub email: String,   // synthetic email for robots
    pub name: String,    // "robot:<name>" for robots
    pub caps: Vec<String>,
    pub is_robot: bool,  // NEW
}
```

When the `Authorization: Bearer` value starts with `cbrk_`:

1. Compute SHA-256 of the raw token.
2. Cache lookup: on hit, validate expiry, load user via `load_authed_user`,
   return with `is_robot = true`.
3. On miss: extract first 12 hex chars as the lookup prefix; query
   `robot_tokens` for candidate rows with matching prefix and `revoked = 0`.
4. Argon2id-verify the raw token against each candidate (typically one) inside
   `spawn_blocking`.
5. Check `users.active = 1` and `robot_tokens.expires_at` (NULL or future).
6. Update `first_used_at` (if NULL) and `last_used_at` on the verified row.
7. Cache the verified entry with `kind = RobotToken`.
8. Load effective caps and scopes via the shared `load_authed_user` path (which
   reads `user_roles`, `role_caps`, `role_scopes`). Strip forbidden caps.
9. Return `AuthUser` with `is_robot = true` (read from the `users` row).

Cache entries are invalidated on token revocation, robot disable, and robot
tombstone/revive. The existing `remove_by_owner` and `remove_by_prefix` methods
suffice.

### Disable vs. Revoke vs. Tombstone

| Action       | `users`    | `robot_tokens`       | Reversible?    |
| ------------ | ---------- | -------------------- | -------------- |
| Disable      | `active=0` | Unchanged            | Yes (enable)   |
| Revoke token | Unchanged  | Active row → revoked | No (new token) |
| Tombstone    | `active=0` | All rows → revoked   | Yes (revive)   |

**Disable** leaves the token rows intact so that re-enabling restores
authentication immediately — no new token issuance.

**Revoke** kills a specific credential without touching the account; a new token
can be issued afterward via `token new` (no `--renew` needed because only
revoked rows remain).

**Tombstone** is the robot-deletion primitive: the account is deactivated and
all token material is revoked in the same transaction, but the `users` row is
preserved so that `builds.user_email` history survives. Calling
`POST /api/admin/robots` with the tombstoned name reactivates it as a
semantically new identity via the revive path described in the REST API section;
there is no separate `revive` verb.

## REST API

All robot-specific endpoints are under `/api/admin/robots`. Entity endpoints at
`/api/admin/entity/{email}` and `/api/admin/entities` (introduced in P2) handle
role assignments, default channel, activate/deactivate, and listing — these
apply to both humans and robots.

### Robot-Specific Endpoints

| Method   | Path                                   | Cap             |
| -------- | -------------------------------------- | --------------- |
| `POST`   | `/api/admin/robots`                    | `robots:manage` |
| `GET`    | `/api/admin/robots`                    | `robots:view`   |
| `GET`    | `/api/admin/robots/{name}`             | `robots:view`   |
| `PUT`    | `/api/admin/robots/{name}/description` | `robots:manage` |
| `POST`   | `/api/admin/robots/{name}/token`       | `robots:manage` |
| `DELETE` | `/api/admin/robots/{name}/token`       | `robots:manage` |
| `DELETE` | `/api/admin/robots/{name}`             | `robots:manage` |

Operation summary:

- `POST /api/admin/robots` — create a new robot, or revive a tombstoned one by
  name. Returns the plaintext token once.
- `GET /api/admin/robots` — list all robot accounts.
- `GET /api/admin/robots/{name}` — robot details including token status, roles,
  and effective caps.
- `PUT .../description` — update the `robot_description` field.
- `POST .../token` — create a token, or rotate with `--renew`.
- `DELETE .../token` — revoke the active token without replacement.
- `DELETE /api/admin/robots/{name}` — tombstone the robot; does not remove the
  `users` row.

Role management, enable/disable, and default-channel assignment go through the
entity endpoints introduced in P2 using the robot's synthetic email.

### Create or Revive Robot — `POST /api/admin/robots`

Request body:

```json
{
  "name": "ci-builder",
  "description": "Nightly Ceph build pipeline",
  "expires": "2027-04-17",
  "roles": ["builder"]
}
```

- `name`: required; validated against the name pattern.
- `expires`: required for the initial token; `null` for no expiry; otherwise a
  `YYYY-MM-DD` string. Omitted → 400.
- `roles`: optional list of role names; defaults to `[]`.
- `description`: optional.

Behavior depends on the current state of the row with synthetic email
`robot+<name>@robots`:

| Current state                             | Action                   |
| ----------------------------------------- | ------------------------ |
| No row                                    | Create new robot + token |
| Tombstone (`active = 0, is_robot = 1`)    | Revive (see below)       |
| Active robot (`active = 1, is_robot = 1`) | 409 Conflict             |
| Human row (`is_robot = 0`)                | 409 Conflict             |

The **revive** transaction runs under `BEGIN IMMEDIATE` (SQLite acquires a
reserved lock before any read, serialising concurrent writers) and performs the
following steps atomically:

1. Re-read the current `users` row under the write lock. If the row is now
   `active = 1` (a concurrent revive committed first), abort with 409 Conflict.
2. Delete all `user_roles` rows for this email.
3. Delete all `robot_tokens` rows for this email (including revoked tombstones
   from the prior identity).
4. Update `users`: `active = 1`, `name = "robot:<name>"`, `robot_description` =
   request value or `NULL`, `default_channel_id = NULL`,
   `created_at = unixepoch()`.
5. Insert `user_roles` rows for the requested roles.
6. Insert the new `robot_tokens` row.
7. Invalidate any stray cache entries by owner email.

The tombstone transaction (`DELETE /api/admin/robots/{name}`) uses the same
`BEGIN IMMEDIATE` isolation. Both operations mutate `users.active` based on a
prior read of the same row and must serialise to avoid the classic
read-then-write race.

Past `builds.user_email` rows remain attributed to the same synthetic email.
Anyone inspecting history sees a continuous identity thread, but all
account-level metadata is reset as though the robot were created fresh.

Response (200 OK) — token shown **once**:

```json
{
  "name": "ci-builder",
  "display_name": "robot:ci-builder",
  "description": "Nightly Ceph build pipeline",
  "active": true,
  "token": "cbrk_...",
  "token_prefix": "a1b2c3d4e5f6",
  "token_expires_at": 1798761600,
  "created_at": 1745136000,
  "revived": false
}
```

`revived` is `true` when the request resulted in a tombstone revive, `false` on
fresh create. This is advisory for tooling and logging.

### Get Robot — `GET /api/admin/robots/{name}`

Response (200 OK) — no token content:

```json
{
  "name": "ci-builder",
  "display_name": "robot:ci-builder",
  "description": "Nightly Ceph build pipeline",
  "active": true,
  "created_at": 1745136000,
  "token_status": {
    "state": "active",
    "prefix": "a1b2c3d4e5f6",
    "expires_at": 1798761600,
    "first_used_at": 1745200000,
    "last_used_at": 1745998000,
    "token_created_at": 1745136000
  },
  "roles": ["builder"],
  "effective_caps": [
    "builds:create",
    "builds:revoke:own",
    "builds:list:own",
    "channels:view",
    "workers:view"
  ]
}
```

`token_status.state` is one of `"active"`, `"expired"`, `"revoked"`, or `"none"`
(no token row exists). For tombstoned robots, the top-level `active` is `false`
and `token_status.state` is `"revoked"`.

### Create / Rotate Token — `POST /api/admin/robots/{name}/token`

Request body:

```json
{
  "expires": "2027-04-17",
  "renew": false
}
```

- `expires`: required; `YYYY-MM-DD` string or `null`. Omitted → 400.
- `renew`: optional boolean, default `false`. Must be `true` if an active or
  expired (non-revoked) token exists; otherwise the request is rejected with
  HTTP 409.

Response (200 OK) — token shown **once**:

```json
{
  "token": "cbrk_...",
  "token_prefix": "9f8e7d6c5b4a",
  "expires_at": 1798761600,
  "created_at": 1746000000
}
```

### List Robots — `GET /api/admin/robots`

Response (200 OK):

```json
[
  {
    "name": "ci-builder",
    "display_name": "robot:ci-builder",
    "active": true,
    "token_state": "active",
    "token_expires_at": 1798761600,
    "last_used_at": 1745998000
  }
]
```

### Modified Non-Robot Endpoints

- `GET /api/auth/whoami` — response body gains `is_robot: bool`. For robot
  bearer tokens, `email` is the synthetic form and `name` is `"robot:<name>"`.
- `POST /api/auth/api-keys` — if the target of key creation resolves to
  `is_robot = 1`, the handler returns `400` with a message directing the caller
  to use robot token endpoints. This covers both self-creation (prevented by the
  forbidden cap `apikeys:create:own`) and any admin-on-behalf path.
- Build response bodies (`POST /api/builds`, `GET /api/builds`,
  `GET /api/builds/{id}`) gain an `is_robot: bool` sibling to `user_email`. The
  `user_email` field itself remains the raw value stored in `builds.user_email`
  (the synthetic email for robot-submitted builds); callers distinguish via
  `is_robot`.

## `cbc` CLI

All robot account management commands live under `cbc admin robots`. Role
management, enable/disable, and default-channel assignment for robots use the
existing `cbc admin users` verbs pointed at the robot's synthetic email, or
equivalent shortcuts on `cbc admin robots` that wrap the entity endpoints.

```
# Create a robot account.
# If <name> matches an existing tombstone (deleted robot), the same
# command revives it: all prior roles/tokens are cleared and the
# account is reinitialised from the arguments given here, exactly as
# though it were a fresh create. There is no separate `revive` verb —
# calling `create` with a tombstoned name IS the revive path.
cbc admin robots create <name>
    (--expires <YYYY-MM-DD> | --no-expires)   Exactly one required;
                                              applies to initial token
    [--description <text>]
    [--role <role>]...

# List all robot accounts
cbc admin robots list

# Get robot account details
cbc admin robots get <name>

# Description update
cbc admin robots set-description <name> <text>

# Enable / disable (wraps entity endpoint)
cbc admin robots enable  <name>
cbc admin robots disable <name>

# Role management (wraps entity endpoint)
cbc admin robots roles set    <name> --role <role>...   # replaces all
cbc admin robots roles add    <name> <role>
cbc admin robots roles remove <name> <role>

# Default channel (wraps entity endpoint)
cbc admin robots default-channel set   <name> <channel>
cbc admin robots default-channel clear <name>

# Token management
cbc admin robots token new <name> \
    (--expires <YYYY-MM-DD> | --no-expires) [--renew]
    # Creates a new token.
    # Fails if a non-revoked token exists and --renew is not given.
    # With --renew: atomically revokes existing token and issues a
    # new one.
    # Prints the new token plaintext once.

cbc admin robots token revoke <name> --yes-i-really-mean-it
    # Revokes the current token without creating a replacement.

# Tombstone the robot. The users row is preserved so that build history
# remains attributed to this synthetic email; reactivating the name
# requires `create`, which reinitialises the account from scratch.
cbc admin robots delete <name> --yes-i-really-mean-it
    # Sets users.active = 0 and revokes all robot_tokens rows.
    # Does NOT remove the users row; build history stays
    # attributed to this synthetic email.
```

### `--yes-i-really-mean-it` Policy

The `--yes-i-really-mean-it` flag is required on all irreversible `cbc`
operations. Its adoption across existing commands is preparatory work P1 and is
a prerequisite for any robot-specific command work.

## Open Questions

### Q1 — Rate limiting for `cbrk_` auth failures

Argon2id verification is expensive by design. A burst of auth failures from a
misconfigured robot could tie up server threads. The unified token cache
mitigates this for well-behaved clients (cache hit = no Argon2 cost). Per-robot
rate limiting on auth failures is not addressed in this design; deferred to a
follow-on.
