## Findings

### F1 — Medium: `--type` not truly optional

`cbc/src/builds.rs:68` still has `default_value = "dev"`
on the `--type` flag, and `version_type` is a required
`VersionType` enum in `BuildDescriptor`. The help text
says "omit to use channel default" but the client
always sends `"dev"`. The server cannot distinguish
"user chose dev" from "user omitted --type."

The design (v3, line 163-173) says the server should
fall back to `channel.default_type_id` when the user
omits `--type`. The `resolve_and_rewrite` doc comment
(line 42) says "or fall back to the channel's
`default_type_id`" but the code (lines 75-80) always
uses `descriptor.version_type` directly. The
`default_type_id` column, `set_default_type` DB
function, and `set-default` REST endpoint are
all wired up but never consulted during resolution.

This is dead infrastructure. Either:
**(a)** Make `version_type` optional in the proto
(wire-format change) and implement the fallback, or
**(b)** Remove the help text claim and document that
`default_type_id` is advisory only (for display in
`cbc channel list`), not used in resolution.

### F2 — Medium: trigger errors all classified as Transient

`scheduler/trigger.rs:105` maps all
`resolve_and_rewrite` errors to
`TriggerError::Transient`. Several failure modes are
permanent:

- "insufficient scope for channel/type 'X'" — scopes
  revoked, retrying won't help
- "channel 'X' not found" — deleted, retrying won't
  help
- "type 'Y' not configured for channel 'X'" — removed

These burn through `MAX_RETRY_COUNT` retries (with
exponential backoff) before the task is disabled,
when they should be `Fatal` (immediate disable).

**Fix:** Either return a typed error from
`resolve_and_rewrite` that distinguishes DB errors
(transient) from "not found / denied" (fatal), or
classify the string errors by pattern.

### F3 — Medium: `soft_delete_type` not atomic

`db/channels.rs:288-311` performs two separate SQL
statements (soft-delete the type, then clear
`default_type_id`) without a transaction. If the
process crashes between them, a channel's
`default_type_id` points to a soft-deleted type.

The next resolution attempt would still work (the
`default_type_id` is never consulted per F1), but
`cbc channel list` would display a deleted type as the
default. An admin API call or server restart would not
auto-heal this.

**Fix:** Wrap both statements in a transaction, or
combine into a single CTE query.

### F4 — Medium: `user_can_view_channel` swallows DB errors

`routes/channels.rs` (the scope-filtering helper) uses
`.unwrap_or_default()` on the DB query result. A
transient DB failure silently treats the user as having
no assignments — all channels disappear from listing
with no error or log.

**Fix:** Log the error and/or propagate it as 500.

### F5 — Medium: `soft_delete_channel` doesn't cascade to types

When a channel is soft-deleted, its child types remain
active (`deleted_at IS NULL`). The resolution path is
safe (it JOINs `WHERE c.deleted_at IS NULL`), but
orphaned active types are visible via `get_type()` by
ID and could confuse admin tooling.

**Consider:** Soft-delete all types for the channel in
the same operation.

### F6 — Low: `set_default_type` doesn't validate ownership

`db/channels.rs:314-327` blindly sets
`default_type_id = type_id` without checking the type
belongs to the channel. A type from channel A could be
set as the default for channel B.

The route handler in `routes/channels.rs` validates
ownership before calling this, so the DB function is
only called with correct values in practice. But the
DB layer should be self-protecting.

### F7 — Low: auto-set default type silently discards error

`routes/channels.rs` (first type auto-sets default):


```rust
let _ = db::channels::set_default_type(...).await;
```

If the DB update fails, the type is created but the
channel's `default_type_id` stays NULL. Should at
minimum log the error.

### F8 — Low: duplicated helpers

`is_unique_violation` is defined in both
`routes/channels.rs` and `routes/admin.rs`.
`require_cap` is defined in both `routes/channels.rs`
and `routes/permissions.rs`. These should be extracted
to a shared utility module.

### F9 — Low: `#[allow(dead_code)]` not cleaned up

`db/channels.rs` has `#[allow(dead_code)]` on
`get_channel_by_name` (line 101),
`resolve_channel_type` (line 333), and
`ResolvedChannelType` (line 45). These are used by
`channels/mod.rs` (commit 5), so the annotations
should have been removed in commit 5.

Similarly, `db/users.rs` has `#[allow(dead_code)]` on
`set_default_channel` (line 60) and
`default_channel_id` (line 19) which are used by
commits 4-5.

### F10 — Low: dead `default_marker` variable in cbc

`cbc/src/channels.rs:89` declares `default_marker = ""`
which is always empty and used only as the else-branch
in an if/else. Remove and use `""` directly.

### F11 — Low: inconsistent display for missing channel

`cbc build get` (builds.rs:408) prints empty string for
`None` channel. `cbc build new` prints
`"(server default)"`. `cbc periodic get` prints `"-"`.
Pick one convention.

## What's Done Well

**Shared resolution helper.** `channels/mod.rs` with
`resolve_and_rewrite` is clean, well-documented, and
tested. Both submission and scheduler paths call it
without duplication. The scope checking, prefix
template resolution, and image rewriting are all in
one place.

**Migration quality.** The SQL is clean: partial
unique indexes for soft-delete, CHECK constraint for
type names, proper FK references. The autogenerated
`.sqlx/` cache is committed and compilation works
with `SQLX_OFFLINE=true`.

**Channel CRUD endpoints.** Complete and consistent:
proper capability checks, scoped visibility for reads,
inline types in responses, unique violation handling.
Good error messages throughout.

**Proto backwards compatibility.** The
`Option<String>` with `#[serde(default)]` on
`BuildDescriptor.channel` correctly handles old
descriptors (deserialize to `Some(value)`) and new
ones (absent field = `None`).

**Scope pattern matching.** Clean implementation with
unit tests covering exact match, wildcard suffix, and
global wildcard. The scope check in
`resolve_and_rewrite` constructs the `channel/type`
value correctly.

## Summary

| Severity | # | Finding |
|----------|---|---------|
| Medium | 5 | F1-F5 |
| Low | 6 | F6-F11 |


No blockers. The medium issues:

- F1 (`--type` / `default_type_id` dead code) is a
  design gap — decide whether to implement the
  fallback or document it as deferred.
- F2 (trigger error classification) could cause
  10 unnecessary retries for permanent failures.
- F3 (non-atomic soft-delete) is a data consistency
  risk on crash.
- F4 (swallowed DB error) is a silent failure mode.
- F5 (cascade) is a data hygiene concern.
